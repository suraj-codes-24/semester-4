§14 Defining a Package

What is a Package?

A package is a namespace that organises related classes and interfaces into groups — like folders on a computer. It prevents name conflicts and controls access.

Two types: Built-in packages (java.lang, java.util, java.io…) and User-defined packages.

Why Use Packages?

  • Name collision prevention: Two classes with the same name can coexist if in different packages (e.g., java.util.Date vs java.sql.Date).
  • Access control: Package-private (default) access restricts visibility to classes in the same package.
  • Organisation: Logical grouping makes large projects manageable.
  • Reusability: Packages can be compiled into JAR files and reused across projects.

Defining a Package

The package statement must be the very first line of the source file (before any imports or class declaration). Only one package statement per file. 
// File: com/example/shapes/Circle.java
package com.example.shapes;      // ← MUST be first line

import java.lang.Math;

public class Circle {
    private double radius;

    public Circle(double r) { this.radius = r; }

    public double area() { return Math.PI * radius * radius; }
}
Directory structure must match package name!
Package com.example.shapes → file must be in folder com/example/shapes/
The JVM looks for the class file in directories that match the package hierarchy.
If no package statement is given, the class belongs to the default (unnamed) package. This is fine for small programs but never use default package in production — you cannot import classes from the default package into named packages.

Package as Access Boundary

Members with default (package-private) access — declared with no access modifier — are accessible to all classes in the same package, but not to classes in other packages. This is a key reason to group related classes in a package.
🧪 Quick Check — Package access
Class A (in pkg com.a) has field int x = 5; (default access).
Class B (in pkg com.a) — can B access A.x? YES — same package.
Class C (in pkg com.b) — can C access A.x? NO — different package, not public/protected.
✓ Default access = package-visible only.

§15 CLASSPATH & Compiling Packages

What is CLASSPATH?

CLASSPATH is an environment variable (or command-line option) that tells the Java compiler (javac) and JVM (java) where to look for user-defined classes and packages. It is a list of directories, JAR files, or ZIP files.

Setting CLASSPATH

Three ways to set CLASSPATH:
1
Environment Variable (permanent):
Windows: set CLASSPATH=C:\myproject\classes;.
Linux/Mac: export CLASSPATH=/home/user/myproject/classes:.
The dot . means "current directory" — always include it!
2
Command-line flag (temporary, recommended):
javac -classpath /path/to/classes MyClass.java
java -classpath /path/to/classes MyClass
Also: -cp is a shorthand for -classpath.
3
In IDE: Eclipse / IntelliJ / NetBeans manage CLASSPATH automatically via project settings (Build Path / Module dependencies).
Multiple paths are separated by ; on Windows and : on Linux/Mac.
If CLASSPATH is not set, the JVM only looks in the current directory by default. Setting CLASSPATH without . means the current directory is no longer searched!

Compiling & Running Packages — Step by Step

Suppose package is com.example.shapes and file is Circle.java:
1
Create directory structure:
mkdir -p com/example/shapes
2
Place source file in matching directory:
com/example/shapes/Circle.java
3
Compile from project root:
javac com/example/shapes/Circle.java
Output: com/example/shapes/Circle.class
4
Run using fully qualified class name:
java com.example.shapes.Circle
Note: use dots, not slashes, for the class name!

The -d Flag for Compilation

javac -d . Circle.java

The -d flag specifies the destination directory for compiled .class files. With -d ., the compiler automatically creates the package subdirectories and places .class files in the correct location.

Example: javac -d bin src/com/example/shapes/Circle.java
Creates: bin/com/example/shapes/Circle.class
🧪 Full compile + run workflow
Source file: src/com/demo/Hello.java with package com.demo;
Compile: javac -d out src/com/demo/Hello.java
Run: java -cp out com.demo.Hello
✓ Always use fully qualified name (package.ClassName) when running packaged classes.
JAVA_HOME vs CLASSPATH: JAVA_HOME points to the JDK installation directory (used by build tools). CLASSPATH tells JVM where your compiled classes are. These are different environment variables with different purposes!

§16 JAR Files for Library Packages

What is a JAR File?

JAR (Java ARchive) — a file format based on ZIP that bundles multiple compiled .class files, resources (images, config files), and a MANIFEST.MF file into a single distributable archive. Used to distribute libraries and executable programs.
  • Extension: .jar
  • Created with the jar command-line tool (part of JDK).
  • Can be used in CLASSPATH just like a directory.
  • Executable JAR: can be run directly with java -jar myapp.jar if Main-Class is set in MANIFEST.

Creating a JAR File

1
Compile your classes first:
javac -d out src/com/example/shapes/*.java
2
Create the JAR from the out/ directory:
jar cf shapes.jar -C out .
cf = create file. -C out . = change to out/ directory and include everything.
3
Create executable JAR (with entry point):
jar cfe app.jar com.example.Main -C out .
e flag sets the entry point (Main-Class) in the manifest.
4
Run the executable JAR:
java -jar app.jar

MANIFEST.MF — The JAR Metadata File

Every JAR contains a META-INF/MANIFEST.MF file that describes the archive. Key manifest entries: 
# MANIFEST.MF example
Manifest-Version: 1.0
Main-Class: com.example.Main        # for executable JARs
Class-Path: lib/helper.jar .        # additional JARs this JAR depends on
Created-By: 17.0.5 (Oracle Corporation)

Using a JAR as a Library

Add the JAR to your CLASSPATH when compiling and running: 
javac -cp shapes.jar:. MyApp.java
java  -cp shapes.jar:. MyApp
In Windows: use ; instead of : 
javac -cp shapes.jar;. MyApp.java
IDEs handle this via "Add External JAR" in the Build Path settings.

JAR vs Class Files — Why JAR?

Raw .class files Multiple files to distribute
Directory structure must be preserved
No version/metadata info
Harder to manage large libraries
JAR file Single file — easy to share
Package structure preserved internally
Contains MANIFEST with metadata
Can be signed for security
Standard format for Java libraries
Popular Java libraries (Apache Commons, Google Guava, Gson, JUnit) are all distributed as JAR files. When you add a dependency in Maven/Gradle, it downloads the JAR automatically.

§17 Import & Static Import

The import Statement

import allows you to use a class from another package by its simple name instead of the fully qualified name. It is a convenience — the class is not physically copied. 
// Without import — use fully qualified name every time
java.util.ArrayList<String> list = new java.util.ArrayList<>();

// With import — use simple name
import java.util.ArrayList;
ArrayList<String> list = new ArrayList<>();

// Wildcard import — imports all PUBLIC classes from java.util
import java.util.*;
// Does NOT import sub-packages! java.util.* does not import java.util.concurrent.*

// Both import — compiler resolves by which one you actually use
import java.util.Date;
import java.sql.Date;       // ❌ Ambiguous! Must use fully qualified names.
Import does NOT:
  • Make the code larger — it's purely a compile-time tool.
  • Import sub-packages: import java.util.* does NOT include java.util.concurrent.*
  • Affect runtime performance.
java.lang is always implicitly imported — never write import java.lang.String.

Resolving Name Conflicts

import java.util.Date;    // import one
// use the other via fully qualified name
java.sql.Date sqlDate = new java.sql.Date(System.currentTimeMillis());
Date utilDate = new Date();
Wildcard vs Specific import: import java.util.* and import java.util.ArrayList have identical performance. Specific imports are preferred in professional code for clarity. IDEs (like IntelliJ) auto-manage imports.

Static Import (Java 5+)

Static import allows you to use static members (fields and methods) of a class directly, without prefixing with the class name.
Syntax: import static packageName.ClassName.memberName; 
// Without static import
double area = Math.PI * Math.pow(radius, 2);
System.out.println(Math.sqrt(16));

// With static import
import static java.lang.Math.PI;
import static java.lang.Math.pow;
import static java.lang.Math.sqrt;
import static java.lang.Math.*;    // wildcard — imports all static members

double area = PI * pow(radius, 2);
System.out.println(sqrt(16));      // 4.0

// Very common in JUnit tests:
import static org.junit.Assert.*;
assertEquals(42, result);           // instead of Assert.assertEquals(...)
Regular import Imports a class type
import java.util.ArrayList;
Lets you write ArrayList instead of java.util.ArrayList
Static import Imports static members
import static java.lang.Math.PI;
Lets you write PI instead of Math.PI
Overuse of static import is bad practice. It reduces code clarity — a reader can't tell which class sqrt() or PI comes from. Use only when the static member is very well-known (e.g., Math constants, JUnit assertions).
🧪 Static import conflict — what happens?
import static java.lang.Integer.MAX_VALUE;
import static java.lang.Long.MAX_VALUE; ← ambiguous!
Compiler error: "reference to MAX_VALUE is ambiguous." Must use Integer.MAX_VALUE or Long.MAX_VALUE explicitly.
✓ Just like regular import conflicts — resolve by using the fully qualified form.

Package Naming — Reverse Domain Convention

Convention: use your organisation's domain name in reverse order to guarantee global uniqueness.
  • Domain: example.com → Package: com.example
  • Full: com.example.project.module
  • Open source: org.springframework.core, io.netty.channel
  • Google: com.google.common.collect
  • Apache: org.apache.commons.lang3
Package naming rules:
  • All lowercase only (to avoid conflicts with class names).
  • No hyphens — use underscores if needed: com.example.my_module
  • No Java reserved words.
  • Start with a letter (not a digit).
  • If domain starts with digit/reserved word: prefix with underscore: _123abc.com → com._123abc
These are conventions, not enforced by the compiler. But violating them makes your code unprofessional and confusing. Every exam and interview expects you to follow them correctly.

OOP Pillars — One-Liners

Encapsulation
private fields + public getters/setters = data hiding + control.
Inheritance
extends = child gets parent's members. IS-A relationship. Code reuse.
Polymorphism
Same method, different behaviour. Overloading = compile-time. Overriding = runtime.
Abstraction
Hide HOW. Show WHAT. Via abstract class or interface.

Critical Keyword Cheatsheet

thisReference to current object. Used to resolve name conflicts or call another constructor.
superReference to parent class. super() calls parent constructor — must be first statement.
staticBelongs to class, not instance. Shared. Can't use this/super in static methods.
finalVariable = constant. Method = can't override. Class = can't extend.
abstractClass = can't instantiate. Method = no body, must be overridden.
newAllocates heap memory + calls constructor. Returns reference.
instanceofTests if object is of a type. Returns boolean. Safe before casting.
extendsClass inherits from one class. Interface extends interface(s).
implementsClass commits to implementing all methods of interface(s).
interfacePure contract. All fields = public static final. Methods = public abstract by default.

Overloading vs Overriding — 3-Line Summary

Overloading: Same class · Different params · Compile-time · Return type can differ.
Overriding: Subclass · Same params · Runtime · Return type same (or covariant) · Cannot narrow access · Cannot override static/final/private.

Abstract Class vs Interface — 5-Point Diff

1. Constructor
Abstract class HAS constructor. Interface has NO constructor.
2. Fields
Abstract class: any. Interface: only public static final.
3. Multiple
Class extends ONE abstract class. Class implements MULTIPLE interfaces.
4. Methods
Abstract class: any mix. Interface: abstract + default + static (Java 8+).
5. When to use
Abstract class: shared code + IS-A. Interface: contract + multiple capabilities.
6. Access
Abstract class methods: any modifier. Interface methods: public only.
