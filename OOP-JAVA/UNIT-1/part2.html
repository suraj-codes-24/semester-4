§7 Classes, Objects & The 4 OOP Pillars

Class vs Object

Class — Blueprint A template / prototype.
Defines properties (fields) and behaviours (methods).
Does NOT occupy memory by itself.
e.g., class Car { ... }
Object — Instance A real-world entity created from a class.
Occupies memory in the heap.
Has its own copy of instance variables.
e.g., Car myCar = new Car(); 
class Car {
    // Fields (instance variables)
    String brand;
    int speed;

    // Method (behaviour)
    void accelerate() {
        speed += 10;
        System.out.println(brand + " speed: " + speed);
    }
}

// Creating objects
Car c1 = new Car();   // c1 is a reference to a Car object in heap
c1.brand = "Toyota";
c1.speed = 60;
c1.accelerate();      // Toyota speed: 70
new keyword — allocates memory in the heap for the object and returns a reference. It also calls the constructor to initialise the object.

The 4 Pillars of OOP

1. Encapsulation
Bundling data + methods together. Hiding internal state using private fields + public getters/setters.
2. Inheritance
A class (child) acquires properties and behaviours of another class (parent) using extends.
3. Polymorphism
"Many forms." One method/interface, multiple implementations. Achieved via overriding (runtime) and overloading (compile-time).
4. Abstraction
Hiding implementation details; showing only essential features. Achieved via abstract classes and interfaces.
Exams love: "What are the 4 pillars of OOP? Explain each with an example." Memorise the names and one-line definitions for each.

Class Anatomy — Full Example

public class Student {

    // 1. Fields
    private String name;
    private int rollNo;
    static int totalStudents = 0;   // class variable

    // 2. Constructor
    public Student(String name, int rollNo) {
        this.name = name;
        this.rollNo = rollNo;
        totalStudents++;
    }

    // 3. Getter / Setter
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }

    // 4. Instance method
    public void display() {
        System.out.println(rollNo + ": " + name);
    }

    // 5. Static method
    public static int getTotal() { return totalStudents; }
}

this keyword

this is a reference to the current object. Used to:
  • Distinguish instance variables from parameters with same name: this.name = name
  • Call another constructor in the same class: this(args) — must be first statement.
  • Pass current object as argument: method(this)
  • Return current object from a method: return this (method chaining).
🧪 What is the output?
Student s1 = new Student("Alice", 1);
Student s2 = new Student("Bob", 2);
System.out.println(Student.getTotal());
✓ Output: 2 — totalStudents is static, shared and incremented by both constructors.

§8 Constructors & Methods

Constructors

A constructor is a special method that is called automatically when an object is created using new. It initialises the object's state.
Rules: same name as class · no return type (not even void) · can be overloaded.

Types of Constructors

class Box {
    double length, width, height;

    // 1. Default constructor (no parameters)
    Box() {
        length = width = height = 1.0;
        System.out.println("Default Box created");
    }

    // 2. Parameterised constructor
    Box(double l, double w, double h) {
        length = l; width = w; height = h;
    }

    // 3. Copy constructor (Java doesn't provide one automatically!)
    Box(Box other) {
        this.length = other.length;
        this.width  = other.width;
        this.height = other.height;
    }

    double volume() { return length * width * height; }
}
If you do NOT write any constructor, Java provides a default constructor (no-arg, does nothing). Once you write ANY constructor, the default is no longer provided automatically!
Constructor chaining with this(): this(args) calls another constructor in the SAME class and must be the very first statement. super(args) calls the parent constructor.

Methods

A method is a block of code that performs a specific task. It is defined inside a class and can be called on objects (instance methods) or on the class (static methods).
Syntax: accessModifier returnType methodName(parameters) { body }

Method Categories

Instance methodOperates on instance variables. Needs an object to call.obj.method()
Static methodBelongs to class. Cannot access instance variables directly.ClassName.method()
GetterReturns value of private field. Convention: getFieldName()obj.getName()
SetterSets value of private field. Convention: setFieldName(val)obj.setName(x)

Pass by Value in Java

Java is always pass-by-value — but for objects, the value is the reference (memory address). So:
  • Primitive types: method gets a copy — changes don't affect original.
  • Object references: method gets a copy of the reference — you CAN modify the object's fields, but you CANNOT reassign the original reference.
🧪 Tricky Pass-by-Value Example
void change(int x) { x = 99; } int a = 5; change(a); // a is still 5!
void change(int[] arr) { arr[0] = 99; } int[] a = {5}; change(a); // a[0] = 99!
✓ Primitives: no change to original. Arrays/objects: fields CAN be changed via reference.

Varargs (Variable Arguments)

// Accepts zero or more int arguments
int sum(int... nums) {
    int total = 0;
    for(int n : nums) total += n;
    return total;
}
sum(1, 2, 3);    // 6
sum();           // 0 — zero arguments is valid

§9 Access Specifiers, Static & Final

Access Specifiers (Modifiers)

Access specifiers control the visibility (scope) of classes, fields, methods, and constructors.
ModifierSame ClassSame PackageSubclass (diff pkg)Other Package
private✅❌❌❌
default (no keyword)✅✅❌❌
protected✅✅✅❌
public✅✅✅✅
Most restrictive → Most open: private < default < protected < public
Best practice (encapsulation): make fields private, methods public.
Top-level classes can only be public or default — NOT private or protected. Inner classes can use all 4.

Static Keyword

static members belong to the class, not to any particular object. Shared across all instances.
  • Static variable: one copy per class. Initialised when class loads.
  • Static method: can be called without object. Cannot use this or access instance variables directly.
  • Static block: runs once when class is first loaded. Used for static initialisation.
  • Static nested class: inner class declared static — doesn't need outer class instance.
class Counter {
    static int count = 0;

    static {                       // static block
        System.out.println("Class loaded!");
    }

    Counter() { count++; }

    static int getCount() { return count; }
}

Final Keyword

final means "cannot be changed." Applied to variables, methods, and classes.
Applied toMeaningExample
VariableValue cannot be reassigned (constant)final double PI = 3.14;
MethodCannot be overridden in subclassfinal void show() {}
ClassCannot be subclassed (extended)final class String {}
final + static = constant. Convention: ALL_CAPS with underscores.
static final double MAX_SPEED = 200.0; 
final class Immutable {        // cannot be extended
    final int value;             // must be initialised in constructor

    Immutable(int v) { value = v; }

    final void show() {         // cannot be overridden
        System.out.println(value);
    }
}
// Java's String class is final — that's why you can't extend it!

Blank Final & final with Objects

Blank final variable: declared final but not initialised at declaration — must be initialised in the constructor.

final reference: the reference cannot change (point to a different object), but the object's fields CAN be modified! 
final int[] arr = {1, 2, 3};
arr[0] = 99;   // ✅ allowed — modifying content
arr = new int[5]; // ❌ error — cannot reassign reference
Common exam trap: "Can you change the value of a final array?" — YES you can change its elements. Only the reference is final, not the array contents.

§10 Inheritance — Super & Sub Classes

What is Inheritance?

Inheritance is the mechanism by which one class (child / subclass) acquires the properties and behaviours of another class (parent / superclass). Promotes code reuse and IS-A relationship.
Keyword: extends
Animal ←extends Dog ←extends GoldenRetriever

Types of Inheritance in Java

TypeDescriptionJava supports?
SingleOne child inherits from one parent.✅ Yes
MultilevelA → B → C (chain of inheritance).✅ Yes
HierarchicalMultiple children from one parent.✅ Yes
MultipleOne child from multiple parents (classes).❌ No (via classes)
HybridCombination of above types.❌ No (via classes)
Java does NOT support multiple inheritance through classes to avoid the Diamond Problem (ambiguity when two parents have same method). It CAN be achieved through interfaces.

super keyword

super refers to the immediate parent class. Used to:
  • Access parent class fields: super.fieldName
  • Call parent class method: super.methodName()
  • Call parent class constructor: super(args) — must be FIRST statement in constructor.
If you don't explicitly write super() in a constructor, Java automatically inserts a call to the no-arg parent constructor. If the parent has no no-arg constructor, you MUST call super(args) manually or you get a compile error.

Inheritance Example — Single & Multilevel

// Superclass
class Animal {
    String name;
    Animal(String name) { this.name = name; }
    void eat()   { System.out.println(name + " eats"); }
    void breathe() { System.out.println(name + " breathes"); }
}

// Single inheritance — Dog IS-A Animal
class Dog extends Animal {
    String breed;
    Dog(String name, String breed) {
        super(name);           // calls Animal constructor
        this.breed = breed;
    }
    void bark() { System.out.println(name + " barks!"); }
}

// Multilevel inheritance
class GoldenRetriever extends Dog {
    GoldenRetriever(String name) {
        super(name, "Golden Retriever");
    }
    void fetch() { System.out.println(name + " fetches!"); }
}

// Usage
GoldenRetriever g = new GoldenRetriever("Buddy");
g.eat();    // inherited from Animal
g.bark();   // inherited from Dog
g.fetch();  // own method

What is Inherited?

✅ Inherited public fields & methods
protected fields & methods
default members (same package)
All methods except constructors
❌ NOT Inherited private fields & methods
Constructors (but called via super)
static members (inherited but not overridden in true sense)

IS-A vs HAS-A Relationship

IS-A (Inheritance)
Use extends.
Dog IS-A Animal ✅
Car IS-A Vehicle ✅
Check: "Is a Dog an Animal?" — Yes.
HAS-A (Composition)
Class contains an object of another class as a field.
Car HAS-A Engine ✅
Student HAS-A Address ✅
Prefer composition over inheritance when IS-A doesn't fit.

Object Class — Root of All Classes

Every class in Java implicitly extends java.lang.Object if no explicit parent is specified. The Object class provides default implementations of:
  • toString() — returns "ClassName@hashcode" by default. Override for readable output.
  • equals(Object o) — default checks reference equality (==). Override for content equality.
  • hashCode() — returns integer hash. Must override when you override equals.
  • getClass() — returns runtime class of object.
  • clone() — creates a copy (must implement Cloneable).
  • finalize() — called before GC (deprecated in Java 9+).
🧪 Why override toString()?
Default: Student@6d06d69c — not helpful.
Override: public String toString() { return "Student[" + name + "," + rollNo + "]"; }
✓ Now System.out.println(s) prints: Student[Alice,1] — much more readable.

§11 Overriding & Overloading

Method Overloading (Compile-time Polymorphism)

Overloading — same method name, different parameter lists (type, count, or order) in the same class. Resolved at compile time (static/early binding). 
class Calculator {
    int    add(int a, int b)           { return a + b; }
    double add(double a, double b)   { return a + b; }
    int    add(int a, int b, int c)   { return a + b + c; }
    String add(String a, String b)    { return a + b; }
}
Return type alone CANNOT distinguish overloaded methods — this does NOT compile:
int foo() { return 1; } and double foo() { return 1.0; } in the same class = compile error.

Method Overriding (Runtime Polymorphism)

Overriding — subclass provides its own implementation of a method already defined in the superclass. Same name, same parameters, same return type (or covariant). Resolved at runtime (dynamic/late binding). 
class Shape {
    void draw() { System.out.println("Drawing a shape"); }
}

class Circle extends Shape {
    @Override                      // annotation — optional but recommended
    void draw() { System.out.println("Drawing a circle ⭕"); }
}

class Rectangle extends Shape {
    @Override
    void draw() { System.out.println("Drawing a rectangle ▭"); }
}

// Runtime polymorphism — method called based on actual object type
Shape s = new Circle();     // Shape reference, Circle object
s.draw();                   // "Drawing a circle ⭕"

Rules for Overriding:
  • Method signature (name + params) must be identical.
  • Return type must be same or covariant (subtype of parent return type).
  • Cannot override static, final, or private methods.
  • Access modifier can be widened (private→public) but NOT narrowed.
  • Can throw same or narrower checked exceptions, or unchecked exceptions.
  • Use @Override annotation — helps compiler catch mistakes.
🧪 Polymorphism with Array of Shapes
Shape[] shapes = { new Circle(), new Rectangle(), new Circle() };
for(Shape s : shapes) s.draw();
✓ Each object's draw() is called based on actual runtime type — this is runtime polymorphism in action!

§12 Encapsulation & Abstraction

Encapsulation

Encapsulation is the process of wrapping data (fields) and the methods that operate on the data into a single unit (class), and restricting direct access to the data from outside the class.
How to achieve encapsulation:
1
Declare all fields as private.
2
Provide public getter methods to read the values.
3
Provide public setter methods to modify the values (with validation if needed).
public class BankAccount {
    private double balance;      // hidden from outside
    private String accountNo;

    public double getBalance() { return balance; }

    public void deposit(double amount) {
        if (amount > 0) balance += amount;  // validation!
        else System.out.println("Invalid amount");
    }

    public void withdraw(double amount) {
        if (amount <= balance) balance -= amount;
        else System.out.println("Insufficient funds");
    }
}
Benefits of Encapsulation:
  • Data hiding — internal state protected from unintended modification.
  • Flexibility — can change internal implementation without affecting external code.
  • Increased security and control (add validation in setters).
  • Makes code easier to test and maintain.

Abstraction

Abstraction is the process of hiding complex implementation details and exposing only what is essential to the user. Focus on what an object does, not how it does it.
Real-world analogy: When you drive a car, you use the steering wheel, accelerator, and brakes — you don't need to know HOW the engine or braking system works internally. That's abstraction.

Two ways to achieve abstraction in Java:

Abstract Class
Partial abstraction (0–100%).
Can have abstract + concrete methods.
Can have constructors and fields.
A class can extend only ONE abstract class.
Interface
Full abstraction (100% by tradition).
All methods are abstract by default (Java 7).
Can have default/static methods (Java 8+).
A class can implement MULTIPLE interfaces.

Encapsulation vs Abstraction

AspectEncapsulationAbstraction
FocusHides dataHides implementation
Howprivate fields + getters/settersabstract classes + interfaces
PurposeData protection & controlSimplify complexity for user
Exampleprivate balance in BankAccountShape.draw() without knowing how each shape draws
These two are often confused in exams. Remember: Encapsulation = wrapping + hiding DATA. Abstraction = hiding HOW things work. Both work together — a well-designed class is both encapsulated and abstracted.

§13 Interfaces & Abstract Classes

Abstract Class

A class declared with the abstract keyword. It may contain abstract methods (no body) and/or concrete methods (with body). Cannot be instantiated directly — must be subclassed. 
abstract class Shape {
    String color;
    Shape(String color) { this.color = color; }

    // Abstract method — no body, MUST be overridden
    abstract double area();

    // Concrete method — has body, can be inherited as-is
    void describe() {
        System.out.println("I am a " + color + " shape with area " + area());
    }
}

class Circle extends Shape {
    double radius;
    Circle(double r, String c) { super(c); radius = r; }

    @Override
    double area() { return Math.PI * radius * radius; }
}

// Shape s = new Shape();  ❌ Cannot instantiate abstract class
Shape s = new Circle(5.0, "Red");  // ✅ polymorphism
s.describe();                       // I am a Red shape with area 78.53...
Abstract Class Rules:
  • If a class has even one abstract method, the class must be declared abstract.
  • An abstract class CAN have a constructor (called via super() from subclass).
  • A subclass MUST implement all abstract methods, or be declared abstract itself.
  • Abstract class CAN have static, final, and concrete methods.

Interfaces

An interface is a completely abstract type that defines a contract — a set of methods that a class must implement. It specifies what to do but not how. Keyword: implements. 
// Defining an interface
interface Drawable {
    // All fields are implicitly: public static final
    int MAX_SIZE = 100;    // same as: public static final int MAX_SIZE = 100;

    // All methods are implicitly: public abstract (Java 7 and below)
    void draw();
    void resize(int factor);

    // Default method — Java 8+ (has a body, subclass can override)
    default void describe() {
        System.out.println("I am a drawable object");
    }

    // Static method — Java 8+ (called as Drawable.info())
    static void info() {
        System.out.println("Drawable interface");
    }
}

// Implementing interface
class Square implements Drawable {
    @Override
    public void draw() { System.out.println("Drawing square □"); }
    @Override
    public void resize(int f) { System.out.println("Resizing by " + f); }
}

// Multiple interfaces
class FancyShape implements Drawable, Serializable, Comparable<FancyShape> {
    // must implement all abstract methods from all interfaces
}
All methods in an interface are implicitly public. When you implement them, you MUST use public access (or you'll get a compile error — can't reduce visibility).

Choosing between them:
Use abstract class when classes share code and have a natural parent-child hierarchy.
Use interface when you want to define a capability that multiple unrelated classes can have (e.g., Flyable for Bird, Airplane, Superman).

Interface Extending Interface

interface Animal   { void breathe(); }
interface Pet extends Animal { void play(); }
// An interface can extend multiple interfaces!
interface SmartPet extends Pet, Trainable { void doTrick(); }
🧪 Classic Exam Question — Can abstract class have constructor?
YES — abstract classes CAN have constructors, but they cannot be called directly with new.
They are called implicitly via super() when a subclass object is created.
✓ Purpose: to initialise the common fields in the abstract class when subclass is constructed.
Functional Interface (Java 8): An interface with exactly ONE abstract method. Used with lambda expressions. e.g., Runnable, Comparator, Callable. Annotate with @FunctionalInterface.
