§5 Default Methods in Interfaces Java 8

The Problem Default Methods Solve

Before Java 8, adding a new method to an interface broke all existing implementing classes — they had to implement the new method or fail to compile. With millions of classes implementing core interfaces like Collection, this was a disaster. Default methods were the solution.
A default method is a method in an interface that has a concrete implementation (body). Implementing classes inherit it automatically and may override it if they wish.
Keyword: default 
interface Vehicle {
    void start();                        // abstract — must implement
    void stop();                         // abstract — must implement

    // default method — provides a default behaviour
    default void honk() {
        System.out.println("Beep beep!");
    }

    default String getType() {
        return "Generic Vehicle";
    }
}

class Car implements Vehicle {
    public void start() { System.out.println("Car started"); }
    public void stop()  { System.out.println("Car stopped"); }
    // honk() and getType() are inherited from Vehicle interface
}

class Truck implements Vehicle {
    public void start() { System.out.println("Truck started"); }
    public void stop()  { System.out.println("Truck stopped"); }

    @Override
    public void honk() { System.out.println("HOOOONK!"); }  // override default
}

Car c = new Car();
c.honk();       // Beep beep! (inherited default)
new Truck().honk(); // HOOOONK! (overridden)

Default Method Conflict Resolution

If a class implements two interfaces that both define the same default method, the class MUST override it — otherwise the compiler throws an error ("inherits unrelated defaults"). 
interface A { default void show() { System.out.println("A"); } }
interface B { default void show() { System.out.println("B"); } }

class C implements A, B {
    @Override
    public void show() {
        A.super.show();     // explicitly call A's default
        // or B.super.show(); or your own implementation
    }
}

// Class method wins over interface default (another rule):
class Base { public void show() { System.out.println("Base class"); } }
class D extends Base implements A {
    // Base.show() wins — class method always wins over interface default
}
Priority rules for default method resolution:
  • 1st priority: Class / superclass methods — always win over interface defaults.
  • 2nd priority: More specific interface — if interface B extends A and both have the same default, B's wins.
  • 3rd priority: Explicitly override — if still ambiguous, you MUST override and resolve manually using InterfaceName.super.method().

Default methods = "virtual extension methods." They let Java evolve existing interfaces without breaking backward compatibility. This is how the entire Collections API was upgraded for streams in Java 8.

§6 Static Methods in Interfaces Java 8

Static Interface Methods

Java 8 allows static methods in interfaces. Unlike default methods, static methods are NOT inherited by implementing classes — they belong to the interface itself and are called via the interface name. 
interface MathOperations {
    int operate(int a, int b);    // abstract

    // static method — utility, not inherited
    static int add(int a, int b)      { return a + b; }
    static int multiply(int a, int b) { return a * b; }
    static boolean isPositive(int n)  { return n > 0; }
}

// Call static methods via INTERFACE NAME, not via object/class
System.out.println(MathOperations.add(3, 4));       // 7
System.out.println(MathOperations.multiply(3, 4)); // 12

class Addition implements MathOperations {
    public int operate(int a, int b) { return a + b; }
    // Cannot call MathOperations.add() as add() here — static not inherited!
}
default method Has body. Inherited by implementing classes.
Can be overridden.
Called via object: obj.method()
For behaviour with fallback.
static method Has body. NOT inherited.
Cannot be overridden.
Called via interface: Interface.method()
For utility/factory methods.
Use static interface methods for:
  • Factory methods: Comparator.comparing(), Predicate.not()
  • Utility methods that don't need instance context
  • Grouping helper methods with the interface they serve (instead of a separate Util class)
Java API examples: Comparator.comparing(), Function.identity(), Predicate.not()

§7 Base64 Encode & Decode Java 8

What is Base64?

Base64 is a binary-to-text encoding scheme that represents binary data using 64 printable ASCII characters (A–Z, a–z, 0–9, +, /). It is used to safely transmit binary data over text-based protocols (HTTP, email, JSON, XML).
Why encode? Binary data can contain control characters that break protocols. Base64 converts any binary data to safe printable text — at the cost of ~33% size increase (3 bytes → 4 characters).

Three Flavours in java.util.Base64

TypeMethodUse case
BasicBase64.getEncoder()Standard Base64, uses + and /. No line breaks.
URL-safeBase64.getUrlEncoder()Uses - and _ instead of + and /. Safe in URLs.
MIMEBase64.getMimeEncoder()Adds line breaks at 76 chars. Used in email.
import java.util.Base64;

// ── Basic Encoding ──
String original = "Hello, Java 8!";
String encoded = Base64.getEncoder()
                       .encodeToString(original.getBytes());
System.out.println("Encoded: " + encoded);
// Encoded: SGVsbG8sIEphdmEgOCE=

// ── Basic Decoding ──
byte[] decodedBytes = Base64.getDecoder().decode(encoded);
String decoded = new String(decodedBytes);
System.out.println("Decoded: " + decoded);
// Decoded: Hello, Java 8!

// ── URL-safe encoding (no +, /) ──
String urlEncoded = Base64.getUrlEncoder()
                          .encodeToString(original.getBytes());
System.out.println("URL Encoded: " + urlEncoded);

String urlDecoded = new String(Base64.getUrlDecoder().decode(urlEncoded));
System.out.println("URL Decoded: " + urlDecoded);

MIME Encoding & Encoding without Padding

// ── MIME encoding — adds line breaks every 76 chars ──
String mimeEncoded = Base64.getMimeEncoder()
                           .encodeToString("LongDataString...".getBytes());

// ── Encoding byte array directly ──
byte[] imageBytes = {71, 73, 70};   // "GIF" in bytes
byte[] encodedBytes = Base64.getEncoder().encode(imageBytes);
System.out.println(new String(encodedBytes));  // R0lG

// ── Without padding (removes trailing =) ──
String noPadding = Base64.getEncoder()
                         .withoutPadding()
                         .encodeToString("test".getBytes());
System.out.println(noPadding);  // dGVzdA  (no trailing =)

// ── Encode file to Base64 (e.g. to embed in JSON/HTML) ──
byte[] fileBytes = Files.readAllBytes(Paths.get("image.png"));
String base64Image = Base64.getEncoder().encodeToString(fileBytes);
// Now you can embed this in: "data:image/png;base64," + base64Image
Base64 Cheatsheet:
  • getEncoder() → Encoder / getDecoder() → Decoder
  • encodeToString(byte[]) → returns String
  • encode(byte[]) → returns byte[]
  • decode(String) → returns byte[]
  • decode(byte[]) → returns byte[]
  • Padding character: = (Base64 output is always a multiple of 4 chars)
🧪 Encode and decode "Java"
Encode: Base64.getEncoder().encodeToString("Java".getBytes()) → "SmF2YQ=="
Decode: new String(Base64.getDecoder().decode("SmF2YQ==")) → "Java"
✓ The trailing == is padding. 4 bytes → 6 Base64 chars padded to 8.
Base64 is NOT encryption! It is just encoding — the data can be decoded by anyone. Do not use Base64 for security. Use it only for safe transmission of binary data in text protocols.

§8 forEach Method Java 8

What is forEach?

forEach is a default method added to Iterable (and hence all Collections) in Java 8. It accepts a Consumer and performs the given action for each element. Replaces the traditional for-each loop with a more functional style.
Signature: void forEach(Consumer<? super T> action)

forEach on Collections

List<String> fruits = List.of("Apple", "Banana", "Cherry");

// Old style
for (String f : fruits) { System.out.println(f); }

// forEach with lambda
fruits.forEach(f -> System.out.println(f));

// forEach with method reference (cleanest)
fruits.forEach(System.out::println);

// forEach with block lambda (multiple statements)
fruits.forEach(f -> {
    String upper = f.toUpperCase();
    System.out.println(upper + " has " + f.length() + " chars");
});

// Map.forEach — takes BiConsumer(key, value)
Map<String, Integer> scores = Map.of("Alice", 92, "Bob", 85);
scores.forEach((name, score) ->
    System.out.println(name + " scored " + score));

forEach on Streams

// Stream.forEach — terminal operation
List.of(1, 2, 3, 4, 5).stream()
    .filter(n -> n % 2 == 0)
    .forEach(System.out::println);    // 2, 4

// forEachOrdered — maintains encounter order even in parallel streams
List.of(1, 2, 3, 4, 5).parallelStream()
    .forEachOrdered(System.out::println);  // always 1,2,3,4,5 in order

// IntStream.forEach
IntStream.range(1, 6).forEach(i -> System.out.print(i + " ")); // 1 2 3 4 5
forEach vs for-loop: forEach cannot use break or continue — those keywords don't work inside lambdas. If you need to stop iteration early, use a traditional for-loop or stream().findFirst().

Map.forEach, Map.getOrDefault & Map.putIfAbsent

Map<String, Integer> wordCount = new HashMap<>();
wordCount.put("hello", 1);
wordCount.put("world", 2);

// getOrDefault — safe get without null check
int count = wordCount.getOrDefault("java", 0);  // 0 — "java" not present

// putIfAbsent — only inserts if key not present
wordCount.putIfAbsent("hello", 99);  // no change — "hello" already exists
wordCount.putIfAbsent("java", 3);   // inserted — "java" was absent

// compute — update value based on existing value
wordCount.compute("hello", (k, v) -> v == null ? 1 : v + 1); // hello=2

// merge — combine existing with new value
wordCount.merge("hello", 1, Integer::sum);  // hello = existing + 1

// forEach on Map
wordCount.forEach((k, v) -> System.out.println(k + " → " + v));

replaceAll on List

List<String> names = new ArrayList<>(List.of("alice", "bob", "charlie"));

// replaceAll — transforms every element in-place
names.replaceAll(String::toUpperCase);
System.out.println(names);  // [ALICE, BOB, CHARLIE]

// removeIf — removes elements matching predicate
names.removeIf(n -> n.length() < 4);
System.out.println(names);  // [ALICE, CHARLIE] — BOB removed
Summary: New iterable methods (Java 8+):
  • forEach(Consumer) — on Iterable, Stream, Map
  • removeIf(Predicate) — on Collection
  • replaceAll(UnaryOperator) — on List
  • sort(Comparator) — on List
  • getOrDefault(key, default) — on Map
  • putIfAbsent(key, value) — on Map
  • merge(key, value, BiFunction) — on Map
  • compute(key, BiFunction) — on Map

§9 Try-with-Resources Java 7

The Problem — Resource Leaks

Before Java 7, developers had to explicitly close resources (files, DB connections, sockets) in a finally block. This was error-prone — forgetting to close caused resource leaks, and the finally block could itself throw an exception hiding the original one. 
// ❌ Old, verbose, error-prone way
BufferedReader br = null;
try {
    br = new BufferedReader(new FileReader("file.txt"));
    System.out.println(br.readLine());
} catch (IOException e) {
    e.printStackTrace();
} finally {
    if (br != null) {
        try { br.close(); } catch (IOException e) { e.printStackTrace(); }
    }
}

// ✅ Java 7+ try-with-resources — clean and safe
try (BufferedReader br = new BufferedReader(new FileReader("file.txt"))) {
    System.out.println(br.readLine());
} catch (IOException e) {
    e.printStackTrace();
}
// br.close() called automatically — even if exception occurs

AutoCloseable Interface

Any class that implements java.lang.AutoCloseable (or its subinterface Closeable) can be used in try-with-resources. Its close() method is called automatically at the end of the try block. 
// Custom AutoCloseable resource
class DatabaseConnection implements AutoCloseable {
    DatabaseConnection() { System.out.println("DB connected"); }

    public void query(String sql) { System.out.println("Query: " + sql); }

    @Override
    public void close() { System.out.println("DB connection closed"); }
}

try (DatabaseConnection db = new DatabaseConnection()) {
    db.query("SELECT * FROM users");
}   // db.close() called automatically here
// Output: DB connected → Query: SELECT * FROM users → DB connection closed

Multiple Resources & Java 9 Enhancement

// Multiple resources — closed in REVERSE order of declaration
try (FileInputStream fis = new FileInputStream("in.txt");
     FileOutputStream fos = new FileOutputStream("out.txt");
     BufferedInputStream bis = new BufferedInputStream(fis)) {
    // use streams...
}
// Close order: bis → fos → fis (reverse of declaration)

// Java 9 enhancement — use effectively-final variables declared outside
BufferedReader br = new BufferedReader(new FileReader("file.txt"));
try (br) {             // Java 9+ — no need to redeclare
    System.out.println(br.readLine());
}

Suppressed Exceptions

If both the try block AND the close() method throw exceptions, the exception from close() is suppressed (added to the primary exception via addSuppressed()). The primary exception propagates; suppressed ones can be retrieved via getSuppressed(). 
try {
    try (DatabaseConnection db = new DatabaseConnection()) {
        throw new RuntimeException("Primary exception");
        // If close() also throws, it becomes a suppressed exception
    }
} catch (RuntimeException e) {
    System.out.println("Primary: " + e.getMessage());
    for (Throwable s : e.getSuppressed()) {
        System.out.println("Suppressed: " + s.getMessage());
    }
}
Try-with-resources rules:
  • Resource must implement AutoCloseable or Closeable.
  • Resources are closed in reverse order of declaration.
  • close() is called even if an exception occurs in the try block.
  • If close() throws, it is suppressed (added to primary exception).
  • Java 9: pre-declared effectively-final variables can be used directly.
  • A catch and finally block can still be used alongside try-with-resources.

§10 Type & Repeating Annotations Java 8

Annotations — Quick Recap

An annotation is metadata attached to code elements (class, method, field, parameter, etc.). Annotations don't directly change program behaviour — they provide information to the compiler, tools, or runtime frameworks.

Type Annotations (Java 8)

Before Java 8, annotations could only be applied to declarations (class, method, field). Java 8 introduced type annotations — annotations can now be applied anywhere a type is used.
Meta-annotation: @Target(ElementType.TYPE_USE) 
import java.lang.annotation.*;

// Define a type annotation
@Target(ElementType.TYPE_USE)
@Retention(RetentionPolicy.RUNTIME)
@interface NonNull {}

@Target(ElementType.TYPE_USE)
@Retention(RetentionPolicy.RUNTIME)
@interface Readonly {}

// Using type annotations — anywhere a type appears
@NonNull String name;                        // field declaration
List<@NonNull String> items;                // generic type argument
Map<@NonNull String, @Readonly Object> map; // multiple in generics
@NonNull String getName() { return name; }   // method return type

// On casts
String s = (@NonNull String) obj;

// On throws clause
void process() throws @Readonly Exception {}

// On new expressions
Object o = new @NonNull Object();

// On array creation
String @NonNull [] arr = new String[5];
Type annotations themselves don't enforce anything at runtime by default. Their power comes from pluggable type-checkers like the Checker Framework, which uses them to catch null pointer bugs, type safety violations, etc. at compile time.

Repeating Annotations (Java 8)

Before Java 8, you couldn't apply the same annotation multiple times to the same element. Java 8 introduced repeating annotations using the @Repeatable meta-annotation. 
// Step 1: Define the repeatable annotation
@Repeatable(Schedules.class)     // points to container annotation
@Retention(RetentionPolicy.RUNTIME)
@interface Schedule {
    String day();
    String time() default "09:00";
}

// Step 2: Define the container annotation
@Retention(RetentionPolicy.RUNTIME)
@interface Schedules {
    Schedule[] value();    // must have value() returning array of @Schedule
}

// Step 3: Use multiple annotations on same element
@Schedule(day = "Monday",    time = "10:00")
@Schedule(day = "Wednesday", time = "14:00")
@Schedule(day = "Friday")                        // default time 09:00
class WeeklyMeeting { }

// Step 4: Read annotations at runtime
Schedule[] schedules = WeeklyMeeting.class
                          .getAnnotationsByType(Schedule.class);
for (Schedule s : schedules) {
    System.out.println(s.day() + " at " + s.time());
}
