§11 Java Module System (JPMS) Java 9

What is JPMS?

The Java Platform Module System (JPMS, Project Jigsaw) introduced in Java 9 provides a higher level of abstraction above packages. A module is a named, self-describing collection of packages, resources, and a module descriptor (module-info.java).

Problems JPMS Solves

  • Classpath hell: Before Java 9, any class could access any other public class — no enforcement of package boundaries across JARs.
  • JDK monolith: All JDK classes were available even if you only needed a fraction. JPMS splits JDK into ~98 modules.
  • Reliable configuration: Modules declare dependencies explicitly — missing dependencies detected at startup, not runtime.
  • Strong encapsulation: Internal packages not explicitly exported are invisible to other modules.

module-info.java

// File: src/com.myapp/module-info.java
module com.myapp {

    // Declare what this module NEEDS
    requires java.sql;           // needs java.sql module
    requires java.logging;       // needs java.logging
    requires transitive java.xml;// transitive: modules requiring com.myapp also get java.xml

    // Declare what this module EXPOSES
    exports com.myapp.api;       // everyone can use this package
    exports com.myapp.model to com.myapp.ui, com.myapp.service; // qualified export

    // For reflection (e.g., frameworks like Spring, JPA)
    opens com.myapp.entities;    // allows deep reflection at runtime
    opens com.myapp.impl to spring.core; // qualified open

    // Service provider/consumer (ServiceLoader pattern)
    uses com.myapp.spi.Plugin;   // this module consumes Plugin service
    provides com.myapp.spi.Plugin
        with com.myapp.impl.DefaultPlugin;  // this module provides an implementation
}

Compiling & Running Modules

# Project structure:
# src/com.myapp/module-info.java
# src/com.myapp/com/myapp/Main.java

# Compile
javac --module-source-path src -d mods -m com.myapp

# Run
java --module-path mods -m com.myapp/com.myapp.Main

# List all JDK modules
java --list-modules

# Show module dependencies
java --describe-module java.sql
Key JPMS benefits:
  • Strong encapsulation — unexported packages truly private.
  • Reliable configuration — missing modules detected at app startup.
  • Smaller deployments — jlink tool creates custom JRE with only needed modules.
  • Better security — reduces attack surface by hiding internal APIs.

§12 Diamond Syntax with Anonymous Inner Class Java 9

The Evolution of Diamond Operator <>

The diamond operator <> allows the compiler to infer generic type arguments. Java 7 introduced it for regular class instantiation. Java 9 extended it to work with anonymous inner classes. 
// Java 5-6 — must repeat type parameter
List<String> list = new ArrayList<String>();   // verbose

// Java 7 — diamond for regular classes
List<String> list = new ArrayList<>();           // ✅ type inferred
Map<String, List<Integer>> map = new HashMap<>(); // infers complex generic

// Java 7 — anonymous class with diamond — ❌ NOT allowed
Comparator<String> cmp = new Comparator<>() {    // ❌ Java 7 error!
    public int compare(String a, String b) { return a.compareTo(b); }
};

// Java 9+ — anonymous class WITH diamond ✅
Comparator<String> cmp = new Comparator<>() {    // ✅ works in Java 9+
    public int compare(String a, String b) { return a.compareTo(b); }
};

// Generic abstract class with anonymous class + diamond
abstract class Container<T> {
    abstract T get();
}
Container<String> c = new Container<>() {        // ✅ Java 9
    public String get() { return "Hello"; }
};
In practice, anonymous classes are rarely needed now — lambdas serve the same purpose for functional interfaces more concisely. The diamond operator for anonymous classes mainly helps when using generic abstract classes where lambdas don't apply.
Quick summary — Diamond <> history:
Java 5: Generics introduced, must write full type.  |  Java 7: <> for regular classes.  |  Java 9: <> also for anonymous inner classes.

§13 Local Variable Type Inference — var Java 10

What is var?

var is a reserved type name (NOT a keyword) that tells the compiler to infer the type of a local variable from its initialiser. The variable still has a static type — it's just written implicitly. 
// Without var
ArrayList<String> names = new ArrayList<String>();
Map<String, List<Integer>> map = new HashMap<>();

// With var — cleaner, DRY (Don't Repeat Yourself)
var names = new ArrayList<String>();       // inferred: ArrayList<String>
var map   = new HashMap<String, List<Integer>>();
var score = 42;                             // inferred: int
var pi    = 3.14;                           // inferred: double
var msg   = "Hello";                        // inferred: String

// var in for-each and traditional loops
for (var name : names) System.out.println(name);
for (var i = 0; i < 10; i++) System.out.print(i);

// var with try-with-resources (Java 11+)
try (var br = new BufferedReader(new FileReader("file.txt"))) {
    var line = br.readLine();
}

// ❌ var CANNOT be used for:
var x;                        // no initialiser
var y = null;                 // can't infer from null
var arr = {1, 2, 3};         // array initialiser without type
// var cannot be used for fields, parameters, or return types
var restrictions — only allowed for: local variables with an initialiser, for-loop index variable, enhanced for-loop element variable, try-with-resources resource variable, and lambda parameter (Java 11+: (var x) -> x.length()).
NOT allowed for: fields, method parameters, return types, catch variable types.
var is statically typed — the inferred type is fixed at compile time. It is NOT dynamic typing like JavaScript. After declaration, the variable cannot hold a different type.

§14 Switch Expressions & Yield Java 14

Switch Statement (Traditional) vs Switch Expression

// ── Old switch STATEMENT (Java 1+) ──
int day = 3;
String name;
switch (day) {
    case 1: name = "Monday"; break;
    case 2: name = "Tuesday"; break;
    case 3: name = "Wednesday"; break;
    default: name = "Unknown";
}

// ── New switch EXPRESSION (Java 14+, preview Java 12/13) ──
String name = switch (day) {
    case 1 -> "Monday";       // arrow syntax — no break needed, no fall-through
    case 2 -> "Tuesday";
    case 3 -> "Wednesday";
    case 4, 5 -> "Thu/Fri";   // multiple labels per case
    default -> "Weekend";
};                              // note: switch expression ends with semicolon

// ── Switch expression with blocks and yield ──
int numLetters = switch (day) {
    case 1, 2, 3, 4, 6, 7 -> 6;    // expression — returned directly
    case 5 -> {
        System.out.println("Friday!");
        yield 6;                       // yield — return value from block
    }
    default -> {
        System.out.println("Unknown day: " + day);
        yield -1;
    }
};

The yield Keyword

yield (Java 13+) is used inside a switch expression block to return a value from that block. It's like return but for switch expression cases that need multiple statements before producing a value. Not usable outside switch expressions.

Switch Expressions with Enums & Pattern Matching

enum Season { SPRING, SUMMER, AUTUMN, WINTER }

String activity = switch (season) {
    case SPRING -> "Gardening";
    case SUMMER -> "Swimming";
    case AUTUMN -> "Hiking";
    case WINTER -> "Skiing";
};  // no default needed — enum is exhaustive

// Pattern matching in switch (Java 21 — standard)
Object obj = "Hello";
String result = switch (obj) {
    case Integer i -> "Integer: " + i;
    case String s  -> "String: " + s.toUpperCase();
    case null      -> "null value";
    default        -> "Other: " + obj;
};

Exhaustiveness: Switch expressions MUST cover all possible values (either with explicit cases or a default). For enums with all values covered, no default needed. Missing a case → compile error.

§15 Text Blocks Java 15

What are Text Blocks?

A text block is a multi-line string literal that eliminates the need for most escape sequences and string concatenation. It starts with """ (triple double-quote) on its own line and ends with """. 
// ── Old way — ugly, escape-heavy ──
String html = "<html>\n" +
              "  <body>\n" +
              "    <p>Hello</p>\n" +
              "  </body>\n" +
              "</html>";

// ── Text block — clean and readable ──
String html = """
              <html>
                <body>
                  <p>Hello</p>
                </body>
              </html>
              """;

// JSON text block
String json = """
        {
            "name": "Alice",
            "age": 30,
            "city": "Mumbai"
        }
        """;

// SQL text block
String query = """
        SELECT name, age
        FROM students
        WHERE score > 90
        ORDER BY name
        """;

// Trailing """ position controls indentation stripping
// Incidental whitespace = whitespace common to all lines is removed

// String methods on text blocks
String noTrailingNewline = """
        Hello!""";       // closing """ on same line = no trailing newline

// Escape sequences in text blocks
String withBackslash = """
        First line \
        continues here""";  // \  = line continuation (no newline inserted)

String withNewlines = """
        A\nB\nC""";      // explicit \n = newline (can mix)
Text block rules:
  • Opening """ must be followed by a newline (no content on same line).
  • Indentation is stripped based on the minimum common leading whitespace.
  • Trailing whitespace on each line is also stripped.
  • \ at end of line = line continuation (no newline in output).
  • \s = explicit single space (prevents trimming of trailing spaces).

§16 Records Java 16

What are Records?

A record is a special class designed to be a transparent, immutable data carrier. The compiler automatically generates: constructor, getters (accessor methods), equals(), hashCode(), and toString() — eliminating the boilerplate of a typical POJO/DTO.
Keyword: record 
// ── Old POJO — 50+ lines of boilerplate ──
class PointOld {
    private final double x, y;
    public PointOld(double x, double y) { this.x=x; this.y=y; }
    public double getX() { return x; }
    public double getY() { return y; }
    public boolean equals(Object o) { /* ... */ }
    public int hashCode() { /* ... */ }
    public String toString() { return "Point["+x+","+y+"]"; }
}

// ── Record — same thing, ONE line ──
record Point(double x, double y) {}

// Using the record
Point p = new Point(3.0, 4.0);
System.out.println(p.x());          // 3.0 — accessor (not getX()!)
System.out.println(p.y());          // 4.0
System.out.println(p);              // Point[x=3.0, y=4.0]
System.out.println(p.equals(new Point(3.0, 4.0)));  // true

Records with Custom Behaviour

record Person(String name, int age) {

    // Compact constructor — add validation (no params needed)
    Person {
        if (name == null || name.isBlank()) throw new IllegalArgumentException("Name required");
        if (age < 0) throw new IllegalArgumentException("Age cannot be negative");
        name = name.trim();   // can transform components before assignment
    }

    // Additional instance methods
    boolean isAdult() { return age >= 18; }
    String greeting() { return "Hi, I'm " + name + ", age " + age; }

    // Static factory method
    static Person anonymous() { return new Person("Anonymous", 0); }
}

record Car(String brand, int year) implements Describable {

      .forEach(System.out::println);

§17 Sealed Classes Java 17

What are Sealed Classes?

A sealed class restricts which other classes may extend (or implement) it. The permitted subclasses are listed explicitly in the permits clause. This creates a closed, controlled class hierarchy.
Keywords: sealed, permits, non-sealed
Why sealed? Previously you could either allow ALL classes to extend (open), or allow NONE (final). Sealed gives a middle ground — only specific, listed classes can extend. 
// ── Sealed class hierarchy ──
sealed class Shape permits Circle, Rectangle, Triangle {}

// Permitted subclass options:

// 1. final — no further subclassing
final class Circle extends Shape {
    double radius;
    Circle(double r) { radius = r; }
}

// 2. sealed — further restricted subclassing
sealed class Rectangle extends Shape permits Square {
    double width, height;
}
final class Square extends Rectangle {}

// 3. non-sealed — reopens for any subclassing
non-sealed class Triangle extends Shape {}
class EquilateralTriangle extends Triangle {}  // ✅ allowed — Triangle is non-sealed
// class Hexagon extends Shape {} ❌ NOT permitted — not in Shape's permits list
Every permitted subclass MUST be: final, sealed, or non-sealed. The subclass must be in the same package (or same module) as the sealed class.

Sealed Interfaces

// Sealed interface — works the same way
sealed interface Expr permits Num, Add, Mul {}

record Num(int value) implements Expr {}
record Add(Expr left, Expr right) implements Expr {}
record Mul(Expr left, Expr right) implements Expr {}

// Pattern matching switch with sealed types — exhaustive, no default needed!
int eval(Expr e) {
    return switch (e) {
        case Num n  -> n.value();
        case Add a  -> eval(a.left()) + eval(a.right());
        case Mul m  -> eval(m.left()) * eval(m.right());
    };  // no default needed — sealed interface, compiler knows all subtypes
}

Sealed Classes + Pattern Matching — Power Combo

Sealed classes work beautifully with pattern matching in switch — when all permitted subtypes are covered in a switch expression, no default is needed. The compiler can verify exhaustiveness at compile time. This is the main motivator for sealed classes.
Modifier on subclassMeaning
finalNo further subclassing allowed from this class
sealedFurther subclassing allowed but restricted to permits list
non-sealedReopens the class for unrestricted subclassing
Sealed = controlled OOP hierarchy. Together with records and pattern matching, sealed classes complete Java's support for Algebraic Data Types (ADT) — a concept common in functional languages like Haskell and Scala.
