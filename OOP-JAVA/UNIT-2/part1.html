§1 The Idea Behind Exception Handling

What is an Exception?

An exception is an abnormal condition or event that disrupts the normal flow of program execution at runtime. The word comes from "exceptional condition" — something unexpected that the program must handle gracefully.

The Problem Without Exception Handling

// Without exception handling — program crashes abruptly
int[] arr = {1, 2, 3};
System.out.println(arr[10]);   // ArrayIndexOutOfBoundsException!
System.out.println("This never prints");  // dead code
Without exception handling, one runtime error kills the entire program. In real applications (ATMs, hospital systems, web servers), abrupt crashes are unacceptable. We need a way to detect, report, and recover from errors gracefully.

The Java Solution — Structured Exception Handling

  • Separation of concerns: Normal code and error-handling code are kept separate (no if-error chains everywhere).
  • Error propagation: Exceptions bubble up the call stack automatically.
  • Graceful recovery: Catch the exception, handle it, and continue execution.
  • Resource cleanup: finally block ensures resources (files, DB connections) are always closed.
  • Object-oriented: Exceptions are objects — they carry message, stack trace, and cause.

Without vs With Exception Handling

❌ Old approach (C-style) Return error codes (−1, null…)
Caller must check every return value
Easy to forget error checks
Normal code mixed with error code
Hard to read, maintain
✅ Java exception handling Throw exception objects
Exceptions propagate automatically
Compiler forces checked exception handling
Clean separation of logic and error handling
Carries full context (message + stack trace)

Exception as an Object

In Java, an exception is an instance of a class that extends java.lang.Throwable. When an exception occurs, Java creates an object of the relevant exception class and throws it.
Every exception object carries:
  • getMessage() — human-readable description of the error.
  • getClass().getName() — the exception class name.
  • printStackTrace() — full call stack at the point of exception.
  • getCause() — the original exception that caused this one (chained exceptions).
  • toString() — class name + message.

Five Keywords — The Big Picture

KeywordRole
tryEncloses code that might throw an exception.
catchHandles a specific type of exception thrown by the try block.
finallyRuns always — whether or not an exception occurred. Used for cleanup.
throwManually throw an exception object from within code.
throwsDeclares that a method may throw certain exceptions — caller must handle them.
🧪 Real-World Analogy
try = attempt a risky action (e.g., take money from ATM)
catch = handle the problem if it fails (e.g., show "Insufficient funds" instead of crashing)
finally = always return the ATM card, whether successful or not
throw = bank raises an alert (creates and throws an exception)
✓ Exception handling = Try → something goes wrong → Catch → Handle → Finally → Clean up

§2 Exceptions vs Errors & Types

The Throwable Hierarchy

java.lang.Object
  └─ java.lang.Throwable
     ├─ java.lang.Error   (serious, usually unrecoverable)
     │   ├─ OutOfMemoryError
     │   ├─ StackOverflowError
     │   └─ VirtualMachineError
     └─ java.lang.Exception   (recoverable)
         ├─ IOException · SQLException · ClassNotFoundException (Checked)
         └─ RuntimeException (Unchecked)
             ├─ NullPointerException
             ├─ ArrayIndexOutOfBoundsException
             ├─ ArithmeticException
             └─ ClassCastException · NumberFormatException

Exception vs Error

Error Serious system-level problems
Usually caused by JVM / environment
NOT expected to be caught
Cannot recover normally
Examples: OutOfMemoryError, StackOverflowError
Exception Application-level problems
Caused by program logic or input
SHOULD be caught and handled
Recovery is possible
Examples: IOException, NullPointerException
Never catch Error in normal code — these indicate JVM failure. Catching OutOfMemoryError and continuing is almost always wrong. Catch Exception or its specific subclasses instead.

§3 Control Flow & JVM Reaction to Exceptions

Control Flow in Exception Handling

When an exception occurs inside a try block:
1
Execution of the try block stops immediately at the line where the exception occurred. Remaining lines in try are skipped.
2
JVM searches for a matching catch block — one whose parameter type matches the thrown exception (exact match or superclass).
3
If match found → matching catch block executes. Other catch blocks are skipped.
4
If no match found → exception propagates up the call stack to the caller method.
5
finally block executes in ALL cases — after catch (if handled) or before propagation (if not handled).
try {
    System.out.println("A");    // ✅ executes
    int x = 10 / 0;           // ❌ ArithmeticException thrown here
    System.out.println("B");    // ⛔ SKIPPED
} catch (ArithmeticException e) {
    System.out.println("C");    // ✅ executes — match found
} finally {
    System.out.println("D");    // ✅ ALWAYS executes
}
System.out.println("E");       // ✅ executes — after try-catch-finally
// Output: A C D E
Critical: Statements after the exception-throwing line inside try are ALWAYS skipped. The JVM immediately jumps to the catch block — no way to "resume" from where it left off.

Exception Propagation (Call Stack Unwinding)

If a method does not handle an exception (no try-catch), the exception propagates to the calling method. This continues up the call stack until either a handler is found or the exception reaches main(). 
public static void method3() {
    int x = 10 / 0;    // throws ArithmeticException — no handler here
}
public static void method2() {
    method3();          // exception propagates to method2 — no handler
}
public static void method1() {
    try {
        method2();      // exception caught here!
    } catch (ArithmeticException e) {
        System.out.println("Caught in method1: " + e.getMessage());
    }
}
// Call stack unwinds: method3 → method2 → method1 (caught here)

JVM Reaction to Unhandled Exceptions

If an exception propagates all the way to main() without being caught:
  • JVM invokes the default exception handler.
  • It prints the exception name, message, and full stack trace to stderr.
  • The current thread terminates. (In single-threaded programs, the program ends.)
  • In multithreaded programs, only the thread with unhandled exception dies; others continue.
🧪 Output Prediction — Exception Propagation
void A() { B(); } and void B() { throw new RuntimeException("oops"); }
Call: try { A(); } catch(Exception e) { System.out.println(e.getMessage()); }
✓ Output: "oops" — exception propagates from B() → A() → caught in caller.
Key propagation rules:
  • Unchecked exceptions propagate automatically — no declaration needed.
  • Checked exceptions must be declared with throws in each propagating method, or handled with try-catch.

§4 try, catch & finally

try Block

A try block encloses code that might throw an exception. It must be followed by at least one catch or one finally block (or both).

catch Block — Multiple Catches

try {
    String s = null;
    System.out.println(s.length());    // NullPointerException
    int[] a = new int[3];
    a[10] = 5;                         // ArrayIndexOutOfBoundsException
}
catch (NullPointerException e) {
    System.out.println("Null: " + e.getMessage());
}
catch (ArrayIndexOutOfBoundsException e) {
    System.out.println("Array: " + e.getMessage());
}
catch (Exception e) {               // catches any remaining exception
    System.out.println("General: " + e.getMessage());
}
catch ordering rule: Always put more specific exception types before more general ones. If you put catch(Exception e) first, the compiler gives a "exception already caught" error for any specific catches below it.
Order: Specific → General (child before parent)

Multi-catch (Java 7+)

// Instead of two separate catch blocks for same handling:
catch (NullPointerException | ArrayIndexOutOfBoundsException e) {
    System.out.println("Either NPE or ArrayIndex: " + e);
}
// Note: e is implicitly final in multi-catch

finally Block

The finally block always executes — regardless of whether an exception was thrown or caught. Its primary purpose is resource cleanup: closing files, database connections, releasing locks, etc.
When does finally NOT execute?
Only two cases: (1) System.exit() is called inside try/catch. (2) The JVM crashes (power failure, kill signal). In all other cases — including return statements inside try — finally runs. 
// finally with return — tricky exam question!
static int test() {
    try {
        return 1;          // return 1... but wait
    } finally {
        return 2;          // finally runs and overrides the return!
    }
}
// test() returns 2, not 1 — finally's return overrides try's return!

// Resource cleanup pattern:
FileInputStream fis = null;
try {
    fis = new FileInputStream("data.txt");
    // read file...
} catch (IOException e) {
    System.out.println("Error: " + e.getMessage());
} finally {
    if (fis != null) fis.close();   // always close!
}

Try-with-Resources (Java 7+)

A cleaner way to handle resources — automatically closes any AutoCloseable resource at the end of the try block, even if an exception occurs. Eliminates the need for explicit finally blocks for resource cleanup. 
// try-with-resources — file closed automatically
try (FileInputStream fis = new FileInputStream("data.txt");
     BufferedReader br = new BufferedReader(new InputStreamReader(fis))) {
    String line;
    while ((line = br.readLine()) != null) System.out.println(line);
} catch (IOException e) {
    e.printStackTrace();
}
// fis and br are automatically closed — no finally needed!

Nested try-catch

try {                              // outer try
    System.out.println("Outer try");
    try {                          // inner try
        int x = 10 / 0;
    } catch (ArithmeticException e) {
        System.out.println("Inner catch: " + e.getMessage());
        throw e;                   // re-throw to outer
    }
} catch (ArithmeticException e) {
    System.out.println("Outer catch");
} finally {
    System.out.println("Outer finally");
}
// Output: Outer try → Inner catch: / by zero → Outer catch → Outer finally
🧪 Complete the output — classic exam question
try { System.out.print("1 "); int x=1/0; System.out.print("2 "); }
catch(ArithmeticException e){ System.out.print("3 "); }
finally{ System.out.print("4 "); }
System.out.print("5");
✓ Output: 1 3 4 5  — "2" is skipped (after exception), "4" always runs.
Common mistakes with try-catch-finally:
  • Catching Exception first (hides specific catches below it — compile error).
  • Assuming catch runs even without an exception — it doesn't.
  • Forgetting that finally with return overrides try's return.
  • Not closing resources in finally (resource leak).

§5 throw & throws

throw — Explicitly Throwing an Exception

throw is used to manually throw an exception object. You create an exception instance and throw it with the throw keyword. Only one exception can be thrown at a time.
Syntax: throw new ExceptionType("message"); 
public static void validateAge(int age) {
    if (age < 18) {
        throw new IllegalArgumentException("Age must be 18+, got: " + age);
    }
    System.out.println("Valid age: " + age);
}

public static void main(String[] args) {
    try {
        validateAge(15);
    } catch (IllegalArgumentException e) {
        System.out.println("Error: " + e.getMessage());
    }
}
// Output: Error: Age must be 18+, got: 15
After throw: Execution of the current method stops immediately. Any code after the throw statement in the same block is unreachable. The compiler will warn about unreachable statements.

throws — Declaring Exceptions

throws is used in a method signature to declare that the method may throw certain checked exceptions. The caller of that method must then either handle or re-declare those exceptions.
Syntax: returnType methodName(params) throws ExceptionType1, ExceptionType2 
// Method declares it may throw IOException
public void readFile(String path) throws IOException {
    FileReader fr = new FileReader(path);    // may throw FileNotFoundException
    // ... read ...                           // may throw IOException
}

// Caller MUST handle it
try {
    readFile("test.txt");
} catch (IOException e) {
    System.out.println("File error: " + e.getMessage());
}

// OR: caller can also declare and propagate
public void processFile() throws IOException {
    readFile("test.txt");   // propagate, not handle
}

Re-throwing exceptions: You can catch an exception and then throw it again (or wrap it in a new exception). Wrapping preserves the original cause — useful for logging then re-raising. 
catch (SQLException e) {
    System.out.println("DB error logged");
    throw new RuntimeException("Database unavailable", e); // e = cause

// getCause() on the new exception returns original SQLException
🧪 throw vs throws — Which is correct?
void m() throws new Exception(); → ❌ Wrong — throws takes class names, not instances.
void m() throws Exception { throw new Exception("err"); } → ✅ Correct.
throw Exception; → ❌ Wrong — must be an object: throw new Exception();
✓ throw = instance · throws = class name. Remember this distinction!
Unchecked exceptions don't NEED throws: You can throw RuntimeException (and its subclasses) without declaring throws. But you CAN declare them — it just documents intent.

§6 Built-in, User-Defined, Checked & Unchecked

Checked vs Unchecked Exceptions

Checked Exceptions Subclass of Exception (but NOT RuntimeException)
Checked by compiler at compile time
MUST be handled (try-catch) or declared (throws)
Represent recoverable conditions
Examples: IOException, SQLException, ClassNotFoundException
Unchecked Exceptions Subclass of RuntimeException
NOT checked by compiler
Handling is optional (no compile error)
Represent programming bugs
Examples: NullPointerException, ArithmeticException, ArrayIndexOutOfBoundsException
The rule: If you call a method that throws a checked exception and you don't catch it or declare throws, the compiler gives an error: "Unhandled exception type IOException."
For unchecked exceptions — no such obligation.
Classification summary:
Throwable → Error (unchecked) + Exception
Exception → RuntimeException (unchecked) + everything else (checked)
Mnemonic: RE is UnChecked, everything else in Exception is Checked
Design debate: Some argue checked exceptions lead to "exception swallowing" (empty catch blocks). Java 8+ style prefers unchecked exceptions. But in standard exams: know the rule — checked = must handle or declare.

User-Defined (Custom) Exceptions

You can create your own exception classes by extending Exception (for checked) or RuntimeException (for unchecked). This lets you represent domain-specific error conditions meaningfully. 
// 1. Define a custom checked exception
class InsufficientFundsException extends Exception {
    private double amount;

    public InsufficientFundsException(double amount) {
        super("Insufficient funds. Needed: " + amount);
        this.amount = amount;
    }

    public double getAmount() { return amount; }
}

// 2. Use it in a class
class BankAccount {
    private double balance = 500;

    public void withdraw(double amount) throws InsufficientFundsException {
        if (amount > balance) {
            throw new InsufficientFundsException(amount - balance);
        }
        balance -= amount;
        System.out.println("Withdrawn: " + amount + ". Balance: " + balance);
    }
}

// 3. Handle it
try {
    new BankAccount().withdraw(1000);
} catch (InsufficientFundsException e) {
    System.out.println(e.getMessage());           // Insufficient funds. Needed: 500.0
    System.out.println("Need more: " + e.getAmount());
}

Best Practices for Custom Exceptions

1
Name should end with "Exception": InvalidAgeException, ProductNotFoundException.
2
Always provide at least two constructors: no-arg and a message constructor. Ideally also a cause constructor.
3
Extend Exception for conditions the caller should handle. Extend RuntimeException for programmer errors.
4
Add domain-specific fields (like amount in InsufficientFundsException) for richer error context.

Custom Unchecked Exception

// Unchecked — no need to declare with throws
class InvalidAgeException extends RuntimeException {
    public InvalidAgeException(String msg) { super(msg); }
    public InvalidAgeException(String msg, Throwable cause) { super(msg, cause); }
}

void setAge(int age) {    // no throws declaration needed!
    if(age < 0) throw new InvalidAgeException("Age cannot be negative: " + age);
}
🧪 Design Question — Checked or Unchecked?
"File not found" → Checked — caller should know and handle this (maybe ask user for new path).
"Array index -1" → Unchecked — it's a bug in the code that should be fixed, not caught.
"Database connection failed" → Checked — external condition, caller must handle/retry.
✓ Rule of thumb: If the caller can reasonably handle it → Checked. If it's a programmer bug → Unchecked.
Exception chaining / wrapping: Use the Throwable cause constructor to preserve original exception when wrapping: throw new AppException("high-level msg", originalException). Use getCause() to retrieve original.
