§11 Thread & Thread Life Cycle

What is a Thread?

A thread is the smallest unit of execution within a process. Multiple threads can run concurrently within the same program, sharing the same memory space (heap, static variables) but each having its own stack and program counter.
Process vs Thread:
A process has its own memory space (address space). Heavy to create.
A thread lives inside a process and shares its memory. Lightweight to create.
Java programs are multi-threaded by default — even a simple "Hello World" has at least 2 threads: main and garbage collector.

Why Multithreading?

  • Concurrency: Multiple tasks appear to execute simultaneously (on single CPU via context switching, truly parallel on multi-core).
  • Responsiveness: UI thread stays responsive while background thread downloads data.
  • Resource utilization: CPU stays busy during I/O waits.
  • Performance: Parallelism on multi-core processors speeds up computation.
  • Examples: web server handling thousands of requests, music player, IDEs, games.

Thread Life Cycle (States)

NEW

There is no RUNNING state in Java's Thread.State enum. The JVM uses RUNNABLE for both "ready to run" and "currently running". The OS scheduler decides which RUNNABLE thread actually gets CPU time.

Daemon vs User Threads

User Thread Default type of thread
JVM waits for ALL user threads to finish
Keeps application alive
Examples: main thread, worker threads
Daemon Thread Background service thread
JVM exits when only daemon threads remain
Created: t.setDaemon(true)
Examples: Garbage Collector, JIT compiler thread 
Thread t = new Thread(() -> {
    while(true) System.out.println("Background work...");
});
t.setDaemon(true);   // must set BEFORE start()
t.start();
// When main() ends, JVM exits even though daemon is running

Thread.sleep() and InterruptedException

try {
    System.out.println("Sleeping for 2 seconds...");
    Thread.sleep(2000);           // pauses current thread for 2 sec
    System.out.println("Awake!");
} catch (InterruptedException e) {
    System.out.println("Sleep interrupted!");
    Thread.currentThread().interrupt();  // restore interrupt flag — best practice
}

// join() example — wait for thread to complete
Thread worker = new Thread(() -> {
    try { Thread.sleep(1000); } catch(InterruptedException e){}
    System.out.println("Worker done");
});
worker.start();
worker.join();            // main waits here until worker finishes
System.out.println("Main continues after worker");
Never call run() directly! Calling t.run() instead of t.start() just runs the code in the CURRENT thread — no new thread is created. This is a very common exam trap.

§12 Creating Threads

Method 1 — Extending Thread Class

class MyThread extends Thread {
    private String taskName;

    MyThread(String name) {
        super(name);           // sets thread name
        this.taskName = name;
    }

    @Override
    public void run() {       // thread logic goes here
        for(int i = 1; i <= 5; i++) {
            System.out.println(taskName + ": step " + i);
            try { Thread.sleep(200); } catch(InterruptedException e){}
        }
    }
}

// Creating and starting threads
MyThread t1 = new MyThread("Task-A");
MyThread t2 = new MyThread("Task-B");
t1.start();   // starts new thread — runs concurrently with t2
t2.start();
Limitation of extending Thread: Java allows only single inheritance. If your class already extends another class, you CANNOT also extend Thread. Use the Runnable approach instead.

Method 2 — Implementing Runnable Interface

class MyTask implements Runnable {
    @Override
    public void run() {
        System.out.println("Running: " + Thread.currentThread().getName());
    }
}

// Pass Runnable to Thread constructor
Runnable task = new MyTask();
Thread t = new Thread(task, "Worker-1");
t.start();

// Anonymous class shorthand
Thread t2 = new Thread(new Runnable() {
    public void run() { System.out.println("Anonymous thread"); }
});
t2.start();

// Lambda (Java 8+) — cleanest way
Thread t3 = new Thread(() -> System.out.println("Lambda thread!"));
t3.start();

Thread Class vs Runnable — Comparison

extends Thread Simple to write

Each object IS a thread
Use when no other superclass needed

Best practice: Prefer implements Runnable (or lambda). It promotes separation of concerns — the task logic (Runnable) is independent of the thread execution mechanism. Also works with ExecutorService.

Method 3 — Callable and Future (Java 5+)

import java.util.concurrent.*;

// Callable — like Runnable but can return a value and throw checked exceptions
Callable<Integer> sumTask = () -> {
    int sum = 0;
    for(int i = 1; i <= 100; i++) sum += i;
    return sum;
};

ExecutorService executor = Executors.newSingleThreadExecutor();
Future<Integer> future = executor.submit(sumTask);

// Do other work while sumTask runs in background...
System.out.println("Waiting for result...");

int result = future.get();     // blocks until result is ready
System.out.println("Sum = " + result);   // 5050
executor.shutdown();
🧪 Key output question — Thread creation
Thread t = new Thread(() -> System.out.println("A")); t.run();
This prints "A" — but in the MAIN thread, not a new thread!
Change to t.start() to run in a new thread.
✓ t.run() = no new thread. t.start() = new thread created and run() called inside it.

§13 Thread Priorities

Priority Constants & Range

Every Java thread has a priority — an integer from 1 (lowest) to 10 (highest). The thread scheduler (OS-level) uses priority as a hint to decide which RUNNABLE thread to execute next.
ConstantValueMeaning
Thread.MIN_PRIORITY1Lowest priority
Thread.NORM_PRIORITY5Default priority (main thread)
Thread.MAX_PRIORITY10Highest priority
Thread high = new Thread(() -> System.out.println("High priority"));
Thread low  = new Thread(() -> System.out.println("Low priority"));

high.setPriority(Thread.MAX_PRIORITY);   // 10
low.setPriority(Thread.MIN_PRIORITY);    // 1

System.out.println(high.getPriority()); // 10
high.start();
low.start();

// New thread inherits parent's priority by default
Thread child = new Thread(() -> {});
// child.getPriority() == Thread.currentThread().getPriority()
Priority is just a HINT! The JVM does not guarantee that higher-priority threads always run before lower ones. This is entirely platform-dependent (Windows, Linux, Mac handle it differently). Never rely on priorities for correctness — use synchronization mechanisms instead.
Starvation: If a very low-priority thread never gets CPU time because high-priority threads always preempt it, this is called thread starvation. Modern schedulers use aging (gradually increase priority of waiting threads) to prevent this.

§14 Synchronizing Threads

The Problem — Race Condition

A race condition occurs when two or more threads access shared mutable data simultaneously and the outcome depends on the order of execution. This leads to inconsistent / corrupted data. 
// Classic race condition — two threads increment counter
class Counter {
    int count = 0;
    void increment() { count++; }   // NOT atomic: read → add → write (3 steps!)
}

Counter c = new Counter();
// Thread A and Thread B both call c.increment() 1000 times each
// Expected: count = 2000 | Actual: could be 1400, 1876, ... (unpredictable)

synchronized — The Solution

The synchronized keyword ensures that only one thread at a time can execute a block or method on a given object. Internally, it uses a monitor lock (intrinsic lock) associated with every object. 
// Method-level synchronization
class SafeCounter {
    private int count = 0;
    synchronized void increment() { count++; }    // only one thread at a time
    synchronized int getCount() { return count; }
}

// Block-level synchronization (finer-grained)
class BankAccount {
    private double balance;
    void transfer(BankAccount to, double amount) {
        synchronized(this) {         // lock on THIS object
            balance -= amount;
        }
        synchronized(to) {            // lock on TARGET object
            to.balance += amount;
        }
    }
}

// Static synchronized — uses Class object as lock
static synchronized void sharedMethod() { ... }

How Monitor Lock Works

1
Thread A enters a synchronized method/block → acquires the monitor lock of the object.
2
Thread B tries to enter the same synchronized method → sees lock is held → moves to BLOCKED state.
3
Thread A finishes → releases the lock.
4
Thread B acquires lock → moves to RUNNABLE → executes.

Deadlock

Deadlock occurs when two or more threads are blocked forever, each waiting for the other to release a lock. 
// Classic deadlock scenario
Object lock1 = new Object(), lock2 = new Object();

Thread A: synchronized(lock1) { synchronized(lock2) { ... } }
Thread B: synchronized(lock2) { synchronized(lock1) { ... } }
// A holds lock1, waits for lock2
// B holds lock2, waits for lock1 → DEADLOCK!

// Prevention: always acquire locks in the same ORDER
Thread A: synchronized(lock1) { synchronized(lock2) { ... } }
Thread B: synchronized(lock1) { synchronized(lock2) { ... } }
// Both threads try lock1 first → one waits, other proceeds → no deadlock
Four conditions for deadlock (Coffman):
  • Mutual exclusion — only one thread can hold a lock at a time.
  • Hold and wait — thread holds one lock while waiting for another.
  • No preemption — locks cannot be forcibly taken from a thread.
  • Circular wait — T1 waits for T2's lock, T2 waits for T1's lock.
Break any one condition to prevent deadlock.

volatile keyword

volatile ensures that reads and writes to a variable are always done directly from/to main memory, not from the thread's local CPU cache. Prevents visibility issues (where one thread's write is invisible to another thread). 
class StopFlag {
    private volatile boolean stop = false;

    void run() {
        while (!stop) { /* do work */ }   // always sees latest value of stop
    }

    void halt() { stop = true; }         // write immediately visible to run()
}
volatile vs synchronized:
volatile — only guarantees visibility (threads see latest value). Does NOT guarantee atomicity.
synchronized — guarantees both visibility AND atomicity (exclusive access).
volatile is good for simple flags; use synchronized or AtomicInteger for compound operations (read-modify-write).

java.util.concurrent — Thread-safe Utilities

The java.util.concurrent package (Java 5+) provides higher-level thread-safe utilities — prefer these over raw synchronized for new code.

§15 Inter-thread Communication

The Problem — Polling

Without inter-thread communication, a thread must poll (repeatedly check) a condition in a loop — wasting CPU. Java provides wait(), notify(), and notifyAll() for efficient coordination.

wait(), notify(), notifyAll()

MethodWhat it doesConstraint
wait()Releases lock and waits (WAITING state) until notifiedMust be inside synchronized block
wait(ms)Waits at most ms millisecondsMust be inside synchronized block
notify()Wakes up ONE thread waiting on this object's monitorMust be inside synchronized block
notifyAll()Wakes up ALL threads waiting on this object's monitorMust be inside synchronized block
Critical: wait(), notify(), notifyAll() are methods of Object (not Thread!) and must be called from within a synchronized block/method on the same object. Calling outside synchronized → IllegalMonitorStateException.

Producer-Consumer Pattern

class SharedBuffer {
    private int data;
    private boolean hasData = false;

    synchronized void produce(int value) throws InterruptedException {
        while (hasData) wait();    // wait if buffer is full
        data = value;
        hasData = true;
        System.out.println("Produced: " + value);
        notify();                    // wake up consumer
    }

    synchronized int consume() throws InterruptedException {
        while (!hasData) wait();   // wait if buffer is empty
        hasData = false;
        System.out.println("Consumed: " + data);
        notify();                    // wake up producer
        return data;
    }
}

// Running Producer-Consumer
SharedBuffer buf = new SharedBuffer();

Thread producer = new Thread(() -> {
    for(int i = 1; i <= 5; i++) {
        try { buf.produce(i); } catch(InterruptedException e){}
    }
});

Thread consumer = new Thread(() -> {
    for(int i = 1; i <= 5; i++) {
        try { buf.consume(); } catch(InterruptedException e){}
    }
});

producer.start();
consumer.start();
Always use while() not if() with wait()! A thread can spuriously wake up (wake without being notified) — rare but possible. Using while(condition) wait(); re-checks the condition after waking, preventing spurious wakeup bugs. This is a classic exam question!

join() — Thread Coordination

Thread[] workers = new Thread[3];
for(int i = 0; i < 3; i++) {
    final int id = i;
    workers[i] = new Thread(() -> {
        System.out.println("Worker " + id + " done");
    });
    workers[i].start();
}
// Main waits for ALL workers to finish
for(Thread t : workers) t.join();
System.out.println("All workers done. Aggregating results.");
🧪 notify() vs notifyAll() — when to use which?
notify() — wakes ONE arbitrary waiting thread. Use when all waiting threads do the same thing and any one can proceed.
notifyAll() — wakes ALL waiting threads. Use when different threads wait for different conditions (safer default).
✓ When in doubt, use notifyAll() — it's always correct (just potentially slower). notify() can cause missed signals in complex scenarios.

§16 ⚡ Quick Revision — Exception Handling

5 Keywords — One-Liners

try
Wraps risky code. Must be followed by catch or finally (or both).
catch
Handles specific exception. Must come after try. Specific → General order.
finally
ALWAYS runs. Even if return in try. Not if System.exit() called.
throw
Throw an instance: throw new Ex("msg");. Stops execution immediately.
throws
Method signature declaration: void f() throws IOException. Only for checked.
Exception vs Error
Exception = recoverable, catch it. Error = JVM-level, don't catch it.

Checked vs Unchecked — Rule

RuntimeException (and subclasses) = Unchecked — compiler doesn't force handling.
Everything else extending Exception = Checked — must catch or declare throws.
Error subclasses = Unchecked — serious JVM failures, don't catch.
Mnemonic: "RuntimeException is Unchecked; all other Exception subclasses are Checked."

Exception Handling Cheatsheet

NullPointerExceptionAccessing method/field on null. Unchecked.
ArrayIndexOutOfBoundsIndex <0 or ≥length. Unchecked.
ArithmeticExceptionDivision by zero. Unchecked.
NumberFormatExceptionparseInt("abc"). Unchecked.
IOExceptionFile/network errors. Checked — must handle!
User-definedextend Exception (checked) or RuntimeException (unchecked).
Multi-catch (Java 7)catch(A | B e) { } — e is implicitly final.
try-with-resourcestry(Stream s = ...) { } — auto-close. Best practice.

Synchronization & Inter-thread — Flash Cards

Race Condition
Two threads modify shared data simultaneously → corrupted result. Use synchronized.
synchronized
Only one thread at a time. Method or block level. Uses object's monitor lock.
volatile
Guarantees visibility (no CPU cache). Does NOT guarantee atomicity.
Deadlock
Threads wait for each other's locks forever. Prevent by consistent lock ordering.
wait()
Releases lock + enters WAITING. Must be in synchronized. Object method!
notify()
Wakes ONE waiting thread. notifyAll() wakes ALL. Must be in synchronized.

Top 10 Multithreading Exam Traps

#TrapCorrect Fact
1t.run() creates new thread❌ t.start() creates new thread. t.run() runs in current thread.
2sleep() releases lock❌ sleep() keeps the lock held. wait() releases the lock.
3wait() is in Thread class❌ wait() is in Object class. Every object has it.
4Higher priority always runs first❌ Priority is a hint. Not guaranteed by JVM.
5synchronized prevents deadlock❌ Synchronization can CAUSE deadlock if lock order is inconsistent.
6volatile = synchronized❌ volatile = visibility only. synchronized = visibility + atomicity.
7notify() wakes specific thread❌ notify() wakes an arbitrary waiting thread — no control over which.
8Use if() with wait()❌ Always use while() with wait() to handle spurious wakeups.
9Can restart terminated thread❌ Once TERMINATED, a thread cannot be restarted. Create new Thread.
10setDaemon() after start() works❌ setDaemon() must be called BEFORE start(). After = IllegalThreadStateException.
