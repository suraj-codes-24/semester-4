§1 What is an OS & Its Functions

An Operating System (OS) is a program that acts as an intermediary between the user and the computer hardware. It manages the computer's hardware resources and provides a platform for application programs to run.

It serves as an interface between: User ↔ Hardware, Application Program ↔ Hardware, and Program ↔ Program.

Goals of an Operating System

  • Convenience: Make the computer easier to use for the end user
  • Efficiency: Use computer hardware resources in an efficient manner
  • Ability to Evolve: Allow development & testing of new functions without disrupting service
  • Execute Programs: Provide an environment where user programs can run and problems can be solved

Computer System Components

A computer system has four main components:

ComponentDescriptionExamples
HardwareProvides basic computing resourcesCPU, Memory, I/O devices
Operating SystemControls & coordinates hardware use among applicationsWindows, Linux, macOS
Application ProgramsDefine how resources are used to solve user problemsWord processors, browsers, compilers
UsersPeople, machines, or other computersEnd users, automated systems
Two views of OS:
🔷 Resource Allocator — Manages and allocates all resources (CPU time, memory, I/O, file storage). Decides between conflicting requests for efficient & fair use.
🔷 Control Program — Controls execution of programs to prevent errors and improper use of the computer; especially controls I/O device operations.

Kernel & Shell

Kernel: The one program running at all times on the computer. It is the core of the OS, directly managing hardware resources. Everything else is either a system program or an application program.

Shell: An interface between the user and the kernel. Application programs are started from the shell. Examples: bash, sh, zsh (Linux), cmd, PowerShell (Windows).

Kernel Services

  • I/O Management
  • Process Management
  • Memory Management
  • File System Management
  • Device Drivers Management
  • System Calls (API)
  • Interprocess Communication
  • Protection System
  • Networking

OS Interfaces: ISA, ABI, API

InterfaceFull FormBetween
ISAInstruction Set ArchitectureHardware & OS
ABIApplication Binary InterfaceOS & compiled programs
APIApplication Programming InterfaceOS & application source code

Evolution of OS — Generations

1st Gen (1945–55)Vacuum Tubes · Plugboards · No OS
2nd Gen (1955–65)Transistors · Batch Systems
3rd Gen (1965–80)ICs · Multiprogramming
4th Gen (1980–now)LSI · Personal Computers
Common exam question: "What are the two main roles of an OS?" → Answer: Resource Allocator and Control Program.
Don't confuse the Kernel (always running) with the Shell (user interface to kernel). The kernel IS the OS core; the shell is just a program on top of it.
ISA vs ABI vs API — frequently asked as a short note. ISA = hardware level, ABI = binary level, API = source code level.

§2 Batch Systems & Spooling

A Batch System is an OS where similar jobs are collected (batched) together and executed as a group without user interaction. An operator (not the user) manages job submission. The CPU often sits idle because mechanical I/O devices (e.g. punch card readers) are much slower than the CPU.

Simple Batch System

How it works:
1. Jobs submitted by users to an operator on punch cards
2. Operator sorts jobs into batches of similar type
3. Batch loaded onto input device (card reader)
4. OS runs jobs one by one using automatic job sequencing
5. Output printed; next job begins automatically

Problem: CPU is idle during I/O operations (I/O is very slow compared to CPU speed)

Spooling — Overlapping I/O with Computation

SPOOL = Simultaneous Peripheral Operation On-Line
Jobs are NOT read directly into memory but first written onto a disk (job queue). While one job is executing, the OS simultaneously:
→ Reads the next job from the card reader to disk
→ Sends the previous job's output from disk to the printer
Goal of Spooling: Overlap I/O of one job with CPU computation of another job, thereby reducing CPU idle time.

CPU utilization ↑ because I/O of Job N+1 happens while Job N is computing

Multiprogrammed Batch System

Multiprogramming: Several jobs are kept in main memory simultaneously (called the job pool). The OS picks one job to execute. When that job must wait (e.g., for I/O), the OS switches to another ready job. As long as jobs exist, the CPU is never idle.
Memory Layout: Memory is divided into OS area + multiple job partitions (Job 1, Job 2, Job 3, Job 4 …). All loaded jobs are ready to run — OS selects among them.

Key difference from simple batch: In simple batch, only ONE job is in memory. In multiprogrammed batch, MULTIPLE jobs are in memory simultaneously.

Features Required for Multiprogramming

  • Memory Management: OS must allocate memory to several jobs simultaneously and track their locations
  • CPU Scheduling: OS must choose which of the ready jobs to run next (job scheduling)
  • I/O Routine & Device Management: OS must safely manage allocation of I/O devices to jobs
  • Protection: Jobs must be isolated — one job must not interfere with another's memory

Comparison: Simple vs Multiprogrammed Batch

FeatureSimple BatchMultiprogrammed Batch
Jobs in memoryOne at a timeMultiple simultaneously
CPU when job waitsCPU sits idleCPU switches to next job
CPU utilizationLowHigh
ComplexitySimpleComplex (needs scheduling)
User interactionNoneNone (still batch)
ExampleEarly IBM systemsIBM OS/360

User Mode vs Kernel Mode (introduced in Batch era)

User Mode Programs run with restricted access. Cannot directly access hardware. If a protected instruction is attempted → trap to OS.
Kernel Mode OS (monitor) runs here. Has full access to all hardware and privileged instructions. Switches to user mode when running a job.
Exam tip: SPOOL full form is always asked — Simultaneous Peripheral Operation On-Line.
Remember: Spooling ≠ Buffering. Spooling uses disk as an intermediate; buffering uses memory.
Multiprogramming ≠ Multiprocessing. Multiprogramming = multiple programs sharing ONE CPU. Multiprocessing = multiple CPUs.

§3 Interactive & Time-Sharing Systems

An Interactive System allows direct communication between the user and the OS during program execution. The user can provide input, see output, and modify execution in real time — unlike batch systems where jobs run unattended.
A Time-Sharing (Multitasking) System uses CPU scheduling + multiprogramming to give each user a small slice of CPU time (called a time quantum). The CPU is switched among multiple jobs so rapidly that each user feels they have exclusive use of the machine.

How Time-Sharing Works

Mechanism:
1. Multiple users are logged in simultaneously
2. Each job is given a small time slice (quantum) of CPU time
3. After the quantum expires, the job is preempted (interrupted)
4. CPU moves to the next job's quantum
5. If a job is too large to fit in memory → it is swapped to disk and brought back when its turn comes
6. On-line communication between user and system is maintained at all times

Key concept: A program loaded into memory and currently executing = a Process

Key Features of Time-Sharing OS

  • CPU is multiplexed among several jobs in memory and on disk
  • CPU allocated to a job only while that job is in memory
  • Compensates for slow I/O (e.g., slow keyboard typing doesn't waste CPU)
  • Allows many users to share a computer simultaneously
  • Response time is very short (typically < 1 second)
  • Each user has the illusion of a dedicated machine
Response Time Goal: Time-sharing systems aim to keep response time short enough that users can interact with the system. Batch systems focus on maximizing throughput; time-sharing systems focus on minimizing response time.

Personal Computer (Single-Processor) Systems

A personal computer is a system dedicated to a single user. It borrows technology from large OS systems but is optimized for convenience. Examples: Windows PC, MacBook, Linux desktop. Peripherals: keyboard, mouse, screen, small printer. Can run different OS types — Windows, macOS, UNIX, Linux.

Batch vs Time-Sharing: Full Comparison

User interactionNone (submit & wait)Direct, interactive
Response timeHours/daysSeconds (<1s ideally)
Primary goalMaximize throughputMinimize response time
CPU schedulingJob scheduling (FCFS)Round Robin (time quantum)
Memory usageOne or more jobs loadedMany jobs; swapping used
Suitable forPayroll, billing, data processingEditing, compiling, querying

Key Term: Process

A Process is a program loaded into memory that is currently executing. It is an active entity with:
→ An executable program (text section)
→ Associated data (data section, heap, stack)
→ An execution context (program counter, registers, process state)
Exam question: "Differentiate between batch and time-sharing OS" — always expected in exams.
Key difference: Batch = no user interaction, focuses on throughput. Time-sharing = interactive, focuses on response time.
Remember: Time-sharing is also called multitasking. The term "time quantum" or "time slice" is the CPU time allotted to each process.

§4 Real-Time Systems

A Real-Time Operating System (RTOS) is an OS with well-defined, fixed time constraints. Processing must be completed within these defined constraints — a result is only correct if it is produced within the deadline. If timing constraints are not met, the system fails.

Types of Real-Time Systems

Hard Real-Time Missing a deadline = system FAILURE. Safety-critical systems.

Examples:
→ Pacemakers
→ Airbag controllers
→ Nuclear reactor control
→ Aircraft flight control
→ Anti-lock braking systems (ABS)
Soft Real-Time Missing a deadline = degraded performance, but NOT catastrophic failure.

Examples:
→ Video streaming / DVD players
→ Multimedia applications
→ Online gaming
→ Video conferencing
→ Smartphone apps

Real-Time Embedded Systems

Embedded systems run real-time operating systems. They are typically:
→ Dedicated to a specific task (single purpose)
→ Have no user interface (or very minimal UI)
→ Perform hardware monitoring and control
→ Must respond to sensor data within a very small time interval

Examples: Car engines, manufacturing robots, DVDs, microwave ovens, firewalls, industrial control systems, medical imaging systems, scientific experiments
Key constraint: Data brought in by sensors must be analyzed and corresponding control adjustments must be applied within a fixed small time interval — this is the defining characteristic of RTOS.

Characteristics of a Real-Time OS

  • Determinism: Operations are performed at fixed, predetermined times or within predetermined time intervals
  • Responsiveness: After being interrupted, OS acknowledges and services the interrupt quickly
  • User Control: User has fine-grained control over task priority and scheduling
  • Reliability: Must be highly reliable — used in safety-critical environments
  • Fail-Safe: Ability to maintain safe operation even when components fail

Mobile Real-Time Systems

Mobile / Handheld Systems (PDAs, smartphones, tablets) face unique constraints:
→ Limited memory → Slow processors → Small displays
→ Connect via IEEE 802.11 wireless or cellular networks
→ Leaders: Apple iOS and Google Android
→ Android = Linux-based, includes Dalvik VM, Activity Manager, Window Manager

Hard vs Soft Real-Time: Quick Summary

PropertyHard Real-TimeSoft Real-Time
Deadline missSystem failure / catastropheDegraded quality only
PredictabilityMust be 100% deterministicStatistical guarantees okay
Virtual memoryUsually NOT allowed (unpredictable)May be allowed
Response timeMicroseconds to millisecondsMilliseconds to seconds
Use caseSafety-critical controlMultimedia, entertainment

Remember: Real-time ≠ Fast. Real-time means predictable and timely, not necessarily super fast.
Embedded systems always run RTOS — this is a standard fact asked in exams.
Virtual memory is generally avoided in hard real-time systems because page faults are unpredictable.

§5 Computer Startup & Booting

Bootstrap Program: A small piece of code loaded into memory at power-up or reboot. Stored in ROM or EEPROM (Electrically Erasable Programmable ROM), commonly known as firmware. Its job: initialize all system aspects and load the OS kernel into memory to begin execution.

Complete Boot Sequence

1
Power On Reset (POR): An electrical pulse is sent to the CPU. All CPU registers are initialized to 0 except CS = 0xF000 and IP = 0xFFF0.
Physical address = (CS << 4) + IP = 0xFFFF0 → CPU fetches first instruction from this location.
2
Real Mode: CPU starts in real mode (backward compatible to 8088). Limited to 1MB address space. No protection, no privilege levels, no multitasking. First instruction at 0xFFFFF0 is a jump into BIOS ROM.
3
BIOS (Basic Input/Output System): Stored in Flash/EPROM chip. Performs the following in order:
→ POST (Power-On Self Test) — tests hardware
→ Initialize video card & other devices
→ Display BIOS screen
→ Perform memory test & set DRAM parameters
→ Configure Plug & Play devices, assign DMA channels & IRQs
→ Identify boot device → Read sector 0 from boot device into memory at 0x7C00 → Jump to 0x7C00
4
MBR (Master Boot Record): Sector 0 of the disk. Loaded by BIOS to RAM at address 0x7C00.
MBR Structure (512 bytes total):
→ 446 bytes — Bootable code
→ 64 bytes — Disk partition table (16 bytes × 4 partitions)
→ 2 bytes — Signature (0x55AA)
MBR code looks through partition table → loads the bootloader.

5
Bootloader: Loads the Operating System. May present a menu to choose which OS to load (e.g., GRUB for Linux/Windows dual boot). Also performs:
→ Disable interrupts (don't want interruptions at this critical stage)
→ Setup GDT (Global Descriptor Table)
→ Switch CPU from Real Mode → Protected Mode (full memory access, privilege levels enabled)
→ Read OS from disk into memory
6
OS Initialization: The OS kernel takes over and performs:
→ Set up virtual memory
→ Initialize interrupt vectors
→ Initialize timers, monitors, hard disks, consoles, filesystems
→ Initialize other processors (if multiprocessor system)
→ Start the first user process (e.g., init or systemd on Linux)
