§1 Multiprocessor Systems

Multiprocessor System (Parallel System) — A system with more than one CPU in close communication. Also called a tightly coupled system because processors share the same main memory, I/O devices, and other resources. Communication between processors happens through the shared memory bus.

Advantages of Multiprocessor Systems

  • Increased Throughput: More work is done in the same time. However, increasing processors by N does not decrease time by N due to overhead in coordination
  • Economy of Scale (Economical): Shared peripherals (disk, printer) and power supplies. Multiple programs can run on the same data set on a shared disk
  • Increased Reliability: Failure of one processor does not halt the entire system — remaining processors continue working
  • Graceful Degradation: Ability to continue providing service proportional to the level of surviving hardware (also called fail-soft system)
Fault Tolerance Types (from Stallings):
→ Spatial Redundancy: Multiple copies of hardware components
→ Temporal Redundancy: Repeat the operation if it fails
→ Information Redundancy: Add extra bits (like checksums) to allow error detection/correction

Types of Multiprocessing

Asymmetric Multiprocessing (AMP) One master processor controls the system and schedules tasks to slave processors. Each slave is assigned a specific task. More common in extremely large systems. Simple to implement but the master can become a bottleneck.
Symmetric Multiprocessing (SMP) All processors are peers — no master/slave relationship. Each processor can perform ALL OS tasks. All processors share the same memory and I/O. Most modern operating systems support SMP. Example: Modern Linux, Windows Server.

SMP Architecture (in detail)

In SMP, each CPU has its own set of registers and its own cache, but all CPUs share the same main memory. The structure looks like:

CPU₀ [registers + cache] ─┐
CPU₁ [registers + cache] ─┼── Shared Memory Bus ── RAM
CPU₂ [registers + cache] ─┘

All CPUs can run any OS process. The OS must synchronize access to shared data structures (ready queue, memory allocator, etc.) to avoid race conditions.

Multicore Systems

A multicore processor has multiple computing cores on a single chip. Each core has its own registers and cache but shares the chip-level cache and memory bus.

Advantage over multi-chip: On-chip communication is faster than between-chip communication. Also more power-efficient and cheaper to manufacture.
Dual-Core Design: Two cores on the same chip — CPU core₀ and CPU core₁, each with registers and cache, both connected to shared memory. Modern processors are quad-core (4), octa-core (8), or more.

AMP vs SMP Comparison

CPU Throughput Calculation

🧪 Example: Throughput Speedup
A single CPU system processes 100 jobs per hour. A 4-processor SMP system is used. Assuming 80% efficiency (due to coordination overhead), what is the throughput?
Ideal throughput = 4 × 100 = 400 jobs/hour
Actual throughput = 400 × 0.80 = 320 jobs/hour
Speedup = 320 / 100 = 3.2× (not 4× due to overhead)
✓ This illustrates why "increasing N processors does NOT decrease time by exactly N"
🧪 Example: Graceful Degradation
A 4-processor fail-soft system is running. Suddenly 1 processor fails. How does a graceful degradation system behave?
System does NOT crash entirely.
Remaining 3 processors take over — system continues to run at 75% capacity (3/4 of original power).
This is called graceful degradation — service continues proportional to surviving hardware.
✓ Graceful degradation = partial failure → partial service (not complete failure)
Most asked in exams: "What is SMP? Advantages of multiprocessor systems?"
Remember: Tightly coupled = shared memory (multiprocessor). Loosely coupled = separate memory (distributed).
Graceful degradation vs fail-safe: Graceful degradation continues at reduced capacity. Fail-safe ensures safe shutdown.
Multicore ≠ Multiprocessor: Multicore = multiple cores on ONE chip. Multiprocessor = multiple separate chips.

§2 Multiuser Systems

A Multiuser System is an OS that allows multiple users to access the same computer system simultaneously. Each user appears to have dedicated resources, but actually shares the CPU, memory, and I/O through time-sharing and resource management techniques.

Key Characteristics

  • Simultaneous Access: Multiple users log in and work at the same time via terminals or network connections
  • Resource Sharing: CPU, memory, disk, and peripherals are shared among all users
  • User Isolation: Each user's data and processes are protected from other users
  • Authentication: Each user must log in with credentials (username/password) — OS manages user accounts
  • Fair CPU Allocation: Time-sharing ensures each user gets a fair share of CPU time
  • File Permissions: Users can protect their files from being read or modified by other users

Multiuser OS Requirements

To support multiple users, the OS must provide:
→ User account management (creation, deletion, password management)
→ Memory protection: User A cannot read/write User B's memory space
→ File system permissions: Read/Write/Execute permissions per user
→ CPU scheduling: Fair allocation among competing users
→ Quota management: Limit disk space, CPU time, memory per user
→ Audit/logging: Track who did what for security and accountability
Examples of Multiuser OS:
→ UNIX / Linux: Classic multiuser OS — supports hundreds of simultaneous users
→ Windows Server: Multiuser through Remote Desktop Services
→ mainframe OS: IBM z/OS — thousands of simultaneous users
→ macOS: Supports multiple user accounts (UNIX-based)

Protection & Security in Multiuser Systems

The four main security concerns in multiuser systems (from Stallings):
→ Availability: System and data are available to authorized users when needed
→ Confidentiality: Only authorized users can read sensitive data
→ Data Integrity: Data cannot be altered by unauthorized users
→ Authenticity: Users and data sources can be verified as genuine
Exam tip: Multiuser OS is often confused with Multitasking OS. Remember: Multiuser = multiple people. Multitasking = multiple programs (can be single user).
UNIX permission model: rwxrwxrwx — owner, group, others. This is the foundation of multiuser file security.
Multiuser requires: time-sharing (for CPU) + virtual memory (for memory isolation) + file permissions (for data isolation).

§3 Multiprocess Systems

A Multiprocess System is one where multiple processes exist simultaneously in memory and compete for CPU time. The OS manages their execution through process scheduling, memory management, and IPC (Inter-Process Communication). Processes are independent units with their own memory space.

Process Recap (from context)

A Process = program in execution. It consists of:
→ Text section — the program code
→ Data section — global variables
→ Stack — function calls, local variables, return addresses
→ Heap — dynamically allocated memory (malloc, new)
→ Program Counter + CPU Registers — current execution state

One program can create multiple processes (e.g., Chrome browser creates separate processes for each tab).

Process States

NewBeing created
ReadyWaiting for CPU
RunningExecuting on CPU
WaitingWaiting for I/O
TerminatedFinished

Process Control Block (PCB)

The PCB (Process Control Block) is a data structure maintained by the OS for every process. It stores everything the OS needs to know about a process — it is the "identity card" of a process.
  • Process State (ready/running/waiting)
  • Program Counter
  • CPU Registers
  • CPU Scheduling info (priority)
  • Memory management info
  • Accounting info (CPU time used)
  • I/O status (open files, devices)
  • Process ID (PID)

Context Switching

Context Switch: When the CPU switches from one process to another, the OS must:
1. Save the state of the current process into its PCB
2. Load the state of the next process from its PCB

⚠️ Context switch = pure overhead — no useful computation happens during a switch.

Process Scheduling Queues

QueueContainsManaged by
Job QueueAll processes in the systemLong-term scheduler
Ready QueueProcesses waiting for CPU in memoryShort-term scheduler
Device QueueProcesses waiting for a specific I/O deviceDevice manager

Types of Schedulers

Short-Term (CPU) Scheduler
Selects which ready process runs next on CPU. Runs very frequently (milliseconds). Also called the dispatcher.
Long-Term (Job) Scheduler
Decides which processes enter the ready queue from job queue. Controls degree of multiprogramming. Runs infrequently.
Medium-Term Scheduler: Swaps processes in and out of memory to disk (swapping). Used to reduce the load on memory. Improves the mix of CPU-bound and I/O-bound processes.
Process ≠ Program: A program is passive (a file on disk). A process is active (loaded into memory and running).
Context switch overhead: Every time the CPU switches processes, time is wasted saving/loading state. This is why short time quanta in time-sharing = more overhead.
PCB = identity of a process — if you're asked "what info does OS maintain per process?" the answer is PCB contents.

§4 Multithreaded Systems

Multithreading is the ability of an OS to support multiple threads of execution within a single process. A thread is a lightweight unit of execution — also called a lightweight process (LWP). Threads within the same process share the process's memory space, open files, and resources.

Thread vs Process

Process Heavyweight — has its own memory space, PCB, resources. Creating a process is expensive. Processes are isolated from each other. Communication via IPC (pipes, sockets). Context switch is costly.
Thread Lightweight — shares memory with its process. Creating a thread is cheap. Threads share code, data, and heap. Communication via shared memory directly. Context switch is fast.

What Each Thread Has (Private) vs Shares

Private to each thread:
→ Program Counter (PC)
→ CPU Registers
→ Stack (local variables)
→ Thread state (ready/running/waiting)
Shared by all threads in process:
→ Code section (text)
→ Data section (global vars)
→ Heap (dynamic memory)
→ Open files and I/O resources

Benefits of Multithreading

  • Responsiveness: One thread can respond to user input while another does computation in the background
  • Resource Sharing: Threads share process memory — no need for IPC mechanisms
  • Economy: Much cheaper to create/destroy threads than processes
  • Scalability: In multiprocessor systems, threads can run truly in parallel on different CPUs
  • Utilization: While one thread waits for I/O, another thread can use the CPU

Types of Threads

User-Level Threads (ULT) Managed by a user-space thread library (e.g., POSIX pthreads). OS is unaware — sees only one process. Fast to create and switch. If one thread blocks, the whole process blocks.
Kernel-Level Threads (KLT) Managed directly by the OS kernel. OS can schedule individual threads on different CPUs. If one thread blocks, others can still run. Slower to create due to kernel involvement.

Real-World Multithreading Examples

→ Web Server: Each client request handled by a separate thread — thousands of concurrent connections
→ Web Browser: Separate threads for rendering, JavaScript execution, network I/O, and UI updates
→ Word Processor: One thread for display, one for auto-save, one for spell-check
→ Database Server: One thread per transaction — concurrent queries handled in parallel
→ Video Game: Separate threads for graphics rendering, physics, AI, audio, and input

Multithreading with SMP

In a Symmetric Multiprocessing (SMP) system with multithreading, the OS can schedule threads across multiple CPUs simultaneously. This is true parallelism — two threads of the same process can literally execute at the exact same moment on different processors.
Thread vs Process — most asked comparison: Key answer = threads share memory, processes don't. Creating a thread is faster and cheaper.
Concurrency vs Parallelism: Concurrency = multiple threads making progress (possibly by time-slicing on 1 CPU). Parallelism = threads running simultaneously on multiple CPUs.
Multithreading ≠ Multiprocessing: Multithreading is within ONE process. Multiprocessing involves multiple separate processes.

§5 Distributed Systems

A Distributed System is a collection of physically separate, possibly heterogeneous, computer systems that are networked together and appear to the user as a single coherent system. Each node has its own local memory and processors communicate via message passing over a network (not shared memory).
Tightly Coupled (Multiprocessor) Shared memory. Processors on the same motherboard. Communication via memory bus. Fast communication. Examples: SMP servers, multicore CPUs.
Loosely Coupled (Distributed) Separate memory per node. Communication via network (LAN/WAN). Slower communication. Examples: Cloud computing, clusters, P2P networks.

Advantages of Distributed Systems

  • Resource Sharing: Share printers, files, databases across the network
  • Computation Speedup (Load Sharing): Distribute computation across multiple nodes to reduce processing time
  • Reliability: If one site fails, remaining sites can continue operating — no single point of failure
  • Communication: Easy information exchange (email, file transfer, messaging) between nodes
  • Scalability: Add more nodes easily to increase capacity

Client-Server Model

In the Client-Server model, the network is divided into servers (systems that provide services) and clients (systems that request services). Clients and servers communicate via request-response messages over the network.
Compute-Server System
Provides an interface for clients to request computation services — e.g., running a database query, email processing, or printer services.
File-Server System
Provides an interface for clients to store and retrieve files — e.g., a web server (HTTP), FTP server, NFS (Network File System).

Peer-to-Peer (P2P) Model

In the P2P model, there is no distinction between clients and servers. All nodes are peers — each can act as client, server, or both simultaneously. To join a P2P network, a node either:
→ Registers its service with a central lookup service, OR
→ Broadcasts requests via a discovery protocol
P2P Examples:
→ Napster / Gnutella: File sharing (music/video)
→ BitTorrent: Distributed file download
→ Skype: Hybrid P2P — centralized login server + decentralized peer communication (VoIP)
→ Blockchain networks: Bitcoin, Ethereum — fully decentralized P2P

IPC in Distributed Systems

Distributed systems use message passing for IPC (not shared memory, since there is no shared memory):
→ Sockets: Low-level communication using IP address + port number. Format: IP:PORT e.g., 161.25.19.8:80
→ RPC (Remote Procedure Call): Call a function on a remote machine as if it were local
→ RMI (Remote Method Invocation): Java version of RPC — invoke methods on remote Java objects
→ Pipes: Unidirectional channel between two processes (ordinary pipes) or bidirectional (named pipes)
Tightly vs Loosely coupled — always asked. Tightly = shared memory (multiprocessor). Loosely = message passing over network (distributed).
Skype is hybrid P2P — it has a centralized login server but decentralized communication. This is a classic exam trick question.
Advantages of distributed systems — 4 key ones: resource sharing, computation speedup, reliability, communication.
TCP/IP is the most common network protocol for distributed systems communication.
