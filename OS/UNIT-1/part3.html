§1 OS Structure — Layered Approach

The Layered Structure organizes the OS into a hierarchy of layers (levels), each built on top of the one below it. The bottom layer (Layer 0) is the hardware. The top layer is the user interface. Each layer uses services only from the layer directly below it and provides services only to the layer directly above it.

The Layers (from bottom to top)

0
Hardware: Physical devices — CPU, memory, I/O controllers, buses. The foundation everything is built on.
1
Kernel Interface to Hardware: Device drivers, interrupt handlers, memory controllers, disk controllers. Directly manages physical hardware.
2
Kernel (Core OS): CPU scheduling, memory management (paging, virtual memory), file system, swapping, block I/O, signals, terminal handling.
3
System Call Interface: The boundary between user space and kernel space. System calls like open(), read(), write(), fork(), close().
4
System Programs: Shells (sh, bash), compilers (cc), interpreters, libraries. Use system calls to provide utilities.
5
Application Programs: Text editors (VI, pico), word processors (a.out), user applications. Interact with system programs and system calls.
THE THE key principle: Layer N can ONLY use services provided by Layer N−1. It CANNOT skip layers. This enforces modularity and simplifies debugging — if a layer works correctly, only the layer above it can cause problems at that level.

Advantages of Layered Structure

  • Modularity: Each layer can be designed, implemented, and tested independently
  • Simplicity of construction: Layers are built one at a time — each uses a simpler, tested foundation
  • Easy debugging: Bugs are isolated to specific layers
  • Abstraction: Higher layers don't need to know hardware details
  • Maintainability: Changes in one layer don't affect other layers (if interface is preserved)

Disadvantages of Layered Structure

  • Performance overhead: Each call must traverse multiple layers — slower than a monolithic approach
  • Difficult to define layers clearly: Some functions overlap layers (e.g., disk cache affects both I/O and memory management)
  • Less efficient: Cannot optimize across layer boundaries

Real OS Layered Structure (UNIX)

LayerContents
User ProgramsApplication software, utilities
Shells, Compilers, LibrariesSystem-level programs
System Call Interfacefork, exec, open, read, write, close...
KernelSignals, I/O, file system, CPU scheduling, virtual memory, device drivers
Kernel–Hardware InterfaceInterrupt handlers, device controllers
HardwareCPU, RAM, Disks, Terminals, Controllers

Classic example system: THE Operating System by Dijkstra — first to use layered design (6 layers).
Remember the system call boundary: System calls are how user programs request services from the kernel — this is the key boundary between user mode and kernel mode.

§2 System Components

An OS is composed of several major subsystems (components), each responsible for managing a specific category of resources. These components work together to provide a complete computing environment.

1. Process Management Component

Responsible for creating, scheduling, and terminating processes. Manages process synchronization and communication (IPC). Handles deadlock detection and recovery.
  • Process creation and deletion
  • Process suspension and resumption
  • Mechanisms for process synchronization (semaphores, mutexes)
  • Mechanisms for process communication (pipes, shared memory, sockets)
  • Deadlock handling

2. Memory Management Component

Tracks which parts of memory are in use and by whom. Decides which processes to load into memory when space is available. Allocates and deallocates memory space as needed.
  • Process isolation (each process has its own memory space)
  • Automatic allocation and deallocation
  • Support for modular programming (dynamic linking)
  • Virtual memory support (paging, segmentation)
  • Protection and sharing
  • Long-term storage mapping

3. File System Component

Manages the long-term storage of information. Organizes data into files and directories. Controls access permissions to files and directories.
  • File creation, deletion, reading, writing
  • Directory creation and deletion
  • File permission management (read/write/execute)
  • Mapping files onto secondary storage (disk blocks)
  • Backup and recovery

4. I/O System (Device Management) Component

Hides the peculiarities of hardware devices from the rest of the OS. Provides a uniform interface to all I/O devices.

→ Memory-management component — buffering, caching, spooling
→ General device-driver interface — uniform API for all devices
→ Drivers for specific hardware devices

5. Secondary Storage Management

  • Free-space management (tracking empty disk blocks)
  • Storage allocation (allocating disk blocks to files)
  • Disk scheduling (minimizing seek time)

6. Networking Component

  • Implementation of network protocols (TCP/IP)
  • Socket management
  • Remote file access, distributed computing support

7. Protection System Component

Controls access by programs and users to resources defined by the OS.
→ Distinguish authorized from unauthorized use
→ Specify the controls to be imposed
→ Provide a means of enforcement
→ Hardware protection: dual mode (user/kernel), memory protection, CPU protection (timer)
→ Software protection: access control lists, capability lists, user/group permissions
Exam tip: You may be asked to "list OS system components" — there are 7 major ones. The most important for exams are: Process Management, Memory Management, File System, I/O System, and Protection System.
Virtual Memory is part of Memory Management. It allows programs to use more memory than physically available by using disk as an extension of RAM.

§3 Operating System Services

OS services are the facilities provided by the OS — to users, to programs, and for the efficient operation of the system itself.

Services for User / Program Benefit

ServiceDescription
Program ExecutionLoad program into memory, run it, terminate it (normally or abnormally)
I/O OperationsRunning programs need I/O — OS provides controlled access to I/O devices
File System ManipulationRead, write, create, delete, search, list files and directories
CommunicationsExchange info between processes — on same computer or over network via shared memory or message passing
Error DetectionDetect and handle errors in CPU, memory, I/O devices, and user programs. Take appropriate action

Services for System Efficiency

ServiceDescription
Resource AllocationAllocate CPU, memory, files, I/O devices to multiple users/jobs running concurrently
AccountingTrack which users use how many and which kinds of resources — for billing or quota enforcement
Protection & SecurityEnsure all access to system resources is controlled; defend against insider and outsider threats

System Calls — The Interface to OS Services

System Call: A programmatic way for a process to request a service from the OS kernel. When a user program needs an OS service (e.g., read a file), it makes a system call which causes a trap into the kernel (switches from user mode to kernel mode). The kernel performs the requested operation and returns control to the user program.

System Call Execution Flow

Step-by-step system call flow:
1. User program calls a library function (e.g., printf())
2. Library function invokes a system call (e.g., write())
3. System call places a system call number in a CPU register
4. TRAP instruction switches CPU from User Mode → Kernel Mode
5. Kernel reads the system call number → dispatches to the appropriate handler
6. Handler executes the requested operation (e.g., writes to screen buffer)
7. Returns result to user program; CPU switches back to User Mode
Key OS Service Interfaces:
→ ISA (Instruction Set Architecture): Hardware–OS interface
→ ABI (Application Binary Interface): OS–compiled binary interface
→ API (Application Programming Interface): OS–source code interface (what programmers use)
System call vs Library call: Library calls (like printf) run in user space. System calls cause a mode switch to kernel space. printf internally calls write() system call.
Trap: A software-generated interrupt that switches to kernel mode — this is how system calls work.
OS services = 8 main ones — 5 for users (execution, I/O, file, communications, error detection) + 3 for efficiency (resource allocation, accounting, protection).

§4 Reentrant Kernels

A Reentrant Kernel is one that can be interrupted while executing in kernel mode on behalf of one process, and then re-entered to execute kernel code on behalf of another process. In other words, multiple processes can be simultaneously inside the kernel at the same time.

Why is Reentrancy Needed?

In a non-reentrant kernel (like early UNIX): when one process makes a system call and is executing in kernel mode, if an interrupt arrives (e.g., a hardware interrupt), the kernel cannot switch to another process until the current kernel operation finishes. This causes poor CPU utilization and poor response time.

A reentrant kernel allows:
→ Process A is executing a system call (e.g., read from disk)
→ Disk interrupt arrives — kernel suspends Process A's kernel execution
→ Kernel switches to Process B (which may also be in kernel mode for its own system call)
→ Both processes are "inside" the kernel simultaneously, each with their own kernel stack
→ This gives much better CPU utilization and faster response

How Reentrancy is Achieved

Two main techniques to make a kernel reentrant:

1. Kernel Functions Written as Reentrant Code:
→ Kernel functions use only local variables (on the stack) — no shared global state during execution
→ If a function uses shared data, it must use locking mechanisms (mutexes, spinlocks) to protect it

2. Per-Process Kernel Stack:
→ Each process has its own kernel-mode stack
→ When a process enters the kernel (via system call), it uses its own stack
→ If the kernel switches to another process, that process uses its own separate kernel stack
→ The two stacks don't interfere with each other

Non-Reentrant vs Reentrant Kernel

Non-Reentrant Kernel Only one process can be in kernel mode at a time. Other processes must wait even if they have valid kernel requests. Simple to implement (no synchronization needed). Poor performance in multiprocessor systems. Example: Early UNIX kernels.
Reentrant Kernel Multiple processes can be in kernel mode simultaneously. Each has its own kernel stack. Better performance and utilization. Requires careful synchronization of shared kernel data. Example: Linux kernel (fully preemptive/reentrant). Modern Windows kernel.

Kernel Preemption

A Preemptive Kernel (a stronger form of reentrant kernel) allows a process running in kernel mode to be forcibly preempted (interrupted and switched out) even while executing kernel code — not just at safe suspension points. This gives even better responsiveness for real-time tasks. Linux 2.6+ supports fully preemptive kernel.

Importance in SMP Systems

In Symmetric Multiprocessing (SMP) systems, reentrancy is essential:
→ Multiple CPUs can execute kernel code simultaneously (for different processes)
→ The kernel must be reentrant so that CPU₀ and CPU₁ can both be in kernel mode at the same time without corrupting shared kernel data structures
→ Fine-grained locking (spinlocks, per-CPU data) is used to protect shared kernel data
Key definition for exams: A reentrant kernel allows multiple processes to be in kernel mode simultaneously, each with its own kernel stack.
Reentrant ≠ Multithreaded kernel: Reentrancy is about safe concurrent access to kernel code. A multithreaded kernel actually runs multiple kernel threads.
Per-process kernel stack is the most common implementation technique — this is what enables reentrancy safely.

§5 Monolithic vs Microkernel

Monolithic Kernel

In a Monolithic Kernel, the entire OS runs as a single large program in kernel space. All OS services — process management, memory management, file system, device drivers, networking — run in a single privileged address space. All services can directly call each other without any message passing overhead.
Structure: User programs → System Call Interface → [ONE BIG KERNEL: process mgmt + memory mgmt + file system + device drivers + networking + IPC + ...] → Hardware

Advantages of Monolithic Kernel

  • Very fast performance — all subsystems communicate by direct function calls (no overhead)
  • Simple design — all OS code shares the same memory space
  • Easy to access shared kernel data structures directly
  • Mature, well-optimized (e.g., Linux has been tuned for decades)

Disadvantages of Monolithic Kernel

  • A bug in any driver or subsystem can crash the entire OS (kernel panic / Blue Screen of Death)
  • Large kernel size — complex and hard to maintain
  • Adding new features requires recompiling the kernel
  • Poor security isolation — a compromised driver has full kernel privileges
Examples: Linux, UNIX (traditional), MS-DOS, original Windows (up to 9x)
Linux uses Loadable Kernel Modules (LKMs) — a hybrid approach where device drivers can be added/removed dynamically without recompiling the full kernel.

Microkernel Architecture

In a Microkernel, only the most essential functions are kept in the kernel (kernel space). Everything else — device drivers, file systems, networking, protocol stacks — is moved to user space as separate server processes. Communication between components happens via message passing (IPC).
What stays in the Microkernel (minimal kernel):
→ Basic process management & scheduling
→ Basic memory management (address space management)
→ Inter-Process Communication (IPC / message passing)
→ Basic hardware abstraction

What moves to user space (as servers):
→ File system server
→ Device driver servers
→ Network protocol server
→ Security server
→ Virtual memory server

How services work in a Microkernel

1
User program sends a request message to the kernel
2
Kernel passes the message to the appropriate server process (e.g., file server)
3
Server processes the request in user space and sends a reply message
4
Kernel passes the reply back to the user program

Microkernel: Advantages & Disadvantages

Advantages:
→ Reliability: A crash in a driver/server doesn't crash the OS — just restart that server
→ Security: Services run in user space with limited privileges
→ Portability: Small kernel is easier to port to new hardware
→ Extensibility: Add new services without touching kernel code
→ Flexibility: Services can be added, removed, or modified at runtime
Disadvantages:
→ Performance overhead: Every service request needs multiple IPC message-passing operations — much slower than direct kernel calls
→ Complexity: IPC mechanisms are complex to implement correctly
→ Context switches: More mode switches between user/kernel space

Hybrid Kernels — Most modern OS use a hybrid approach:
→ Windows NT/XP/10/11: Hybrid — microkernel design with some services (like graphics) running in kernel space for performance
→ macOS / iOS: Based on Mach microkernel + BSD Unix services (XNU kernel)
→ Linux: Monolithic but with loadable kernel modules for flexibility
Most asked: "Compare monolithic and microkernel with advantages and disadvantages"
Microkernel examples: Mach, QNX, MINIX — memorize these.
Key microkernel fact: Only IPC + basic scheduling + basic memory management stay in kernel.
Performance tradeoff: Monolithic = fast but fragile. Microkernel = reliable but slow due to IPC.

§6 Quick Revision — Unit 1

🔵 OS Fundamentals

  • OS = intermediary between user and hardware; also Resource Allocator + Control Program
  • Kernel = program always running. Shell = user interface to kernel
  • Goals: Convenience, Efficiency, Ability to evolve
  • 4 System Components: Hardware → OS → Application Programs → Users
  • ISA (hardware-OS) · ABI (OS-binary) · API (OS-source code)

🔴 OS Classification Summary

  • Batch: No user interaction, similar jobs grouped, automatic job sequencing
  • Multiprogrammed Batch: Multiple jobs in memory, CPU never idle, needs scheduling + memory mgmt
  • SPOOL: Simultaneous Peripheral Operation On-Line — overlaps I/O and computation on disk
  • Time-Sharing: CPU multiplexed with time quanta, interactive, minimizes response time
  • Real-Time: Strict timing constraints. Hard (miss = fail) vs Soft (miss = degraded quality)
  • Multiprocessor: Multiple CPUs, shared memory (tightly coupled). AMP (master-slave) vs SMP (all peers)
  • Distributed: Multiple machines, separate memory, message passing (loosely coupled). C-S vs P2P
  • Multithreaded: Multiple threads in one process. Threads share memory, have own PC + stack + registers

🟡 Booting Sequence

  • Power On Reset → CS=0xF000, IP=0xFFF0 → Physical addr = (CS<<4)+IP = 0xFFFF0
  • Real Mode (1MB limit, no protection) → BIOS (POST, device init, identify boot device)
  • BIOS reads Sector 0 (MBR) → loads to 0x7C00
  • MBR = 512 bytes: 446 (code) + 64 (partition table) + 2 (signature 0x55AA)
  • Bootloader: Real Mode → Protected Mode, setup GDT, load OS from disk
  • OS: sets up virtual memory, interrupt vectors, initializes devices, starts user process

🟢 OS Structure

  • Layered: Each layer uses only the layer below it. THE OS by Dijkstra = first layered OS (6 layers). Advantage: modular, easy debug. Disadvantage: overhead, hard to define layers.
  • System Components (7): Process Mgmt, Memory Mgmt, File System, I/O System, Secondary Storage, Networking, Protection
  • OS Services (8): Program execution, I/O ops, File manipulation, Communications, Error detection | Resource allocation, Accounting, Protection & Security
  • System Call: User program → library → TRAP → Kernel mode → handler → return to user mode

🟣 Kernel Architecture

  • Reentrant Kernel: Multiple processes can be in kernel mode simultaneously, each with own kernel stack. Essential for SMP. Linux kernel is fully preemptive/reentrant.
  • Monolithic: All OS in one big kernel. Fast (direct calls) but fragile (one bug = kernel crash). Example: Linux, UNIX.
  • Microkernel: Only IPC + basic scheduling + basic memory in kernel. Rest in user-space servers. Reliable, portable, extensible — but slow (IPC overhead). Examples: Mach, QNX, MINIX.
  • Hybrid: Windows NT/10/11 (monolithic + micro), macOS/iOS (Mach + BSD = XNU), Linux (monolithic + LKMs)

⚡ Key Formulas & Facts

  • Physical Address (Real Mode): (CS << 4) + IP
  • MBR Size: 512 bytes = 446 + 64 + 2
  • First instruction address on power-on: 0xFFFF0
  • BIOS loads MBR to: 0x7C00
  • SPOOL = Simultaneous Peripheral Operation On-Line
  • Multiprocessor throughput with N CPUs: actual < N × single-CPU (overhead)
  • Tightly coupled = shared memory | Loosely coupled = message passing
  • Hard RT: miss deadline = FAIL | Soft RT: miss deadline = degraded quality only
