§1 Process Concept

A Process is a program in execution — an active entity. A program is a passive set of instructions stored on disk; a process is that program loaded into memory and being executed by the CPU, with its own resources and state.

Process vs Program

ProgramPassive entity — a file on disk (.exe, .out). Static — doesn't change. No resources allocated. One program can give rise to multiple processes. Example: chrome.exe file on disk.
ProcessActive entity — executing in memory. Dynamic — changes state. Has CPU, memory, I/O resources. Multiple instances can exist. Example: 5 Chrome tabs = 5 processes.

Process Memory Layout

A process in memory is divided into sections:

Stack (grows downward) — function call frames, local variables, return addresses
↕ Gap (free space)
Heap (grows upward) — dynamically allocated memory (malloc/new)
Data Section — global & static variables (initialized + uninitialized BSS)
Text Section (bottom) — compiled program code (read-only)

Process States

NewProcess being created
ReadyWaiting for CPU assignment
RunningInstructions being executed
WaitingWaiting for I/O/event
TerminatedFinished execution
State transitions:
New → Ready: process admitted to ready queue
Ready → Running: CPU scheduler dispatches process
Running → Waiting: process requests I/O or waits for event
Running → Ready: process preempted (time quantum expires)
Waiting → Ready: I/O complete / event occurs
Running → Terminated: process finishes or is killed

Process Control Block (PCB)

The PCB is a kernel data structure that stores all information about a process. It is the "identity card" of a process. When a context switch occurs, the PCB is saved and loaded.
PCB FieldDescription
Process ID (PID)Unique identifier for the process
Process StateCurrent state: new/ready/running/waiting/terminated
Program CounterAddress of next instruction to execute
CPU RegistersContents of all CPU registers (saved on context switch)
Scheduling InfoPriority, scheduling queue pointers
Memory InfoBase/limit registers, page tables
Accounting InfoCPU time used, wall-clock time, time limits
I/O StatusList of open files, I/O devices allocated

Process Creation (UNIX)

In UNIX/Linux, processes form a tree structure. The first process is init (PID=1). All other processes are descendants.

fork() — creates a new child process as an exact copy of the parent
exec() — replaces the process image with a new program
wait() — parent waits for child to finish
exit() — process terminates and releases resources

After fork(): returns 0 to child, returns child's PID to parent. 
pid_t pid = fork();
if (pid == 0) {
    // Child process
    exec("new_program");
} else {
    // Parent process — pid = child's PID
    wait(NULL); // wait for child
}

Process Termination

Zombie Process
Process has finished executing (called exit()) but its PCB still exists in the OS because the parent hasn't called wait() yet. It shows as "Z" in ps output. Resources are freed but entry remains.
Orphan Process
Parent process has terminated but child is still running. In UNIX, orphans are "adopted" by init (PID 1) which periodically calls wait() to clean them up.

Context Switching

When CPU switches from Process A to Process B:
1. Save state of Process A → PCB_A (registers, PC, stack pointer)
2. Update PCB_A status → "Ready" or "Waiting"
3. Load state of Process B from PCB_B
4. Update PCB_B status → "Running"
5. Jump to Process B's program counter

⚠️ Context switch is pure overhead — no useful work done during the switch itself
Context switch time depends on:
→ Hardware support (some CPUs have fast register save/restore instructions)
→ Number of registers to save
→ OS complexity
→ Memory speed (PCB is in memory)
Typically: 1–1000 microseconds per context switch
Process vs Thread: Process has its own memory space; threads share. Creating a process is expensive; creating a thread is cheap.
Zombie vs Orphan: Zombie = child done, parent hasn't waited. Orphan = parent done, child still running.
fork() return value: 0 → child process. Positive PID → parent process. -1 → fork failed.

§2 Principle of Concurrency

Concurrency refers to the ability of multiple processes or threads to execute overlapping in time. On a single-CPU system, concurrency is achieved by interleaving (time-sharing). On a multi-CPU system, true parallel execution is possible. Concurrency is the foundation of all modern OS design.

Types of Concurrency

Pseudo-Concurrency (Single CPU)Processes are interleaved on ONE CPU. Only one process runs at any instant, but they switch so fast it appears simultaneous. Example: Your PC with 1 core running 50 processes via time-sharing.
True Concurrency (Multi CPU)Multiple processes run on DIFFERENT CPUs at the exact same instant. Real parallelism. Example: A 4-core CPU running 4 different processes simultaneously at the hardware level.

Issues Arising from Concurrency

  • Race Condition: Multiple processes access shared data simultaneously, and the outcome depends on the order of execution — unpredictable results
  • Deadlock: Two or more processes are stuck waiting for resources held by each other — circular wait, no progress
  • Starvation: A process waits indefinitely because other processes keep getting priority — it never gets CPU time
  • Livelock: Processes keep changing state in response to each other but make no progress — like two people stepping aside in the same direction repeatedly
  • Priority Inversion: A high-priority process is blocked waiting for a resource held by a low-priority process
Race Condition Example:
Shared variable: counter = 5
Process A reads counter (gets 5), Process B reads counter (gets 5)
Process A increments → writes 6. Process B increments → writes 6.
Expected result: 7. Actual result: 6. ← This is a race condition!

Advantages of Concurrency

  • Better CPU utilization: While one process waits for I/O, another can use the CPU
  • Improved throughput: More tasks completed per unit time
  • Responsiveness: Interactive applications remain responsive (UI thread unblocked)
  • Resource sharing: Multiple processes can share memory and data
  • Modularity: Complex tasks can be broken into cooperating concurrent units

Conditions for Correct Concurrent Execution

For concurrent processes to work correctly, the OS must ensure:
1. Mutual Exclusion: Only one process accesses a shared resource at a time
2. Synchronization: Processes coordinate their execution order when needed
3. Deadlock avoidance: No circular waiting for resources
4. Starvation freedom: Every process gets CPU time eventually
5. Bounded waiting: There is a limit on how long a process waits to enter a critical section

Operating System's Role in Concurrency

  • Keep track of active processes (via PCBs)
  • Allocate and deallocate resources (CPU, memory, I/O)
  • Protect data and resources from improper concurrent access
  • Provide mechanisms for IPC (shared memory, message passing)
  • Provide synchronization primitives (semaphores, mutexes, monitors)
Race condition definition: Outcome depends on non-deterministic timing of processes — must know this cold.
Concurrency ≠ Parallelism: Concurrency = multiple processes in progress at once (possibly interleaved). Parallelism = truly simultaneous execution on multiple CPUs.
The 3 big problems of concurrency to remember: Race Condition, Deadlock, Starvation.

§3 Producer / Consumer Problem

The Producer-Consumer Problem (also called Bounded Buffer Problem) is a classic synchronization problem. A Producer process generates data and places it in a shared buffer. A Consumer process takes data from the buffer and uses it. The challenge: prevent the producer from adding to a full buffer, and prevent the consumer from taking from an empty buffer.

Problem Setup

Shared Resources:
→ A circular buffer of size N slots
→ in pointer — where producer will place next item
→ out pointer — where consumer will take next item
→ counter — number of filled slots (0 to N)

Producer constraint: Must wait if buffer is full (counter == N)
Consumer constraint: Must wait if buffer is empty (counter == 0)

Unbounded vs Bounded Buffer

Unbounded BufferNo limit on buffer size. Producer never waits. Consumer waits only if buffer is empty. Theoretical — not practical in real systems.
Bounded BufferFixed buffer size N. Producer waits if buffer full. Consumer waits if buffer empty. Practical model used in OS and real applications.

Race Condition in Producer-Consumer

The variable counter is shared. The operation counter++ is NOT atomic — it compiles to 3 machine instructions:
1. register = counter (load)
2. register = register + 1 (increment)
3. counter = register (store)

If the producer and consumer both execute these interleaved, counter gets the wrong value → race condition!

Producer-Consumer Code (Incorrect — Race Condition)

// Shared variables
int buffer[N];
int in = 0, out = 0, counter = 0;

// PRODUCER
while (true) {
    produce_item(item);
    while (counter == N); // busy wait if full
    buffer[in] = item;
    in = (in + 1) % N;
    counter++;  // ← NOT ATOMIC! Race condition here
}

// CONSUMER
while (true) {
    while (counter == 0); // busy wait if empty
    item = buffer[out];
    out = (out + 1) % N;
    counter--;  // ← NOT ATOMIC! Race condition here
    consume_item(item);
}

Race Condition Trace (Numerical Example)

🧪 Race Condition in counter++ and counter--
Initial: counter = 5
Producer executes: register₁ = counter → register₁ = 5
Producer executes: register₁ = register₁ + 1 → register₁ = 6
⚡ Context switch to Consumer!
Consumer executes: register₂ = counter → register₂ = 5 (stale read!)
Consumer executes: register₂ = register₂ - 1 → register₂ = 4
Consumer writes: counter = 4
Context switch back. Producer writes: counter = 6
Final counter = 6. Expected = 5 (one produce, one consume). ❌ WRONG!
✓ This demonstrates the race condition — result depends on scheduling order

The proper solution uses three semaphores:
→ mutex = 1 — ensures mutual exclusion on buffer access
→ empty = N — counts empty slots (producer waits when 0)
→ full = 0 — counts filled slots (consumer waits when 0)

Buffer size N=4. Initial: in=0, out=0, counter=0
Producer inserts A: buffer=[A,_,_,_], in=1, counter=1
Producer inserts B: buffer=[A,B,_,_], in=2, counter=2

counter++ is NOT atomic — this is the key insight of the Producer-Consumer problem.
Busy waiting (spin-lock with while loop) wastes CPU — semaphores are the proper fix.
Bounded buffer = N slots. Know both producer and consumer pseudocode for exams.

§4 Mutual Exclusion & Critical Section

Mutual Exclusion (Mutex): The requirement that when one process is executing in its critical section, no other process is allowed to execute in its critical section. It ensures that shared resources are accessed by only one process at a time.
Critical Section: A segment of code in which a process accesses shared resources (shared variables, files, I/O devices). Only one process may execute in its critical section at any given time.

Critical Section Problem Structure

Every process that uses a critical section must follow this structure:

do {
   entry section ← request permission to enter CS
      critical section
   exit section ← signal that CS is done
      remainder section
} while(true);

Requirements for a Valid Solution

Any valid solution to the critical section problem must satisfy ALL three conditions:

1. Mutual Exclusion: If process Pᵢ is in its CS, no other process Pⱼ can be in its CS simultaneously.

2. Progress: If no process is in the CS and some processes wish to enter, only those not in the remainder section can decide which process enters next, and this decision cannot be postponed indefinitely.

3. Bounded Waiting: There exists a bound (limit) on the number of times other processes can enter their CS after a process has requested to enter and before that request is granted. (Prevents starvation.)

Software Approaches to Mutual Exclusion

Attempt 1: Simple Flag (FAILS)

// Shared: int turn = 0 (or 1)
// Process P0:
while (turn != 0);   // busy wait
// critical section
turn = 1;           // give turn to P1
❌ Problem: Violates Progress — if P0 finishes CS and sets turn=1 but P1 doesn't want to enter, P0 can never re-enter even if P1 is in remainder section.

Attempt 2: Boolean Flags (FAILS)

// Shared: boolean flag[2] = {false, false}
// Process P0:
flag[0] = true;        // signal intention to enter
while (flag[1]);     // wait if P1 is in CS
// critical section
flag[0] = false;
❌ Problem: Deadlock — if P0 sets flag[0]=true and P1 sets flag[1]=true before either checks the other, BOTH wait forever.

Hardware Approaches

Interrupt Disabling: Disable interrupts while in critical section → no context switch possible. Simple but dangerous — if CS is long, other processes starve. Only works on single-CPU systems.

Hardware Atomic Instructions: Modern CPUs provide atomic (indivisible) instructions:
→ Test-and-Set (TSL/TAS): Atomically reads a lock variable and sets it to 1
→ Compare-and-Swap (CAS): Atomically compares and conditionally updates a variable
→ Fetch-and-Add: Atomically increments a counter
These guarantee no interleaving of the read-modify-write cycle.

🧪 Verify: Is Mutual Exclusion Maintained?
Two processes P0 and P1 share variable x=10. Both execute x = x + 1 concurrently.
Without mutual exclusion: Both read x=10, both compute 11, both write 11. Final x=11 (wrong — should be 12).
With mutual exclusion: P0 enters CS, reads x=10, writes x=11, exits CS. P1 enters CS, reads x=11, writes x=12, exits CS.
✓ With mutual exclusion: x=12 (correct). Without: x=11 (wrong). Mutex is essential for shared data correctness.
3 requirements for CS solution — always asked: Mutual Exclusion, Progress, Bounded Waiting.
Busy waiting / spin-lock = process loops checking a condition. Wastes CPU but no context switch overhead.
