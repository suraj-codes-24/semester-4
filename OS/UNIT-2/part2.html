§1 Dekker's Solution

Dekker's Algorithm (1965) is the first correct software solution to the mutual exclusion problem for two processes. It was developed by Dutch mathematician Th. J. Dekker. It combines the "turn" variable and "flag" arrays to satisfy all three requirements: mutual exclusion, progress, and bounded waiting.

Shared Variables

boolean flag[2] — flag[i]=true means Process i wants to enter CS
int turn — whose turn it is when there's a conflict (0 or 1)

Initially: flag[0]=false, flag[1]=false, turn=0

Dekker's Algorithm Code

// Process P0:
do {
    flag[0] = true;               // P0 wants to enter CS
    while (flag[1]) {             // if P1 also wants CS...
        if (turn == 1) {          // it's P1's turn
            flag[0] = false;      // P0 temporarily withdraws
            while (turn == 1);   // wait for turn to change
            flag[0] = true;       // P0 reasserts intention
        }
    }
    // ===== CRITICAL SECTION =====
    turn = 1;                      // give turn to P1
    flag[0] = false;               // P0 no longer wants CS
    // remainder section
} while (true);
P1 is symmetric: Same code with indices swapped (flag[1] instead of flag[0], check turn==0 instead of turn==1, set turn=0 at exit).

How Dekker's Algorithm Works — Step by Step

1
P0 sets flag[0]=true to indicate it wants to enter the critical section.
2
P0 checks if P1 also wants to enter (flag[1]==true). If NOT — P0 enters CS directly.
3
If P1 also wants CS: P0 checks the turn variable. If turn==1 (it's P1's turn), P0 is polite — it temporarily withdraws (flag[0]=false) and waits until it's its turn.
4
Once turn==0, P0 re-asserts its intention (flag[0]=true) and retries.
5
P0 executes critical section, then sets turn=1 (giving next priority to P1) and flag[0]=false.

Correctness Analysis

RequirementSatisfied?Why
Mutual Exclusion✅ YesOnly the process whose turn it is can enter when both want CS
Progress✅ YesIf only one process wants CS, it enters immediately (flag trick ensures this)
Bounded Waiting✅ YesA process waits at most one CS execution by the other process before getting its turn

Trace Example

🧪 Dekker's Execution Trace (Both want CS simultaneously)
Initial: flag=[F,F], turn=0
P0: flag[0]=true. P1: flag[1]=true (both set flags simultaneously)
P0 checks flag[1]==true → checks turn. turn=0 → NOT P1's turn → P0 doesn't withdraw
P1 checks flag[0]==true → checks turn. turn=0 → it IS P1's turn (turn==1? No! turn==0 means it's P0's turn!) → P1 withdraws: flag[1]=false, waits while turn==0
P0 sees flag[1]=false → enters Critical Section ✅
P0 exits CS: sets turn=1, flag[0]=false
P1: turn changed to 1, exits wait. Sets flag[1]=true again. Enters CS ✅
✓ Both processes get to execute CS — mutual exclusion maintained, no deadlock

Key Insight: Why Dekker Works

The genius of Dekker's algorithm is combining two ideas that each fail alone:

→ Flag array alone: Can cause deadlock (both set flags, both wait for each other)
→ Turn variable alone: Works for mutex but violates progress (strict alternation)

Dekker combines them: Flag shows intention. Turn breaks ties when both want CS simultaneously. When there's a conflict, the process that is NOT "in turn" politely backs off and waits. This eliminates both deadlock AND strict alternation.

Limitations of Dekker's Algorithm

  • Only works for exactly 2 processes — does not generalize easily to N processes
  • Uses busy waiting (spin-lock) — wastes CPU cycles while waiting
  • Complex to understand and verify compared to Peterson's solution
  • Assumes atomic reads and writes to shared variables (may not hold on all hardware)
  • Not suitable for modern multi-core systems without memory barriers
Dekker's = first correct software solution to mutual exclusion — always asked.
Two shared variables: flag[2] (intention) + turn (tie-breaker). Remember both.
Dekker vs Peterson: Dekker is older and more complex. Peterson is simpler and more elegant. Both work for 2 processes only.
In exam: Write the code for P0 only — mention P1 is symmetric with indices swapped.

§2 Peterson's Solution

Peterson's Algorithm (1981) is a simpler and more elegant software solution to the critical section problem for two processes. Proposed by Gary L. Peterson. It uses only two shared variables and satisfies all three requirements: mutual exclusion, progress, and bounded waiting.

Shared Variables

boolean flag[2] — flag[i]=true means Process i is ready to enter CS
int turn — indicates whose turn it is to enter CS (0 or 1)

Initially: flag[0]=false, flag[1]=false, turn=0 (or 1)

Key difference from Dekker: In Peterson's, a process sets turn to the OTHER process (being polite — "you go first"), whereas Dekker sets turn to self.

Peterson's Algorithm Code

// Process P0 (i=0, j=1):
do {
    flag[0] = true;           // P0 wants to enter CS
    turn = 1;                  // give turn to P1 (be polite)
    while (flag[1] && turn == 1);  // wait if P1 wants AND it's P1's turn
    // ===== CRITICAL SECTION =====
    flag[0] = false;           // P0 done with CS
    // remainder section
} while (true);

// Process P1 (i=1, j=0):
do {
    flag[1] = true;
    turn = 0;                  // give turn to P0
    while (flag[0] && turn == 0);
    // ===== CRITICAL SECTION =====
    flag[1] = false;
} while (true);

Peterson's Algorithm — Logic Explained

1
Set flag[i]=true: "I (Process i) want to enter the critical section."
2
Set turn=j: "I yield — you (Process j) go first if you want." (Being polite — the last one to set turn loses the tie.)
3
while (flag[j] && turn==j): Wait only if BOTH conditions are true — the other process wants to enter AND it's their turn. If either is false, we proceed.
4
Enter CS: Safe — mutual exclusion guaranteed.
5
Set flag[i]=false: "I'm done — others may enter."

Correctness Proof

Mutual Exclusion: Suppose both P0 and P1 are in CS simultaneously. P0 entered when flag[1]=false OR turn=0. P1 entered when flag[0]=false OR turn=1. But turn can only be 0 OR 1 — contradiction! Both cannot be in CS → ✅ ME holds.

Progress: If P1 is not interested (flag[1]=false), P0's while loop exits immediately → P0 enters CS without waiting → ✅ Progress holds.

Bounded Waiting: After P0 sets turn=1, P1 can enter CS at most once before P0 enters (turn then becomes 0) → ✅ Bounded wait.
🧪 Peterson's Trace: Both want CS at same time
P0: flag[0]=true, turn=1. P1: flag[1]=true, turn=0
After both execute: flag=[T,T]. What is turn? The LAST write wins. Say P1 wrote last → turn=0
P0 checks: flag[1]=true AND turn==1? turn=0 ≠ 1 → condition FALSE → P0 enters CS ✅
P1 checks: flag[0]=true AND turn==0? turn=0 → condition TRUE → P1 waits 🔄
P0 finishes CS, sets flag[0]=false
P1: flag[0]=false → condition FALSE → P1 enters CS ✅
✓ Perfect mutual exclusion. Whichever process sets turn last yields to the other.

Limitation of Both Solutions

Both Dekker's and Peterson's:
→ Work only for 2 processes (Peterson can be extended to N via Lamport's Bakery Algorithm)
→ Use busy waiting — wastes CPU cycles spinning in a loop
→ Assume sequential consistency of memory — may fail on modern CPUs with out-of-order execution without memory barriers/fences
→ Are purely software solutions — no OS or hardware support needed
Peterson's is simpler than Dekker's — Peterson uses just 2 lines in entry section vs Dekker's nested if-while.
The key Peterson insight: Set turn = j (the OTHER process). Last write to turn loses the race — if both processes set turn simultaneously, one will overwrite the other, and the one that wrote LAST will wait.
For N processes: Use Lamport's Bakery Algorithm — takes a "ticket number" like a bakery. Not in syllabus but good to mention.

§3 Semaphores

A Semaphore is an integer variable that, apart from initialization, is accessed only through two standard atomic operations: wait() (also called P or down) and signal() (also called V or up). Introduced by Edsger Dijkstra in 1965. Semaphores are used for both mutual exclusion AND synchronization.

Semaphore Operations

wait(S) — also P(S), down(S)

wait(S) {
  while(S <= 0); (busy wait)
  S--;
}

Decrements S. If S≤0, process waits until S becomes positive. "Request a resource."
signal(S) — also V(S), up(S)

signal(S) {
  S++;
}

Increments S. If processes are waiting, one is woken up. "Release a resource."
CRITICAL: Both wait() and signal() are atomic operations — they cannot be interrupted midway. The OS ensures atomicity by disabling interrupts (single CPU) or using hardware atomic instructions (multi-CPU).

Types of Semaphores

Binary Semaphore (Mutex)Value is only 0 or 1. Used for mutual exclusion. Initialized to 1. When locked (0), other processes wait. Example: mutex semaphore for a shared buffer.
Counting SemaphoreValue ranges over unrestricted integer domain. Used to control access to a resource with multiple instances. Initialized to the number of available resources. Example: N-slot buffer.

Semaphore for Mutual Exclusion

// Shared: semaphore mutex = 1

// Process Pi:
do {
    wait(mutex);        // mutex-- ; if mutex was 0 → block
    // ===== CRITICAL SECTION =====
    signal(mutex);      // mutex++ ; wake up waiting process
    // remainder section
} while (true);

Semaphore for Synchronization

// Ensure S2 in P2 executes AFTER S1 in P1
// Shared: semaphore sync = 0

// Process P1:
S1;
signal(sync);   // sync++ → tells P2 that S1 is done

// Process P2:
wait(sync);     // sync-- ; waits until P1 signals
S2;             // now safe to run after S1

Producer-Consumer with Semaphores (Complete Solution)

// Three semaphores:
semaphore mutex = 1;   // mutual exclusion on buffer
semaphore empty = N;   // counts empty slots (init = N)
semaphore full  = 0;   // counts filled slots (init = 0)

// PRODUCER:
do {
    produce_item(item);
    wait(empty);        // wait if no empty slot
    wait(mutex);        // get exclusive access to buffer
    buffer[in] = item; in = (in+1)%N;
    signal(mutex);      // release buffer
    signal(full);       // one more slot filled
} while (true);

// CONSUMER:
do {
    wait(full);         // wait if buffer empty
    wait(mutex);        // get exclusive access
    item = buffer[out]; out = (out+1)%N;
    signal(mutex);      // release buffer
    signal(empty);      // one more slot freed
    consume_item(item);
} while (true);

Blocking Semaphore (No Busy Waiting)

Instead of busy-waiting (spinning), a better semaphore implementation blocks the process by moving it to a waiting queue. The semaphore has an integer value AND a waiting queue (list). 
// Semaphore structure:
typedef struct {
    int value;
    struct process *list;  // waiting queue
} semaphore;

// wait(S):
S.value--;
if (S.value < 0) {
    add process to S.list;
    block();    // put process to sleep
}

// signal(S):
S.value++;
if (S.value <= 0) {
    remove process P from S.list;
    wakeup(P);  // wake it up
}
Semaphore value interpretation (blocking version):
→ Value > 0: Number of resources available
→ Value = 0: Resource taken, no one waiting
→ Value < 0: |value| = number of processes waiting in queue

Example: S = -3 means 3 processes are currently blocked waiting for this semaphore.
🧪 Semaphore Value Tracing (N=3 buffer)
Initial: mutex=1, empty=3, full=0
Producer produces item 1: wait(empty)→empty=2, wait(mutex)→mutex=0, insert, signal(mutex)→mutex=1, signal(full)→full=1
Producer produces item 2: empty=1, full=2
Producer produces item 3: empty=0, full=3
Producer tries item 4: wait(empty)→empty=-1 → BLOCKED (buffer full)
Consumer consumes: wait(full)→full=2, take item, signal(empty)→empty=0, wakes up producer
✓ Semaphore values precisely track buffer state. Negative = number waiting.

§4 Test and Set Operation

The Test-and-Set (TSL / TAS) instruction is a hardware atomic instruction that atomically reads a memory location, sets it to 1 (true), and returns the original value — all as a single, indivisible operation. No interrupt can occur between the read and the write.

TestAndSet() — Hardware Definition

// This executes ATOMICALLY (cannot be interrupted)
boolean TestAndSet(boolean *target) {
    boolean rv = *target;   // read current value
    *target = true;         // set to true (locked)
    return rv;              // return OLD value
}
How it works:
→ If lock was false (unlocked): TestAndSet returns false (old value), sets lock=true → Process enters CS.
→ If lock was true (already locked): TestAndSet returns true, sets lock=true (no change) → Process keeps spinning.
The magic: reading AND setting happen atomically — no race condition possible.

Mutual Exclusion using TestAndSet

// Shared: boolean lock = false

// Process Pi:
do {
    while (TestAndSet(&lock));  // spin until lock was false
    // ===== CRITICAL SECTION =====
    lock = false;               // release lock
    // remainder section
} while (true);
Correctness: Mutual Exclusion ✅ (only one process can see lock=false and set it to true atomically). Progress ✅ (if no process is in CS, lock=false and next process immediately enters). Bounded Waiting ❌ (standard TAS — starvation possible if one process keeps winning the race).

TestAndSet with Bounded Waiting

To fix starvation, use a waiting array:
→ boolean waiting[n] — waiting[i]=true if process i is waiting
→ boolean lock — the shared lock variable

On exit from CS, a process checks if any other process is waiting and directly passes the lock to it (in order), ensuring bounded wait of at most n-1 turns.

Compare-and-Swap (CAS)

Compare-and-Swap (CAS) is another hardware atomic instruction. It atomically compares a memory value with an expected value, and if they match, replaces it with a new value. Used in modern lock-free data structures and OS kernels. 
int CompareAndSwap(int *value, int expected, int new_value) {
    int temp = *value;
    if (*value == expected)
        *value = new_value;
    return temp;
}
