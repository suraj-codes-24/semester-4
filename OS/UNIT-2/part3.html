§1 Dining Philosophers Problem

The Dining Philosophers Problem (Dijkstra, 1965) is a classic synchronization problem. Five philosophers sit around a circular table. Between each pair of adjacent philosophers is one chopstick (fork). A philosopher needs both chopsticks (left and right) to eat. The problem: design a protocol so philosophers can eat without deadlock or starvation.

Problem Setup

→ 5 philosophers: P₀, P₁, P₂, P₃, P₄
→ 5 chopsticks: C₀, C₁, C₂, C₃, C₄
→ Chopstick Cᵢ is between Pᵢ and P(i+1)%5
→ Philosopher i needs: left chopstick Cᵢ AND right chopstick C(i+1)%5
→ Each philosopher alternates between thinking and eating
→ A chopstick can be held by only one philosopher at a time (shared resource)

Naive Solution (Uses Semaphores — DEADLOCKS!)

// Shared: semaphore chopstick[5] = {1,1,1,1,1}

// Philosopher i:
do {
    think();
    wait(chopstick[i]);          // pick up LEFT chopstick
    wait(chopstick[(i+1)%5]);   // pick up RIGHT chopstick
    eat();
    signal(chopstick[i]);        // put down LEFT
    signal(chopstick[(i+1)%5]); // put down RIGHT
} while (true);
❌ Deadlock scenario: All 5 philosophers simultaneously pick up their LEFT chopstick. Now each philosopher holds one chopstick and waits for the right one — which is held by the neighbor. Circular wait → Deadlock!

Solutions to Dining Philosophers

1
Allow at most 4 philosophers to sit: Use a counting semaphore initialized to 4. At most 4 philosophers can try to eat simultaneously → at least one will always get both chopsticks. Deadlock impossible.
2
Asymmetric solution: Odd philosophers pick LEFT then RIGHT. Even philosophers pick RIGHT then LEFT. This breaks the circular wait condition → no deadlock.
3
Pick up both at once (Monitor solution): A philosopher only picks up chopsticks when BOTH are available simultaneously. If not both available, wait. This requires a critical section around the check-and-pick operation.
4
Resource Hierarchy: Number chopsticks 1–5. Always pick the lower-numbered chopstick first. This imposes a global ordering → breaks circular wait.

Solution 1: Allow Max 4 Philosophers (Code)

// Shared: semaphore chopstick[5]={1,1,1,1,1}
// Shared: semaphore room = 4  (max 4 can try to eat)

// Philosopher i:
do {
    think();
    wait(room);                   // only 4 allowed at a time
    wait(chopstick[i]);
    wait(chopstick[(i+1)%5]);
    eat();
    signal(chopstick[i]);
    signal(chopstick[(i+1)%5]);
    signal(room);                  // free up a seat
} while (true);
🧪 Deadlock Check: Why max-4 prevents deadlock
5 philosophers, 5 chopsticks. If all 5 could try to eat: each picks 1 chopstick → deadlock.
With room=4: only 4 philosophers can try to eat simultaneously.
4 philosophers hold at most 4 chopsticks. The 5th chopstick is FREE.
At least one philosopher can pick up BOTH chopsticks → eats → releases chopsticks → others can proceed.
✓ At most 4 in room → at least 1 chopstick always free → no circular wait → no deadlock!

Philosopher States

In the monitor-based solution, each philosopher can be in one of three states:
→ THINKING: Not hungry, not eating
→ HUNGRY: Wants to eat, trying to get chopsticks
→ EATING: Has both chopsticks, currently eating

A philosopher moves to EATING only when neither neighbor is EATING.

Deadlock vs Starvation in Dining Philosophers

DeadlockAll philosophers hold one chopstick and wait for the other. No one can proceed. System is completely stuck. Caused by circular wait. Fixed by breaking one of the 4 Coffman conditions.
StarvationOne philosopher never gets to eat because neighbors always grab chopsticks first. Progress for others, but this philosopher waits indefinitely. Fixed by ensuring fairness (bounded waiting).
Dining Philosophers demonstrates: Deadlock, Starvation, and Resource allocation in concurrent systems.
Four Coffman conditions for deadlock: Mutual Exclusion, Hold and Wait, No Preemption, Circular Wait. Break ANY ONE → no deadlock.
Asymmetric solution is simplest to code: Odd picks left first, even picks right first. Break circular wait.
This problem models: Multiple processes competing for multiple shared resources (e.g., database transactions needing multiple locks).

§2 Sleeping Barber Problem

The Sleeping Barber Problem (Dijkstra, 1965) models a barber shop with one barber, one barber chair, and a waiting room with N chairs. It demonstrates process synchronization between a service provider (barber) and multiple customers (processes).

Problem Statement

Rules:
→ One barber, one barber chair, N waiting chairs
→ If NO customers: barber sleeps in barber chair
→ Customer arrives, barber sleeping → customer wakes barber → gets haircut
→ Customer arrives, barber busy, waiting chair available → customer sits and waits
→ Customer arrives, all waiting chairs full → customer LEAVES (lost customer)
→ After haircut done → barber checks waiting room. If customers waiting → takes next one. Else → goes back to sleep.

Semaphore Variables

semaphore customers = 0 — counts waiting customers (barber waits on this)
semaphore barbers = 0 — counts idle barbers (customers wait on this)
semaphore mutex = 1 — mutual exclusion for the waiting variable
int waiting = 0 — count of customers currently waiting (shared variable)

Sleeping Barber Solution Code

// BARBER process:
while (true) {
    wait(customers);    // sleep if no customers (block until customer arrives)
    wait(mutex);        // get exclusive access to 'waiting'
    waiting--;          // one less customer waiting
    signal(barbers);    // barber is ready
    signal(mutex);
    cut_hair();         // give haircut (outside mutex)
}

// CUSTOMER process:
void customer() {
    wait(mutex);
    if (waiting < CHAIRS) {   // check if waiting room has space
        waiting++;             // take a seat
        signal(customers);    // wake up barber (or signal one waiting)
        signal(mutex);
        wait(barbers);        // wait for barber to be ready
        get_haircut();
    } else {
        signal(mutex);        // no room → leave without haircut
    }
}

Trace: Sleeping Barber (N=3 chairs)

🧪 Sleeping Barber Execution Trace (3 waiting chairs)
Initially: customers=0, barbers=0, waiting=0. Barber executes wait(customers) → SLEEPS.
Customer 1 arrives: waiting=0 < 3 → waiting=1, signal(customers)→customers=1 → BARBER WAKES UP
Barber: customers-- (=0), waiting-- (=0), signal(barbers)→barbers=1, cut_hair()
Customer 1: wait(barbers)→barbers=0 → gets haircut ✅
Customers 2,3,4 arrive while barber is busy: waiting=1,2,3
Customer 5 arrives: waiting=3 = CHAIRS → leaves without haircut ❌
Barber finishes haircut → loops back → wait(customers) → customers=3 (waiting) → takes next
✓ Waiting room acts as bounded buffer. Full → customers turned away. Barber = consumer, customers = producers.

Key Insights

Race condition to avoid: Without mutex, two customers could simultaneously check waiting < CHAIRS, both see room available, both increment waiting → waiting exceeds CHAIRS. Mutex prevents this.

Without mutex: Barber might check waiting=0 and decide to sleep, while a customer simultaneously increments waiting to 1 and signals. If barber reaches sleep AFTER signal → barber sleeps forever with a waiting customer! Mutex prevents this race.
Sleeping Barber = Producer-Consumer variant with a bounded waiting room.
Three semaphores: customers, barbers, mutex — memorize all three and their initial values.
Real-world analogy: Thread pool in a web server — threads (barbers) wait for requests (customers). If request queue (waiting room) is full, new requests are rejected.
Key synchronization insight: Barber must check for waiting customers and go to sleep atomically — otherwise race condition.

§3 IPC Models and Schemes

Inter-Process Communication (IPC) refers to mechanisms provided by the OS that allow processes to communicate and synchronize with each other. Processes can be independent (no communication) or cooperating (share data/work). Cooperating processes need IPC.

Why IPC?

  • Information Sharing: Multiple processes access the same data (e.g., shared files)
  • Computation Speedup: Break a task into subtasks running in parallel
  • Modularity: Divide system functions into separate processes
  • Convenience: User runs multiple tasks simultaneously

Two Fundamental IPC Models

Shared Memory Processes share a region of memory. One process writes, another reads. Very FAST — just memory access, no kernel involvement after setup. Requires synchronization (semaphores/mutex) to prevent race conditions. Example: POSIX shm_open(), mmap().
Message Passing Processes exchange messages via the OS kernel. No shared memory — each process has own address space. Easier to implement correctly (no sync needed). Slower — kernel involved in every message. Example: Pipes, sockets, MPI.

IPC Schemes in Detail

1. Pipes

A pipe is a unidirectional communication channel between two processes. Data flows in one direction only. Created by the OS. The output of one process is the input of another.
Ordinary Pipes — exist only while processes are running. Parent creates pipe before fork(). Unidirectional. Must use two pipes for bidirectional. UNIX: pipe(fd)
Named Pipes (FIFOs) — persist in file system after processes die. Two unrelated processes can communicate. Bidirectional possible. UNIX: mkfifo()

2. Message Queues

A message queue is a linked list of messages stored in the kernel. Processes send messages to the queue; other processes receive from it. Messages have a type and can be received in order (FIFO) or by type (priority). More flexible than pipes — multiple senders and receivers possible.

3. Shared Memory

A region of physical memory is mapped into the address space of multiple processes. After the OS sets up the mapping, processes read/write directly — no further OS involvement. The fastest IPC mechanism. POSIX: shm_open() + mmap().

4. Sockets

An endpoint for communication. Identified by IP address : port number. Can communicate between processes on the same machine (Unix domain sockets) or different machines (network sockets). Used for client-server communication. Protocols: TCP (reliable), UDP (unreliable/fast).

More IPC Schemes

5. Signals

A signal is an asynchronous notification sent to a process by the OS or another process. The receiving process handles the signal by calling a signal handler function. Examples: SIGINT (Ctrl+C), SIGKILL (force kill), SIGALRM (timer), SIGSEGV (segmentation fault). Signals are the simplest form of IPC — no data transfer, just notification.

6. Remote Procedure Call (RPC)

RPC allows a process to call a function on a remote machine as if it were a local function call. The RPC mechanism handles all network communication transparently. Used in distributed systems. Java version: RMI (Remote Method Invocation).

Direct vs Indirect Communication

Direct CommunicationProcesses name each other explicitly: send(P, message) — send to process P. receive(Q, message) — receive from Q. Simple but tight coupling — if process name changes, must update all senders/receivers.
Indirect Communication (Mailboxes)Messages sent to/received from mailboxes (ports). send(A, message) — send to mailbox A. receive(A, message) — receive from mailbox A. Decoupled — any process with access to mailbox can communicate. Example: Message queues, sockets.

Synchronization in Message Passing

HybridNon-blocking sendBlocking receive (most common)
Shared memory = fast but complex; message passing = slower but simpler.
Pipes are unidirectional — for bidirectional, need TWO pipes.
Socket format: IP:PORT e.g., 192.168.1.1:8080. Below 1024 = reserved/privileged ports.

§4 Process Generation

Process Generation refers to the mechanisms by which new processes are created in an operating system. Processes can be created by the OS at startup, by a user command, by a running process (parent spawning a child), or by a batch job initiation.

Events that Trigger Process Creation

  • System initialization: OS creates daemon/background processes at boot (e.g., init, systemd, network services)
  • User request: User runs a program (double-click, terminal command)
  • Batch job submission: Job scheduler creates a process to execute a batch job
  • Process creates another process: Running process calls fork() (UNIX) or CreateProcess() (Windows) to spawn a child

UNIX Process Creation: fork() + exec()

fork(): System call that creates an exact copy of the calling process (parent). The child gets a copy of the parent's address space. Returns 0 to child, child PID to parent, -1 on error.

exec(): System call that replaces the current process's memory image with a new program loaded from disk. The process ID stays the same but the code changes.

Combined use: Parent calls fork() → child inherits parent's code → child calls exec() to load a new program. This is how the shell creates processes when you type a command.

Process Tree (UNIX)

All processes in UNIX form a tree (hierarchy):
→ init (PID=1) is the root — first process created by kernel
→ init spawns login processes, which spawn shells, which spawn user programs
→ Each process has exactly ONE parent (except init)
→ A process can have zero or more children
→ pstree command shows this tree in Linux

Resource Sharing Options after fork()

After fork(), the parent and child can:
→ Share all resources: Child gets references to all parent's open files
→ Share some resources: Child gets subset (commonly used)
→ Share no resources: Child is completely independent

Execution options:
→ Parent and child execute concurrently (most common)
→ Parent waits (calls wait()) until child terminates before continuing

Windows Process Creation

Windows uses CreateProcess() — a single call that both creates the new process AND loads a program into it (combines fork + exec).
→ Takes 10 parameters including executable path, command line, security attributes, etc.
→ Parent can use WaitForSingleObject() to wait for child
→ TerminateProcess() to kill a process

§5 Quick Revision — Unit 2

🔵 Process & Concurrency Fundamentals

  • Process = program in execution (active). Program = file on disk (passive).
  • Process sections: Text, Data, Heap (↑), Stack (↓). PCB = identity of process.
  • States: New → Ready → Running → Waiting → Terminated
  • Context switch = save PCB of current + load PCB of next = pure overhead
  • Concurrency issues: Race Condition, Deadlock, Starvation, Livelock, Priority Inversion
  • Race condition: outcome depends on timing of processes (non-deterministic)

🔴 Producer-Consumer & Critical Section

  • counter++ is NOT atomic → 3 instructions: load, increment, store → race condition
  • Critical Section requirements: Mutual Exclusion + Progress + Bounded Waiting
  • CS structure: entry section → critical section → exit section → remainder section
  • Simple turn variable: ME ✅, Progress ❌ (strict alternation)
  • Boolean flags alone: ME ✅, Progress ❌ (deadlock possible)

🟡 Synchronization Solutions

  • Dekker's (1965): First correct software solution. Variables: flag[2] + turn. Process withdraws if not its turn.
  • Peterson's (1981): Simpler. Set turn = OTHER process (yield). Wait while: flag[j] AND turn==j.
  • Semaphore: P=wait=down (decrement). V=signal=up (increment). Binary (0/1) vs Counting.
  • Producer-Consumer semaphores: mutex=1, empty=N, full=0
  • Blocking semaphore: S<0 → |S| processes waiting in queue
  • TAS: Atomic read-and-set. Returns OLD value. ME ✅, Bounded Wait ❌ (basic version)

🟢 Classical Problems

  • Dining Philosophers (5 Phil, 5 Chopsticks): Naive solution → deadlock (all pick left chopstick)
  • Solutions: (1) Max 4 philosophers (room semaphore=4). (2) Asymmetric: odd=left first, even=right first. (3) Pick both atomically. (4) Resource hierarchy.
  • Sleeping Barber: Semaphores: customers=0, barbers=0, mutex=1, waiting=0
  • Barber: wait(customers). Customer: check waiting<N → waiting++, signal(customers), wait(barbers)
  • Barber=CPU/Server. Customers=Processes/Jobs. Waiting room=Ready Queue. Full→customer leaves=job rejected
  • Coffman deadlock conditions: Mutual Exclusion + Hold & Wait + No Preemption + Circular Wait

🟣 IPC & Process Generation

  • IPC models: Shared Memory (fast, needs sync) vs Message Passing (slow, safer)
  • IPC schemes: Pipes (unidirectional), Named Pipes (FIFO), Message Queues, Shared Memory, Signals, Sockets (IP:PORT), RPC/RMI
  • Direct comm: name processes explicitly. Indirect: use mailboxes/ports.
  • fork() → child copy. exec() → replace with new program. wait() → parent waits for child.
  • Zombie = finished child, parent not called wait(). Orphan = parent died, child adopted by init.
  • Copy-on-Write (COW): pages shared after fork until one process writes → then copied.

⚡ Key Facts & Top Exam Questions

  • Dekker = first correct solution (1965). Peterson = simpler solution (1981). Both for 2 processes only.
  • Semaphore wait = P = down. Signal = V = up. Dijkstra's notation.
  • In P-C semaphore: ALWAYS wait(empty/full) BEFORE wait(mutex). Reverse → deadlock!
  • Dining Philosophers with N philosophers needs N chopsticks. Allow max N-1 to eat → no deadlock.
  • Sleeping Barber with N chairs: customer leaves if waiting ≥ N (not waiting > N). Careful!
