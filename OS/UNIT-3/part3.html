§1 System Model & Deadlock Characterization

Deadlock: A situation in which two or more processes are blocked forever, each waiting for a resource held by another process in the set. No process can proceed — the system is stuck. Only OS intervention (process termination or resource preemption) can break the deadlock.

System Model

A system consists of resources: R₁, R₂, …, Rₘ (CPU cycles, memory, I/O devices, files).
Each resource type Rᵢ has Wᵢ instances (e.g., 3 printers, 5 semaphores).
Each process uses resources by: Request → Use → Release.

Classic Deadlock Example:
→ P1 requests Resource A → gets it
→ P2 requests Resource B → gets it
→ P1 requests Resource B → blocked (P2 holds it)
→ P2 requests Resource A → blocked (P1 holds it)
→ Neither can proceed → DEADLOCK!

Four Necessary Conditions for Deadlock (Coffman Conditions)

Deadlock can arise if and only if ALL FOUR conditions hold simultaneously:

1. Mutual Exclusion: At least one resource is held in a non-shareable mode — only one process can use it at a time.

2. Hold and Wait: A process is holding at least one resource AND is waiting to acquire additional resources held by other processes.

3. No Preemption: Resources cannot be forcibly taken away — a resource can only be released voluntarily by the holding process.

4. Circular Wait: A circular chain of processes exists where each process holds a resource needed by the next: P0→P1→P2→…→Pn→P0.

Resource-Allocation Graph (RAG)

A Resource-Allocation Graph (RAG) is a directed graph used to represent the state of resource allocation:
→ Vertices (V): Two types — Processes (P₁, P₂, …, Pₙ) shown as circles, and Resources (R₁, R₂, …, Rₘ) shown as rectangles. Dots inside rectangles = instances of that resource.
→ Request edge: Pi → Rj (Pi is waiting for an instance of Rj)
→ Assignment edge: Rj → Pi (an instance of Rj is allocated to Pi)

RAG Rules for Deadlock Detection

If graph contains NO cycle: ⇒ No deadlock. System is safe.

If graph contains a cycle:
→ If each resource type has only one instance ⇒ Deadlock DEFINITELY exists
→ If resource types have multiple instances ⇒ Deadlock may or may not exist (need further analysis)

RAG Example Analysis

🧪 Is there a deadlock? (Cycle analysis)
P1 holds R2, waiting for R1. P2 holds R1 and R2, waiting for R3. P3 holds R3.
Check for cycle: P1→R1→P2→R3→P3→? Does P3 need anything P1 holds? If P3 needs R2: P3→R2→P1 → cycle!
Cycle: P1→R1→P2→R3→P3→R2→P1
If R1, R2, R3 each have 1 instance → deadlock confirmed.
If R2 has 2 instances and one is free → P3 can get R2, finish, release R3 → P2 can finish → P1 can finish. No deadlock.
✓ Cycle with single instances = deadlock. Cycle with multiple instances = need safety check.

Three Methods for Handling Deadlocks

1. Deadlock Prevention: Ensure at least one of the four Coffman conditions NEVER holds. Most restrictive — constrains how resources can be requested.

2. Deadlock Avoidance: Require processes to declare max resource needs in advance. Dynamically check if granting a request leads to an unsafe state. Grant only if safe.

3. Deadlock Detection & Recovery: Allow deadlock to happen. Periodically run a detection algorithm. If detected, recover by terminating processes or preempting resources.

4. Ignore it (Ostrich Algorithm): Pretend deadlocks never occur. Used by most OS including UNIX — deadlocks are rare and recovery is the user's problem.

§2 Deadlock Prevention

Deadlock Prevention — Ensure that at least ONE of the four necessary conditions for deadlock can never hold. This is done by restricting how resources are requested. If any one condition is broken, deadlock is impossible.

Prevention Strategy for Each Coffman Condition

1. Break Mutual Exclusion

Approach: Allow all resources to be shared.
Problem: NOT PRACTICAL for most resources. Printers, tape drives, etc. MUST be used exclusively — sharing them causes errors. Only sharable resources (like read-only files) can bypass this condition.
Conclusion: Cannot be used for prevention in general. Mutual exclusion must hold for non-sharable resources.

2. Break Hold and Wait

Approach 1: Request ALL resources before starting execution. Process gets everything it needs upfront, or waits until all are available at once.
Approach 2: A process may request resources only when it holds NONE. Before requesting new resources, release all currently held resources first.
Problems: Low resource utilization (holds resources for entire duration even if not needed). Starvation possible (process that needs many popular resources may wait forever).

3. Break No Preemption

Approach: If a process holding resources requests a new resource that cannot be immediately granted → release ALL currently held resources. Add them to the list of resources it needs. Process restarts only when it can get ALL needed resources (old + new) together.
Problem: Not practical for most resources. Cannot forcibly take back a printer mid-job. Works for resources whose state can be saved/restored (CPU registers, memory).

4. Break Circular Wait ✅ (Most Practical)

Approach: Impose a total ordering on all resource types. Assign each resource type a unique integer (e.g., R1=1, R2=2, R3=3). Each process must request resources in strictly increasing order of enumeration.

If a process holds Rᵢ and needs Rⱼ, it must be that j > i. This breaks circular wait because you can never have P1 wait for something held by P2 which waits for something held by P1 — the ordering prevents the cycle.

Example (Bank Transfer): Always lock lower account number first. Transaction 1 (A→B) locks A then B. Transaction 2 (B→A) MUST also lock A first then B → no deadlock possible.

Lock Ordering Example (from PDF)

Without Lock Ordering (deadlock possible):
Transaction 1: locks account A, then locks account B.
Transaction 2: locks account B, then locks account A.
If T1 locks A and T2 locks B simultaneously → deadlock!

With Lock Ordering (deadlock impossible):
Rule: Always lock lower-numbered account first.
Transaction 1 (A→B): locks A(lower) then B. Transaction 2 (B→A): MUST lock A first, then B. Now both compete for A first → one wins, no deadlock.

Hold and Wait prevention = "all or nothing" resource allocation — get all resources at start or wait with nothing.
No preemption = save state, release, restart — only works when state can be saved (like CPU context).

§3 Deadlock Avoidance & Banker's Algorithm

Deadlock Avoidance: Requires that processes declare their maximum resource needs in advance. The OS uses this information to dynamically examine the resource allocation state and ensure the system never enters an unsafe state. More flexible than prevention — allows more parallelism.
Safe State: A system is in a safe state if there exists a sequence ⟨P₁, P₂, …, Pₙ⟩ of ALL processes such that for each Pᵢ, the resources that Pᵢ still needs can be satisfied by currently available resources plus resources held by all Pⱼ with j < i.

→ Safe state ⇒ No deadlock
→ Unsafe state ⇒ Possibility of deadlock (not guaranteed, but risky)
→ Goal of deadlock avoidance: ensure system always stays in SAFE state.

Banker's Algorithm — Data Structures

Let n = number of processes, m = number of resource types.

→ Available[m]: Available[j] = k means k instances of resource type Rⱼ are available.
→ Max[n×m]: Max[i][j] = k means process Pᵢ may request at most k instances of Rⱼ.
→ Allocation[n×m]: Allocation[i][j] = k means Pᵢ currently holds k instances of Rⱼ.
→ Need[n×m]: Need[i][j] = k means Pᵢ still needs k more instances of Rⱼ.

Need[i][j] = Max[i][j] − Allocation[i][j]

Safety Algorithm

// Find if system is in safe state
1. Initialize:
   Work = Available          // working copy of available
   Finish[i] = false for all i  // no process finished yet

2. Find index i such that:
   (a) Finish[i] == false
   (b) Need[i] ≤ Work        // can this process proceed?
   If no such i exists → go to step 4

3. Work = Work + Allocation[i]  // process finishes, releases resources
   Finish[i] = true
   Go to step 2

4. If Finish[i] == true for ALL i → SAFE STATE
   Else → UNSAFE STATE

Resource-Request Algorithm (Process Makes a Request)

// Process Pi makes request: Request[i]
Step 1: if Request[i] ≤ Need[i]
        → proceed to step 2
        else → ERROR (exceeds maximum claim)

Step 2: if Request[i] ≤ Available
        → proceed to step 3
        else → Pi must WAIT (not enough resources)

Step 3: PRETEND to allocate (temporary state):
        Available   = Available - Request[i]
        Allocation[i] = Allocation[i] + Request[i]
        Need[i]     = Need[i] - Request[i]

        Run Safety Algorithm on this new state:
        if SAFE  → grant request (make permanent)
        if UNSAFE → Pi waits, RESTORE old state

✓ Request GRANTED — new state is safe!

§4 Deadlock Detection

Deadlock Detection — Allow the system to enter a deadlock state, but periodically run a detection algorithm to check if deadlock has occurred. If detected, take recovery action. More permissive than prevention/avoidance — maximum resource utilization.

Single Instance: Wait-For Graph

For single-instance resources: maintain a Wait-For Graph.
→ Nodes = processes only (no resource nodes)
→ Edge Pᵢ → Pⱼ means Pᵢ is waiting for a resource held by Pⱼ
→ Convert RAG to Wait-For Graph by eliminating resource nodes

Detection: Periodically run a cycle-detection algorithm on the Wait-For Graph.
→ Cycle exists → Deadlock
→ Cycle detection requires O(n²) operations where n = number of processes.

Multiple Instances: Detection Algorithm

Similar to Banker's safety algorithm but uses Request instead of Need:
→ Available[m]: Currently available resources
→ Allocation[n×m]: Current allocation
→ Request[n×m]: CURRENT (not maximum) request of each process 
1. Work = Available
   For i = 1..n: if Allocation[i] ≠ 0 → Finish[i]=false; else Finish[i]=true

2. Find i such that:
   (a) Finish[i] == false
   (b) Request[i] ≤ Work
   If no such i → go to step 4

3. Work = Work + Allocation[i]
   Finish[i] = true → go to step 2

4. If Finish[i] == false for some i → Pi is DEADLOCKED
   Else → No deadlock

Detection vs Avoidance — Key Differences

Deadlock Avoidance Uses Max (maximum need). Requires advance declaration of maximum needs. Prevents entering unsafe state proactively. Uses Need matrix (Max - Allocation). More restrictive — may deny valid requests.
Deadlock Detection Uses Request (current actual request). No advance declaration needed. Reacts after deadlock occurs. More permissive — allows more concurrency. Needs recovery after detection.

When to Run the Detection Algorithm?

Trade-off in detection frequency:
→ Run TOO OFTEN (every request): High overhead but immediate detection and recovery
→ Run TOO RARELY: Low overhead but deadlock may persist long before detected

Strategies:
1. Run every time a resource request cannot be granted
2. Run at fixed time intervals (e.g., every hour)
3. Run when CPU utilization drops below threshold (sign of deadlock)

Problem with infrequent detection: Many processes may be involved in multiple cycles — hard to identify which process "caused" the deadlock.
Detection algorithm ≈ Safety algorithm but uses Request (actual) instead of Need (max).
Single instance → Wait-For Graph. Multiple instances → Detection algorithm (matrix-based).
Finish[i] initially true if Allocation[i]=0 (process holds nothing, can't be in deadlock).
Cycle detection complexity: O(n²) where n = number of processes.

§5 Recovery from Deadlock

Once a deadlock is detected, the system must recover. There are two main approaches: terminate one or more deadlocked processes, or preempt resources from one or more deadlocked processes.

Method 1: Process Termination

Option A — Abort ALL deadlocked processes:
Terminates all processes in the deadlock cycle. Definitely breaks the deadlock. High cost — work done by processes is lost. Simple to implement.

Option B — Abort ONE process at a time (until deadlock broken):
Terminate processes one by one. After each termination, run detection algorithm again to check if deadlock is resolved. More expensive in time (many detection runs) but less wasteful of work.

Which process to abort? Selection criteria (minimize cost):
  • Priority: Abort lowest-priority process first
  • Progress: How much has the process computed? How much longer to completion?
  • Resources used: How many resources does it hold? (fewer = cheaper to abort)
  • Resources needed: How many resources does it still need to complete?
  • Processes to terminate: How many other processes will need to be terminated?
  • Interactive vs batch: Batch processes are easier to abort than interactive ones

Method 2: Resource Preemption

Forcibly take resources from some processes and give them to others to break the deadlock.

Issues to address:
→ Selecting a victim: Which process to preempt? Minimize cost (similar criteria as process termination).
→ Rollback: After preemption, the victim process must be rolled back to a safe state. Usually simplest to roll back to the beginning (total rollback). Checkpoint-based rollback is better.
→ Starvation: The same process might always be selected as the victim. Solution: include the number of rollbacks in the cost factor — after being preempted too many times, give the process higher priority.

§6 Quick Revision — Unit 3

🔵 Scheduling Foundations

  • Process = active (executing). Program = passive (on disk). CPU-I/O burst cycle alternates.
  • I/O-bound = many short bursts. CPU-bound = few long bursts. Long-term scheduler mixes both.
  • Short-term (CPU) scheduler: very frequent, must be fast. Long-term (job): infrequent, slow OK. Medium-term (swapper): swaps in/out.
  • PCB = process state + PID + PC + registers + scheduling info + memory info + accounting + I/O status
  • Context switch = save PCB of old + load PCB of new = pure overhead.
  • 4 scheduling situations: Running→Waiting(1), Running→Ready(2), Waiting→Ready(3), Termination(4). NP=1,4 only. P=all 4.
  • Dispatcher: gives CPU to selected process. Dispatch latency = time to switch.

🔴 Key Formulas

  • TAT = CT - AT    WT = TAT - BT    RT = first_cpu_time - AT
  • Min resources (deadlock-free) = n×(max-1) + 1
  • Need[i][j] = Max[i][j] - Allocation[i][j]
  • SJF prediction: τₙ₊₁ = α·tₙ + (1-α)·τₙ
  • RR: no process waits more than (N-1)×q time units

🟡 Scheduling Algorithms

  • FCFS: Non-preemptive. First arrival runs first. Convoy effect (short behind long). No starvation.
  • SJF (NP): Shortest BT first when CPU is free. Optimal for given set. Starvation possible for long jobs.
  • SRTF (P-SJF): Preempt if new arrival has shorter BT than remaining BT. Globally optimal (min avg WT).
  • RR: Preemptive. Time quantum q. Fair, no starvation. q large→FCFS, q small→high overhead. Best response time.
  • Priority: Higher priority runs first. Starvation of low-priority → fix with Aging.
  • Multilevel Queue: Multiple queues, processes permanently assigned. Each queue has own algorithm.
  • MLFQ: Processes can move between queues. Most flexible. Used in real OS.

🟢 Deadlock

  • 4 Coffman conditions (ALL must hold): Mutual Exclusion + Hold & Wait + No Preemption + Circular Wait
  • RAG: cycle + single instance → deadlock. Cycle + multiple instances → maybe deadlock.
  • Prevention: Break one condition. Most practical: Circular Wait (resource ordering).
  • Avoidance: Banker's Algorithm. Need = Max - Alloc. Safety algorithm: find Work sequence. Request: pretend-allocate → run safety → grant if safe.
  • Detection: Like safety but uses actual Request (not Need). Finish[i] initially true if Alloc[i]=0. If Finish[i]=false for any i after algorithm → Pi is deadlocked.
  • Recovery: Terminate processes (all or one-by-one) OR preempt resources (rollback + restart).
  • Most OS use Ostrich algorithm (ignore deadlocks) — UNIX, Linux, Windows.
