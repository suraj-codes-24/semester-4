§1 FCFS Scheduling

First-Come, First-Served (FCFS) — The process that arrives first in the ready queue is assigned the CPU first. In case of a tie (same arrival time), process with smaller PID is executed first. Always non-preemptive — once CPU is given, it keeps it until it finishes or waits for I/O.
Advantages:
→ Simple and easy to implement
→ Implemented using a simple FIFO queue
→ No starvation (every process eventually runs)
Disadvantages:
→ Ignores process priority and burst time
→ Convoy effect — short processes wait behind long ones
→ Poor average waiting time
Convoy Effect: Short processes are stuck behind a long CPU-bound process. All I/O-bound processes (which have short CPU bursts) must wait for the one long CPU-bound process to finish → I/O devices become idle while waiting → inefficient.

FCFS Step-by-Step Procedure

Step 1: Sort processes by Arrival Time (AT). Break ties by PID.
Step 2: Start from t=0. If no process has arrived yet → CPU is Idle until next process arrives.
Step 3: Draw Gantt chart. Each process runs for its full BT without interruption.
Step 4: CT of process = time when it finishes. CT of Pₙ = CT of Pₙ₋₁ + BT of Pₙ (if no idle gap).
Step 5: TAT = CT − AT. WT = TAT − BT.
Step 6: Compute averages.

FCFS with Idle Gaps — How to Handle

If the CPU finishes a process and the next process hasn't arrived yet → CPU is Idle.
Next process starts at its arrival time, not at the end of the previous process.

Example (PDF Problem 3): P1(AT=0,BT=2), P2(AT=3,BT=1), P3(AT=5,BT=6)
→ P1 finishes at t=2. P2 hasn't arrived (arrives at 3) → CPU idle from 2→3
→ P2 finishes at t=4. P3 hasn't arrived (arrives at 5) → CPU idle from 4→5
→ P3 runs 5→11
→ TAT: P1=2, P2=1, P3=6. WT: P1=0, P2=0, P3=0. Avg WT = 0!

§2 SJF & SRTF Scheduling

Shortest-Job-First (SJF): Among all available processes, CPU is assigned to the one with the shortest burst time. Ties are broken by FCFS. Works in both preemptive and non-preemptive modes. Optimal algorithm — gives minimum average waiting time for a given set of processes.
Non-Preemptive SJF Once CPU given to process, it runs to completion (or until I/O wait). No interruption even if a shorter process arrives. When CPU is free, pick process with shortest BT from all arrived processes.
Preemptive SJF = SRTF (Shortest Remaining Time First) If a new process arrives with BT less than remaining time of current process → preempt current process, run new one. Always checks at each arrival.
Advantages:
→ Optimal: minimum avg WT
→ SRTF guarantees shortest average wait
→ Standard for other algorithms
Disadvantages:
→ Cannot know BT in advance (must predict)
→ Starvation for long processes
→ Cannot set priorities

SRTF Key Rule

SRTF Decision Rule: At every time unit when a new process arrives, compare its BT with the remaining burst time of the current process.
→ New BT < Current Remaining → Preempt, run new process
→ New BT ≥ Current Remaining → Continue current process

SRTF is guaranteed to be OPTIMAL — no other algorithm achieves a lower average waiting time.

Predicting Next CPU Burst (Exponential Averaging)

Since burst time cannot be known in advance, SJF uses exponential averaging to predict:
τₙ₊₁ = α·tₙ + (1-α)·τₙ

Where: τₙ₊₁ = predicted next burst, tₙ = actual last burst, τₙ = previous prediction, α = weight (0 to 1)

→ α = 0: Recent history has no influence (use only old prediction)
→ α = 1: Only actual last burst matters (ignore history)
→ α = 0.5 (most common): Equal weight to recent and historical

Each successive term has less weight than its predecessor → exponential decay of history.

§3 Round Robin Scheduling

Round Robin (RR) — Each process gets a small unit of CPU time called a time quantum (q). After the quantum expires, the process is preempted and added to the END of the ready queue. Designed specifically for time-sharing systems. Always preemptive.
Key Properties:
→ With N processes and time quantum q: each process gets 1/N of CPU time in chunks of at most q
→ No process waits more than (N-1) × q time units
→ Timer interrupt fires every q units to schedule next process
→ q large → degenerates to FCFS
→ q small → many context switches → high overhead
→ Typical q: 10ms–100ms. Context switch < 10 microseconds.
Advantages:
→ Fairness — each process gets equal CPU time
→ No starvation
→ Good response time for interactive systems
→ Simple to implement
Disadvantages:
→ Higher avg TAT than SJF
→ More context switches → overhead
→ Performance depends heavily on quantum size
→ Not optimal for throughput

RR Step-by-Step Procedure

Step 1: Sort by arrival time. Maintain a circular ready queue.
Step 2: Run first process for min(q, remaining BT) time units.
Step 3: After quantum: if process not done → add to END of ready queue. If done → remove.
Step 4: If new processes arrived during this quantum → add them to ready queue (in arrival order) BEFORE adding the preempted process back.
Step 5: Repeat until all processes done.
Key rule: If remaining BT ≤ q → process runs to completion in less than one quantum (no preemption needed).

Effect of Quantum Size

Too small quantum: Excessive context switches. If q=1ms and context switch=0.5ms → 33% of CPU time wasted on switching!
Too large quantum: Degenerates to FCFS — poor response time for interactive processes.
Rule of thumb: q should be large relative to context switch time but small enough to allow quick response. 80% of CPU bursts should be shorter than q (so most processes complete within one quantum).

§4 Priority Scheduling

Priority Scheduling — A priority number (integer) is associated with each process. The CPU is allocated to the process with the highest priority. Convention: smaller integer = higher priority (priority 1 is highest). Available in both preemptive and non-preemptive modes. SJF is a special case of priority scheduling where priority = inverse of burst time.
Starvation Problem: Low-priority processes may never execute if high-priority processes keep arriving.
Solution — Aging: Gradually increase the priority of processes that have been waiting for a long time. Eventually every process gets high enough priority to run. Example: increase priority by 1 for every 15 minutes of waiting.

Combining Priority + Round Robin

Priority + RR: System executes the highest-priority process. Among processes with the same priority, use Round Robin. This combines the fairness of RR with the priority ordering of priority scheduling. Used in real systems like Windows (priority classes + RR within each class).
Priority convention MATTERS in exam: Always check if the problem says "lower number = higher priority" or "higher number = higher priority." Most standard problems: lower number = higher priority.
Starvation solution = Aging. This is the standard exam answer.
SJF is priority scheduling where priority = 1/BT (shorter burst = higher priority).

§5 Multilevel Queue & Feedback

Multilevel Queue Scheduling — The ready queue is partitioned into separate queues. Each process is permanently assigned to one queue based on its type (foreground/interactive vs background/batch). Each queue has its own scheduling algorithm. Scheduling must also be done between queues.

Classic Multilevel Queue Setup

  • Queue 1 — System processes: Highest priority, real-time tasks
  • Queue 2 — Interactive processes: Foreground, user-facing (uses Round Robin)
  • Queue 3 — Interactive editing: Text editors, IDEs
  • Queue 4 — Batch processes: Background jobs (uses FCFS)
  • Queue 5 — Student jobs: Lowest priority, non-urgent
Inter-queue scheduling:
→ Fixed priority: Serve all from Queue 1 before Queue 2, etc. Risk: starvation of lower queues.
→ Time slice: Each queue gets a portion of CPU time. E.g., 80% for foreground (RR), 20% for background (FCFS).

Key limitation: Processes are permanently assigned to a queue. Cannot move between queues → inflexible.

Multilevel Feedback Queue

Multilevel Feedback Queue — An extension of Multilevel Queue where processes CAN move between queues. This allows aging to be implemented and adapts to the changing behavior of processes. Most general CPU scheduling algorithm — most OS use a variant.
Classic 3-queue Feedback Example (from PDF):
→ Q0: Round Robin with q=8ms (highest priority)
→ Q1: Round Robin with q=16ms
→ Q2: FCFS (lowest priority)

Process behavior: New job enters Q0. Gets 8ms. If doesn't finish → moved to Q1. Gets 16ms. If doesn't finish → moved to Q2 (FCFS). CPU-bound processes gradually sink to Q2. I/O-bound processes (short bursts) stay in Q0/Q1.

Multilevel Feedback Queue — Parameters

A multilevel feedback queue scheduler is defined by these parameters:
1. Number of queues
2. Scheduling algorithm for each queue (RR, FCFS, priority, etc.)
3. Method to upgrade a process (when to move to higher-priority queue — aging)
4. Method to demote a process (when to move to lower-priority queue — used too much CPU)
5. Which queue a new process enters (usually highest priority queue)

Multilevel Queue vs Multilevel Feedback Queue

Multilevel Queue Processes permanently assigned to a queue. No movement between queues. Simple to implement. May cause starvation of low-priority queues. Example: System vs user vs batch queues.
Multilevel Feedback Queue Processes CAN move between queues. Handles aging (promote long-waiting processes). Adapts to process behavior. Most flexible scheduler. Most complex to implement. Used in real OS.
Multilevel Feedback Queue = most general scheduling algorithm — used in real modern OS.
MLFQ implements aging automatically: if a process in Q2 waits too long → move back to Q0.
CPU-bound processes sink to lower queues; I/O-bound float to top.
For the 3-queue example: Q0=RR(q=8), Q1=RR(q=16), Q2=FCFS. New job starts at Q0.

§6 Multiprocessor Scheduling

Multiprocessor Scheduling deals with scheduling when the hardware consists of multiple CPUs. Much more complex than single-processor scheduling because multiple CPUs must be utilized efficiently and processes must be distributed across them.

Approaches to Multiprocessor Scheduling

Asymmetric Multiprocessing One master processor handles all scheduling, I/O processing, and OS activities. Other slave processors execute user code only. Simple — only master accesses system data structures. Master becomes bottleneck.
Symmetric Multiprocessing (SMP) Each processor is self-scheduling. All processors are peers. Each processor has its own ready queue OR they share a common ready queue. Most common approach in modern systems.

Load Balancing

In SMP, we want to distribute workload evenly among all CPUs — this is load balancing.

→ Push migration: A task periodically checks the load on each processor. If load is imbalanced, it pushes processes from overloaded CPU to idle/less-loaded CPU.
→ Pull migration: An idle processor pulls a process from a busy processor's ready queue.

Both push and pull migration are often used together. Linux uses both with a load balancer that runs every 200ms.

Processor Affinity

When a process migrates to a new CPU, the new CPU's cache must be repopulated — costly! Processor affinity = keeping a process on the same CPU to benefit from warm cache.
→ Soft affinity: OS attempts to keep process on same CPU but can migrate if needed
→ Hard affinity: Process locked to specific CPU(s) — will never migrate
Linux provides system call sched_setaffinity() for hard affinity.

NUMA and Scheduling

NUMA (Non-Uniform Memory Access): Systems where some CPUs have faster access to certain memory banks than others. The OS scheduler must be NUMA-aware — schedule a process on the CPU closest to the memory it uses. Linux NUMA scheduler tries to allocate memory on the node closest to the CPU that will use it.

Multicore Processor Scheduling

Modern processors have multiple cores on one chip. Each core appears as a separate CPU to the OS.

Memory Stall: When a core waits for memory access (cache miss), it is doing nothing. Solution: Multithreaded cores — each core supports multiple hardware threads. When one hardware thread stalls on memory, the core switches to another hardware thread instantly. This is chip multithreading (CMT) or hyperthreading (Intel's term).

Two-level scheduling:
→ Level 1: OS schedules software threads to hardware threads (logical CPUs)
→ Level 2: Each core decides which hardware thread runs next within the core

Real-Time Scheduling

Soft real-time: System provides no guarantee as to when a critical process is scheduled. Just ensures critical processes have higher priority. Example: multimedia streaming.
Hard real-time: Tasks must be serviced within a deadline. Missing deadline = system failure. Requires special scheduling algorithms like Rate Monotonic (RM) and Earliest Deadline First (EDF).
SMP = each CPU self-scheduling. AMP = only master schedules.
Load balancing goal: All CPUs approximately equally busy. Push migration + pull migration.
Processor affinity = cache warmth preference. Avoid migration if cache is warm.
Hyperthreading = 2 hardware threads per core. OS sees 4-core + HT = 8 logical CPUs.
