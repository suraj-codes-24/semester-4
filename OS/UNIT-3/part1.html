§1 Process Concept & Address Space

A Process is a program in execution. A program is a passive entity stored on disk (executable file). A process is an active entity with resources allocated to it. A program becomes a process when the executable file is loaded into memory. One program can create multiple processes (e.g., multiple users running the same program).

Process vs Program

ProgramPassive — file on disk. Static instructions. No resources. No state changes. Example: chrome.exe sitting on hard drive.
ProcessActive — loaded and executing. Dynamic state changes. Has CPU, memory, I/O. Multiple instances possible. Example: 5 Chrome windows running.

Process Address Space (Memory Layout)

A process in memory is organized into sections (from high address to low address):

Stack (highest address, grows ↓) — function call frames, local variables, return addresses, function parameters
↕ Free Space — gap between stack and heap (allows both to grow)
Heap (grows ↑) — dynamically allocated memory at runtime (malloc(), new)
Data Section — global and static variables (initialized + BSS for uninitialized)
Text Section (lowest address) — compiled program code; read-only; shared between multiple instances

CPU-I/O Burst Cycle

Process execution consists of a cycle of CPU execution (CPU burst) and I/O wait (I/O burst). The process alternates between these two states throughout its lifetime.

→ CPU burst: Period where the process is actively using the CPU
→ I/O burst: Period where the process waits for I/O to complete

CPU burst distribution: Most processes have many short CPU bursts (interactive/I/O-bound). Fewer processes have long CPU bursts (CPU-bound like scientific computing).

Types of Processes

I/O-Bound Process Spends more time waiting for I/O than computing. Has many SHORT CPU bursts. Example: text editor, web browser, database query. Response time is more important than throughput.
CPU-Bound Process Spends more time doing computations. Has FEW LONG CPU bursts. Example: video encoding, scientific simulation, matrix multiplication. Throughput and CPU utilization are critical.

Process Creation in UNIX (Linux representation)

In Linux, each process is represented by a C structure called task_struct:

pid_t pid; — unique process identifier
long state; — current state of the process
unsigned int time_slice; — scheduling information
struct task_struct *parent; — pointer to parent process
struct list_head children; — list of child processes
struct files_struct *files; — list of open files
struct mm_struct *mm; — pointer to address space (memory map)

Process Creation (fork + exec)

1
fork(): Creates a child process as a duplicate of the parent. Returns 0 to child, child PID to parent. Child inherits open files, privileges, scheduling attributes.
2
exec(): Child overlays its address space with a new program. The child's code is replaced — but PID stays the same.
3
wait(): Parent suspends until child terminates. Collects child's exit status. Prevents zombie processes.
4
exit(): Process terminates, returns status to parent, OS reclaims all resources.

Process Termination

Zombie Process
Process has finished (called exit()) but parent hasn't called wait() yet. PCB still exists. Consuming no CPU/memory resources but occupying a slot in process table. Shown as "Z" in ps.
Orphan Process
Parent terminated before child. In UNIX, orphans are re-parented to init (PID 1) or systemd, which eventually calls wait() to collect them.

Cascading Termination

Some OSes do not allow a child to continue if its parent terminates. When a parent exits, the OS initiates cascading termination — all children, grandchildren, and descendants are also terminated automatically. This ensures no orphan processes exist in such systems.

Interprocess Communication (IPC)

Processes can be independent (no data sharing) or cooperating (can affect/be affected by other processes). Cooperating processes need IPC for:
→ Information sharing — multiple processes access same data
→ Computation speedup — parallel execution of subtasks
→ Modularity — divide system into separate communicating processes
→ Convenience — multiple tasks simultaneously

Two IPC models: Shared Memory (fast, needs sync) and Message Passing (slower, safer, works over network).
Process address space layout to memorize (top→bottom): Stack → (free space) → Heap → Data → Text.
I/O-bound vs CPU-bound: I/O-bound = many short bursts. CPU-bound = few long bursts.
fork() return: 0 → child. Positive → parent (gets child PID). -1 → error.

§2 Process States & Transition Diagram

As a process executes, it changes state. The state of a process is defined by its current activity. At any point in time, a process is in exactly ONE of these five states.

The Five Process States

StateDescriptionWhere is process?
NewProcess is being createdBeing loaded into memory
ReadyWaiting to be assigned to a processorIn ready queue in memory
RunningInstructions are being executedOn the CPU
WaitingWaiting for some event to occurIn I/O/device queue
TerminatedProcess has finished executionBeing cleaned up

Process State Transition Diagram

NEWCreated, admitted
READYIn ready queue, waiting for CPU
RUNNINGExecuting on CPU
WAITINGI/O or event wait
TERMINATEDFinished

State Transitions (with triggers)

TransitionTriggerDirection
New → ReadyOS admits the process (memory available)Admitted
Ready → RunningCPU scheduler dispatches the processScheduler dispatch
Running → WaitingProcess requests I/O or event (e.g., disk read)I/O or event wait
Running → ReadyProcess preempted (time quantum expired)Interrupt / preemption
Waiting → ReadyI/O completed or event occurredI/O or event completion
Running → TerminatedProcess completes or is killedExit / abort
Key transitions for scheduling decisions:
Situations 1 (Running→Waiting) and 4 (Terminated): No choice — must schedule next process.
Situations 2 (Running→Ready) and 3 (Waiting→Ready): OS has a CHOICE — continue or switch. This is where preemptive vs non-preemptive matters.

Preemptive vs Non-Preemptive Scheduling

Non-Preemptive Scheduling only happens in situations 1 (Running→Waiting) and 4 (Terminated). Once CPU given to a process, it keeps CPU until it finishes OR voluntarily gives it up by waiting for I/O. Simple but can cause convoy effect.
Preemptive Scheduling can happen in all 4 situations including 2 and 3. OS can forcibly take CPU from running process. Allows fairness and time-sharing. Can cause race conditions on shared data. Used by all modern OS (Windows, Linux, macOS).

Context Switch

A context switch occurs when the CPU switches from one process to another. The system must save the state (context) of the old process and load the saved state of the new process.

Steps:
1. Save all CPU registers, PC, stack pointer of current process → into its PCB
2. Update current process state (to Ready or Waiting)
3. Load saved state of new process from its PCB
4. Update new process state to Running
5. Jump to new process's program counter
Context switch is PURE OVERHEAD — no useful computation happens during the switch itself. The more complex the OS and PCB, the longer the context switch takes. Hardware support (multiple register sets) can speed it up. Typical: 1–1000 microseconds per switch. Minimizing context switch frequency is a key OS design goal.

Dispatcher

The Dispatcher is the module that gives control of the CPU to the process selected by the CPU scheduler. It involves: switching context, switching to user mode, jumping to the proper location in the user program.
Dispatch latency = time taken for dispatcher to stop one process and start another running. Should be minimized.

§3 PCB & Process Identification

The Process Control Block (PCB) (also called Task Control Block) is a data structure maintained by the OS for every process. It contains all information needed to manage and track a process. It is essentially the "identity card" of a process — when a context switch occurs, the PCB is saved and restored.

Memory Management InfoBase/limit registers, page tables, segment tablesMemory protection
Accounting InfoCPU time used, wall-clock time, time limits, job numberBilling, auditing
I/O Status InfoList of open files, I/O devices allocated to processI/O management

Process Identification Information

The OS must be able to uniquely identify every process. Key identification fields:

→ PID (Process ID): Unique integer assigned by OS. PID 0 = swapper/idle process. PID 1 = init/systemd (first user process). All other processes have PID ≥ 2.
→ PPID (Parent PID): PID of the process that created this process (parent). Allows tracing the process tree.
→ UID / GID: User ID and Group ID of the process owner. Used for permission/security checks.
→ EUID / EGID: Effective UID/GID — may differ from real UID for setuid programs.
→ Session ID / Group ID: Identifies which terminal session or process group this process belongs to.

Process Scheduling Queues

The OS maintains multiple queues to track processes:
→ Job Queue: Set of ALL processes in the system (including those on disk)
→ Ready Queue: Set of all processes in main memory, ready and waiting to execute. Usually a linked list of PCBs.
→ Device Queues: Set of processes waiting for a specific I/O device. Each device has its own queue.

Processes migrate among these queues based on their state transitions.

Ready Queue and Device Queue Structure

The Ready Queue is implemented as a linked list of PCBs. The queue has a header containing pointers to the first and last PCBs in the list. Each PCB has a pointer to the next PCB in the queue. When a process becomes ready, its PCB is linked to the tail of the ready queue. The CPU scheduler picks from the head.
PCB = identity + state + context of a process. Every time you're asked "what info does OS maintain about a process?" — the answer is PCB fields.
PID 1 = init/systemd. If init dies, the entire OS panics.
Ready queue ≠ device queue. There's ONE ready queue but MULTIPLE device queues (one per I/O device).

§4 Schedulers & Scheduling Concepts

CPU Scheduling is the basis of multiprogrammed operating systems. The objective is to maximize CPU utilization by ensuring the CPU always has a process to execute. Whenever the CPU becomes idle, the OS must select one of the processes in the ready queue to be executed — this is done by the CPU Scheduler.

Three Types of Schedulers

1. Short-Term Scheduler (CPU Scheduler)

→ Selects which process from the ready queue gets the CPU next
→ Invoked very frequently — every milliseconds
→ Must be very FAST (since it runs so often — overhead must be minimal)
→ Sometimes the only scheduler in the system
→ Makes the actual dispatch decision

2. Long-Term Scheduler (Job Scheduler)

→ Selects which processes from the job pool (on disk) to bring into the ready queue (load into memory)
→ Invoked infrequently — seconds to minutes between invocations
→ Can be slow (takes time to make good decisions)
→ Controls the degree of multiprogramming (how many processes are in memory)
→ Strives for a good mix of CPU-bound and I/O-bound processes

3. Medium-Term Scheduler (Swapper)

→ Swaps processes out of memory to disk and back
→ Used when memory is overloaded — reduces the degree of multiprogramming temporarily
→ Swapping out: move a process from memory to disk (free up memory)
→ Swapping in: bring a swapped process back when memory available

Degree of Multiprogramming

The degree of multiprogramming = number of processes currently in memory (ready queue). The long-term scheduler controls this. If it stays constant → long-term scheduler only invoked when a process leaves the system (terminates). The long-term scheduler must carefully select a mix of I/O-bound and CPU-bound processes:
→ All I/O-bound → CPU always idle (waiting for I/O)
→ All CPU-bound → I/O devices idle (waiting for CPU bursts to finish)
→ Good mix → both CPU and I/O devices are busy simultaneously

Scheduling Decisions: When Does Scheduling Occur?

CPU scheduling decisions may take place when a process:
1. Switches from Running → Waiting (e.g., I/O request) — must schedule
2. Switches from Running → Ready (e.g., time quantum expired) — choice
3. Switches from Waiting → Ready (e.g., I/O completion) — choice
4. Process terminates — must schedule

Non-preemptive: scheduling only in cases 1 and 4. Preemptive: scheduling in all 4 cases.

Preemptive Scheduling — Caution Areas

Preemptive scheduling introduces complexity:
→ Race conditions on shared data: If Process A is updating shared data and is preempted, Process B may read inconsistent data → need synchronization.
→ Preemption in kernel mode: If OS is processing a system call and gets preempted, kernel data structures could be left inconsistent → need reentrant kernel.
→ Interrupts during OS activities: Critical sections in OS must be protected from interrupts.

All modern OS (Windows, macOS, Linux, UNIX) use preemptive scheduling despite these challenges.

Gantt Chart

A Gantt Chart is a bar chart used to illustrate a CPU schedule. Time is shown on the x-axis. Each process is shown as a horizontal bar spanning from its start time to its end time on the CPU. Idle CPU time is shown as a gap or labeled "Idle".

Example (FCFS: P1=24, P2=3, P3=3 arriving in order P1, P2, P3):

|——P1——|P2|P3|
0        24 27 30

WT: P1=0, P2=24, P3=27. Average WT = (0+24+27)/3 = 17 ms
Why maximize CPU utilization?
CPU is the most expensive resource in a computer system. Historically, CPUs were very expensive — any idle CPU time = wasted money. In modern systems: CPU utilization = 40% (lightly loaded) to 90% (heavily loaded). Goal: keep CPU as busy as possible by always having a ready process available.

§5 Performance Criteria & Key Times

Scheduling Criteria (Optimization Goals)

CriterionDefinitionGoal
CPU UtilizationPercentage of time CPU is busy (not idle)Maximize
ThroughputNumber of processes completing per unit timeMaximize
Turnaround Time (TAT)Total time from submission to completionMinimize
Waiting Time (WT)Total time spent in ready queueMinimize
Response Time (RT)Time from submission until first response (not output)Minimize

Key Time Definitions & Formulas

Arrival Time (AT): Point of time process enters the ready queue

Burst Time (BT): Total CPU time required by the process to execute (also called Execution Time or Running Time). Cannot be known in advance — must be estimated.

Completion Time (CT): Time at which process finishes execution and exits. Also called Exit Time.

Turnaround Time (TAT): Total time in system from arrival to completion.
TAT = CT - AT    OR    TAT = BT + WT

Waiting Time (WT): Time spent waiting in ready queue (not executing).
WT = TAT - BT    OR    WT = CT - AT - BT

Response Time (RT): Time from arrival until process FIRST gets the CPU.
RT = (First CPU time) - AT

Solved Example: Computing All Times

🧪 FCFS Example — Computing CT, TAT, WT
Given: P1(AT=0, BT=4), P2(AT=1, BT=3), P3(AT=2, BT=1), P4(AT=3, BT=2), P5(AT=4, BT=5)
FCFS Order (by arrival): P1→P2→P3→P4→P5
Gantt: |P1|P2|P3|P4|P5| → 0,4,7,8,10,15
CT: P1=4, P2=7, P3=8, P4=10, P5=15
TAT = CT - AT: P1=4-0=4, P2=7-1=6, P3=8-2=6, P4=10-3=7, P5=15-4=11
WT = TAT - BT: P1=0, P2=3, P3=5, P4=5, P5=6
✓ Avg CT=8.8, Avg TAT=6.8, Avg WT=3.8

Summary: All Formulas at a Glance

CT = time process exits system
TAT = CT - AT (also = BT + WT)
WT = TAT - BT (also = CT - AT - BT)
RT = first CPU time - AT
Avg TAT = sum of all TAT / number of processes
Avg WT = sum of all WT / number of processes
TAT vs WT: TAT includes the burst time (execution time). WT is only the time waiting, not executing.
Response Time ≠ Waiting Time: RT = first CPU time - arrival. WT = total waiting time across all ready queue periods.
For non-preemptive algorithms: WT = TAT - BT = CT - AT - BT. This is the most commonly used formula in exams.

§6 Threads & Their Management

A Thread (thread of execution) is the smallest sequence of programmed instructions that can be managed independently by a scheduler. A thread is a path of execution within a process. Also called a Lightweight Process (LWP). Multiple threads within a process share the process's resources but execute independently — this is called Multithreading.

What Each Thread Has (Own) vs Shares

Thread's OWN (Private):
→ Program Counter (PC)
→ CPU Registers
→ Stack (function calls, local vars)
→ Thread state (ready/running/waiting)
→ Thread ID
SHARED with all threads in process:
→ Code section (text)
→ Data section (global variables)
→ Heap (dynamic memory)
→ Open files and I/O resources
→ Process ID and privileges

Benefits of Multithreading

  • Responsiveness: One thread handles UI while another does computation — app stays responsive
  • Resource Sharing: Threads share process memory — no need for complex IPC
  • Economy: Creating/destroying threads is much cheaper than processes (no new address space)
  • Scalability: On SMP systems, threads can run truly in parallel on different CPUs
  • Parallelism: A single application can divide its task into multiple parallel threads

Thread Management — Types of Threads

User-Level Threads (ULT) Managed by user-space thread library (e.g., POSIX Pthreads, Java threads). OS is unaware — sees only one process. Fast to create and context-switch. If one thread makes a blocking system call, ALL threads in the process block. Cannot run truly in parallel on multiprocessors.
Kernel-Level Threads (KLT) Managed directly by the OS kernel. OS can schedule individual threads independently. If one thread blocks, others can still run. Can run in true parallel on multiple CPUs. Slower to create (kernel involvement). Example: Windows threads, Linux pthreads (mapped to kernel).

Thread Models (Mapping ULT to KLT)

ModelMappingExamplePros/Cons
Many-to-OneMany ULT → 1 KLTGreen threadsFast, but blocks all if one blocks
One-to-One1 ULT → 1 KLTLinux, WindowsTrue parallelism, but kernel overhead per thread
Many-to-ManyMany ULT → Many KLT (≤ ULT)Solaris, older UNIXBest of both; complex to implement

Thread Storage in PCB

Since threads are part of a process, the PCB must be extended to support multiple threads. For each thread, the OS stores: thread ID, thread state, program counter, register set, and stack pointer. The PCB maintains a list of all thread structures for that process. Modern Linux uses a unified task_struct for both processes and threads (threads are just processes sharing address space).
Thread = lightweight process. Key difference: threads share address space; processes don't.
Creating a thread is faster than creating a process — no new page table, no new address space, no copy of parent.
One-to-One is most common today (Linux, Windows). Each user thread maps to one kernel thread.
Multiple PCs per process = multithreading. The PCB needs storage for multiple program counters.
